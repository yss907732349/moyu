# 摸鱼隐者全局 UI 设计与优化 PRD

## 1. 文档定位

本文是 `摸鱼隐者` 小程序用户端全局 UI 设计与优化 PRD，用于统一首页、社区、帖子详情、我的、补给铺、隐者日报、生存账本、工作档案、WebView/兼容页和空状态页面的视觉语言、组件规范、内容规则、交互反馈和验收标准。

本文基于用户提供并已验证满意的“隐者营地”暗色像素 UI 原型提炼而来，并结合当前项目已经落地的工位低暴露使用场景、暗黑忍者 RPG、像素角色、RPG 面板、阵营身份和上班工具属性进行全局化整理。

后续所有小程序 UI 调整应优先遵循本文，而不是继续在单个页面中临时新增一套样式。

### 1.0 当前事实来源

截至 `add-miniapp-workplace-mode-default` 实现后，小程序用户端默认视觉已从暗色营地切换为浅色“工位模式”，暗黑像素忍者 RPG 视觉保留为用户从首页右上角主动进入的“隐者模式”。社区公开个人页、关注/取消关注、关注列表、粉丝列表、“我的”页关注/粉丝/连续签到三项数据、作者头像/昵称公开主页入口、弱化 `IP属地` 和社交敏感字段过滤继续作为当前主规格。后续模型接手时，应把本文第 11.2、11.4 节和 `openspec/specs/community-follow-profile-system/spec.md`、`openspec/specs/visual-system/spec.md`、`openspec/specs/mvp-beta-readiness/spec.md` 作为社交 UI 事实来源，不再回到“是否要做关注系统”的讨论阶段。

### 1.1 第一版视觉基线

当前第一版小程序主视觉、排版、icon 和插画已经暂定，作为后续用户端 UI 工作的默认基线：

- 视觉语言固定为 `工位模式默认 + 隐者模式可切换 + 隐者营地信息流`：默认使用浅色办公友好背景、浅色卡片和深色文字，用户主动切换后进入暗黑忍者 RPG 视觉。
- 首页、社区、我的、补给铺、隐者日报、大陆新闻、生存账本、工作档案、首跑/空状态/错误态的页面结构和主视觉层级按当前实现与本文规则保持。
- 卡片体系固定为重点像素框、轻立体卡和扁平容器三档；重点像素框只用于本文白名单中的页面主视觉位置。
- 底部主导航固定为首页、社区、我的三个微信原生 tabBar 项，使用项目内像素 icon 资产；`pages.json` 默认色彩为工位模式，运行时跟随本地 `moyuxia.visualMode` 切换导航栏和原生 tabBar 色彩。
- 菜单、状态、补给、账本、日报、工作设置等 icon 通过稳定 asset key、resolver 或 registry 收敛。
- 插画、头像、徽章、Banner、账本背景和 tabBar icon 使用当前文档与 `docs/miniapp-visual-illustration-system.md` 记录的稳定 key、路径、比例和安全区；工位模式只调整外框、底座、文字颜色、文字阴影和文字安全区，不重画整套白版资产，也不复用隐者模式紫色像素框作为默认工位边框。

后续视觉优化默认只处理局部图片替换、缺失 icon 补齐、裁切/遮罩/安全区、小屏可读性、普通卡片轻立体质感和包体控制。若要改变整体视觉方向、主导航、核心页面排版、重点像素框白名单、语义 icon registry 或现有 asset key，应先新建 OpenSpec change 重新定义范围和验收边界。

### 1.2 摸鱼模式办公表格视觉例外

`add-stealth-workbench-mode` 已将 `pages/stealth-workbench/*` 定义为用户端视觉例外。该页面组服务于办公室低暴露浏览，不套用普通用户端暗黑忍者 RPG 主视觉，也不跟随“隐者模式”切换，而固定使用浅色办公表格隐喻。

摸鱼模式必须遵守以下边界：

- 入口通过 `stealth_workbench_mode` feature registry 控制，默认从首页快捷入口进入 `/pages/stealth-workbench/index`，不得新增底部主 tab。
- 页面范围仅包含摸鱼模式工作台、表格版日报列表、表格版日报详情、表格版论坛列表和表格版帖子详情。
- 页面使用浅色办公背景、白色或极浅灰表格面、细网格线、表头、字段行、列表行、合并单元格、固定数字列、固定时间列和文本操作列。
- 页面不得使用插画、icon、头像、徽章、阵营图、重点像素框、RPG 面板、深色营地背景、发光、粒子、弹跳、淡入淡出或游戏化按钮。
- 论坛只允许浏览、详情、评论/回复只读、标记和归档；不得提供发帖、评论、回复、举报、图片上传、消息中心、我的帖子、我的评论、我的回复或收藏管理入口。
- 图片附件默认只显示 `附件数量`，附件清单只显示 `附件 N | 查看`，单图查看层无标题栏、无按钮、无保存、无转发、无说明文案，点击任意位置立即关闭。
- 320px、375px 和 414px 小屏下不得横向滚动、内容重叠或遮挡操作列；长正文、日报正文、帖子正文、评论和回复使用合并单元格自然换行。

- 删除、合并或隐藏已有业务板块。

## 3. 设计目标

### 3.1 体验目标

全局 UI 应让用户感到：

- 默认界面像一个低暴露的工位工具，主动切换后才进入暗色、像素、RPG 化的隐者体验。
- 页面有统一的“隐者营地”气质，而不是不同页面各做各的。
- 核心数据清晰可读，尤其是金额、倒计时、状态、评论和文章正文。
- 暗色氛围浓，但不压抑、不脏、不影响阅读。
- 按钮、卡片、tab 有明确触感。
- 插画和头像来自项目真实资产，不像临时外链拼贴。

### 3.2 产品目标

全局 UI 必须强化当前 MVP 的核心循环：

1. 上班时间转化为可见金额和倒计时。
2. 用户获得隐者身份、阵营、等级和资源反馈。
3. 用户通过社区、日报、世界观内容产生轻互动。
4. 用户通过补给铺和生存账本形成“上班续命”消费反馈。

### 3.3 工程目标

全局 UI 优化应尽量做到：

- 优先复用 `apps/miniapp/src/styles/visual-system.css` 和已有页面类名。
- 抽象公共样式 token，减少页面内重复色值。
- 不为了视觉改动破坏接口、缓存、首跑状态和内容安全边界。
- 不新增无定义功能入口。
- 不更换现有插画，除非明确是资产替换任务。

## 4. 视觉总方向

### 4.1 核心风格

全局视觉方向定义为：

`工位模式默认 + 隐者模式可切换 + 隐者营地信息流`

它不是纯赛博朋克，也不是复古网页像素风，而是：

- Material-like 的清晰层级。
- 工位模式用浅色页面底、浅色卡片、深色文字、细描边和办公友好的状态色降低暴露感。
- 隐者模式用像素边框、内阴影、紫色、青色、金色和暗黑忍者插画承载世界观。
- 信息流页面保留移动端扫读效率。

### 4.2 关键词

- 工位低暴露
- 隐者模式
- 隐者营地
- RPG 面板
- Material surface token
- 紫色激活态
- 青色状态反馈
- 金色强操作和金额
- 像素边框
- 内高光和内压边
- 克制发光
- 真实插画
- 移动端信息流
- 文字强可读

### 4.3 禁止方向

全局 UI 不应走向：

- 满屏高对比网格背景。
- 大面积霓虹故障闪烁。
- 纯营销落地页。
- 过度卡片套卡片。
- 空功能宫格。
- 平台导购风。
- 真实品牌露出。
- 过度拟物、厚重木纹或复杂纹理。

## 5. 全局色彩规范

### 5.0 视觉模式规则

- `workplace` 是默认视觉模式；本地没有保存值时，小程序使用工位模式渲染页面、导航栏和原生 tabBar。
- `hermit` 是用户主动切换的隐者模式；首页右上角按钮在工位模式显示“隐者模式”，在隐者模式显示“返回工位”。
- 视觉模式只保存在本地 `moyuxia.visualMode`，不依赖服务端账号同步、远端配置或系统主题。
- 所有普通用户端页面根节点应挂载 `vs-mode-workplace` 或 `vs-mode-hermit`，并优先消费 `--camp-*`、`--vs-*` 等模式化变量。
- `pages/stealth-workbench/*` 固定使用办公表格视觉例外，不跟随隐者模式变暗。
- 模式切换只改变视觉、导航栏和 tabBar 外观，不改变登录、建档、工作档案、隐藏模式、金额计算、社区、补给或账本业务逻辑。

### 5.1 主色板

| 语义       | Token                     | 色值                      | 用途                               |
| ---------- | ------------------------- | ------------------------- | ---------------------------------- |
| 语义       | 工位模式默认              | 隐者模式                  | 用途                               |
| --------   | ------------------------- | ------------------------- | ------------------------           |
| 页面底     | `#f5efe4` / `#f7f0e6`     | `#050508` / `#13131b`     | 小程序通用页面背景                 |
| 卡片面     | `#fffdf8` / `#fffaf1`     | `#1a1a2e` / `#1f1f27`     | 卡片、表单、列表底                 |
| 主文字     | `#34302a`                 | `#e4e1ed`                 | 正文和标题                         |
| 强文字     | `#17130f`                 | `#ffffff`                 | 关键标题、强调文本                 |
| 弱文字     | `#8a7d6d`                 | `#9d8ba0`                 | 时间、低优先级标签                 |
| 默认描边   | `#d8c9b4`                 | `#514255`                 | 卡片、分割线、次级控件             |

| 状态青     | `#248d92`                 | `#00dbe9`                 | 工作中、同步、消息、进度           |

| 错误色     | `#b33f32`                 | `#ffb4ab`                 | 错误、举报风险、失败提示           |

### 5.2 使用原则

- 紫色用于品牌感、当前选中、主要导航激活。
- 青色用于状态、进度、同步、消息和系统反馈。
- 咖啡金用于金额、奖励、tab 选中、徽章和非按钮强调。
- 按钮橙用于保存、发布、关注、发送、重试、进入、切换模式等明确操作按钮；按钮不得复用 `--camp-gold` 作为背景。
- 错误色仅用于错误、驳回、举报、失败，不用于普通强调。
- 单个卡片默认只允许一种主强调色，最多叠加一种辅助色。
- 大面积背景默认保持工位浅色；隐者模式大面积背景保持暗色，不使用主色铺底。
- 不再使用满屏网格背景；网格或扫描线只能作为局部低透明纹理。

### 5.3 页面背景规则

工位模式推荐背景：

```css
background: var(--camp-page-background);
```

禁止背景：

```css
/* 禁止作为工位模式全局整屏背景，容易出现明显暗色断层或格子地板 */
linear-gradient(rgba(236, 178, 255, 0.032) 1rpx, transparent 1rpx),
linear-gradient(90deg, rgba(0, 219, 233, 0.024) 1rpx, transparent 1rpx);
```

局部可用：

- Banner 文字区局部底。
- 入口卡顶部纹理。
- 空状态小插画背景。
- 详情页引用卡背景。

## 6. 字体与文字规范

### 6.1 字体策略

小程序正式实现不依赖外部 Google Fonts。字体策略为：

- 中文主字体：沿用项目当前方正黑体/系统中文字体栈。
- 数字、金额、倒计时、短标签：可使用项目展示字体变量或等宽回退。
- 不打包未授权字体文件。

推荐变量：

```css
--font-display: var(--vs-font-display);
--font-body: var(--vs-font-body);
--font-code: "JetBrains Mono", Consolas, "Courier New", monospace;
```

### 6.2 字号层级

| 层级        | 推荐 rpx        | 用途                 |
| ----------- | --------------- | -------------------- |
| 页面主标题  | `36rpx - 44rpx` | 首页身份、详情标题   |
| 模块标题    | `28rpx - 34rpx` | 卡片标题、区块标题   |
| 列表标题    | `26rpx - 30rpx` | 帖子标题、补给标题   |
| 正文        | `24rpx - 28rpx` | 文章、帖子、评论     |
| 辅助正文    | `20rpx - 22rpx` | 摘要、说明、时间     |
| 标签        | `16rpx - 20rpx` | 阵营、状态、分区     |
| 金额/倒计时 | `38rpx - 64rpx` | 首页主金额、主倒计时 |

### 6.3 文本换行规则

所有用户可输入或服务端返回的长文本必须防止横向溢出：

```css
display: block;
max-width: 100%;
white-space: normal;
overflow-wrap: break-word;
word-break: break-all;
word-wrap: break-word;
```

必须应用的文本类型：

- 帖子标题。
- 帖子正文。
- 帖子摘要。
- 评论正文。
- 回复正文。
- 日报引用标题和摘要。
- 世界观文章标题和摘要。
- 补给标题和说明。
- 错误提示和审核提示。

不建议应用 `break-all` 的文本类型：

- 按钮文案。
- tab 文案。
- 头像昵称单行展示。
- 状态短标签。

这些短 UI 文案应使用单行省略：

```css
overflow: hidden;
text-overflow: ellipsis;
white-space: nowrap;
```

### 6.4 时间格式规范

所有社区、评论、消息、订单、日报列表中的时间展示应统一：

| 时间范围          | 展示格式                                          |
| ----------------- | ------------------------------------------------- |
| 同一自然日        | `HH:mm`，例如 `13:27`                             |
| 1 天内但非同日    | `1天前`                                           |
| 2 天内            | `2天前`                                           |
| 3 天内            | `3天前`                                           |
| 超过 3 天且同一年 | `M月D日 HH:mm`，例如 `6月3日 13:27`               |
| 非同一年          | `YYYY年M月D日 HH:mm`，例如 `2025年12月31日 23:10` |

禁止直接展示 ISO 字符串，例如：

```text
2026-06-03T03:17:27.704Z
```

## 7. 间距、布局与密度

### 7.1 基础单位

使用 `4px / 8rpx` 作为最小网格感知单位。

| Token                 | 推荐值          | 用途             |
| --------------------- | --------------- | ---------------- |
| `page-padding-x`      | `28rpx - 32rpx` | 页面左右边距     |
| `page-padding-bottom` | `140rpx+`       | 底部导航安全区   |
| `stack-gap`           | `16rpx - 20rpx` | 卡片间距         |
| `card-padding`        | `18rpx - 24rpx` | 普通卡片内边距   |
| `dense-padding`       | `12rpx - 16rpx` | 列表、筛选、标签 |
| `control-height`      | `52rpx - 72rpx` | 按钮、输入框     |

### 7.2 页面布局规则

- 首页可以稍微更强视觉化，社区和详情页优先阅读效率。
- 论坛信息流卡片间距控制在 `16rpx` 左右。
- 详情页正文应有更大行高和更少装饰。
- 表单页面保持清晰分组，不把所有字段塞进一个大卡。
- 底部固定输入栏或底部导航必须给页面内容留出安全距离。

### 7.3 卡片嵌套规则

- 不做卡片套卡片，除非内部是评论引用、订单详情或表单分组。
- 大卡内部的小块应使用分割线、浅底或左边线，而不是再放一张完整卡片。
- 三列数据组使用一个外卡 + 内部分割线。


### 8.3 三档卡片层级

小程序用户端卡片必须按页面主任务和信息密度分成三档，不得把所有卡片都做成同等级重框。

| 层级       | 推荐类                                           | 适用位置                                                | 禁止扩散                                                  |
| ---------- | ------------------------------------------------ | ------------------------------------------------------- | --------------------------------------------------------- |
| 重点像素框 | `vs-pixel-frame-primary` / `vs-pixel-frame-gold` | 页面主卡、世界观 Banner、身份主卡、主推荐、页面核心统计 | 不挂到 `vs-panel`、`camp-card`、`camp-feed-card` 默认样式 |
| 轻立体卡   | `vs-card-raised`、`camp-card`、`camp-feed-card`  | 列表卡、入口卡、补给 item、账本报告、倒计时卡、普通统计 | 不使用粗外框、强角标或强辉光                              |
| 扁平容器   | `vs-card-flat`、表单/输入/标签原有控件样式       | 表单字段、评论、回复、订单行、tab、badge、chip、输入框  | 不使用重点像素框                                          |

重点像素框只能通过页面显式 opt-in 类启用。基础面板 `vs-panel`、`camp-card` 和 `camp-feed-card` 保持轻卡语义，避免信息流、表单、评论和订单行被全局加重。

### 8.4 重点像素框白名单

当前白名单：

| 页面       | 可使用重点像素框的位置  |
| ---------- | ----------------------- |
| 首页       | 今日已摸金额主卡        |
| 社区首页   | 顶部世界观 Banner       |
| 我的页     | 身份主卡                |
| 我的角色页 | 阵营身份主卡            |
| 补给铺     | 主推荐卡                |
| 隐者日报   | 今日话题 / 今日参悟主卡 |
| 生存账本   | 顶部今日生存消耗统计卡  |

明确禁用重点像素框的位置：

- `vs-panel`、`camp-card`、`camp-feed-card` 的默认样式。
- 社区帖子卡、文章卡、普通入口卡、倒计时卡、首页今日生存消耗入口。
- 表单分组、表单字段、picker、switch、输入框、底部输入栏。
- 评论、回复、举报入口、互动统计、筛选 tab、chip、badge、小标签。
- 补给分类 section、补给 item、补给标签、转链状态。
- 本周报告、近期订单容器、订单行、订单详情字段。
- 大陆新闻列表、详情正文和评论。

首页今日生存消耗入口允许整块轻立体卡使用偏明亮的像素 RPG 任务公告板背景，并允许三张小卡使用插画铺底和原生文字叠加；深色插画上可使用白字和文字阴影，不在整张插画上叠大面积遮罩、滤镜或扫描线。生存账本页顶部统计卡作为页面核心统计，继续使用原账本大插画背景和重点像素框；工位模式下重点框使用棕金或办公友好的边框，隐者模式保留紫色 RPG 框；其三类统计小卡可复用同一组场景插画，但本周报告、近期订单和订单行仍保持轻立体或扁平层级。

我的页身份主卡信息密度高，应保留重点像素框和暗色纯面板底，可使用低透明纹理、局部微光和像素高光强化质感；不得使用抢占文字可读性的整张复杂插画背景，也不得把身份主卡重框下放给菜单列表、工作档案状态卡或今日生存消耗摘要。

## 9. 动效与触感

### 9.1 按压反馈

可点击按钮：

- 按下位移 `translateY(2rpx)`。
- 阴影降低。
- 不使用大幅缩放。

信息流卡片：

- 点击时可轻微改变边框和阴影。
- 不要大幅移动，避免列表跳动。

底部导航：

- 当前 tab 使用选中像素 icon 和主色文字，原生 tabBar 不额外做自定义紫色容器。
- 点击反馈为 `translateY(1rpx - 2rpx)`。

### 9.2 动效克制原则

允许：

- 当前状态轻微 glow。
- 成功提示淡入淡出。
- 进度条宽度平滑变化。
- 图片加载占位淡入。

禁止：

- 大面积闪烁。
- 持续故障动画。
- 背景粒子常驻飘动。
- 影响正文阅读的扫描线。

### 9.3 Reduced Motion

正式实现应预留：

```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation: none !important;
    transition: none !important;
  }
}
```

## 10. 全局组件规范

### 10.1 页面壳层

所有主页面使用统一暗色背景和底部安全区。

要求：

- 页面背景干净，不铺满明显网格。
- 顶部可有轻微紫色径向氛围。
- 页面内容区域按 `vs-stack` 或统一 stack gap 排列。

### 10.2 顶部栏

适用页面：

- 社区。
- 我的帖子/收藏/消息。
- 详情页。
- 表单页。
- 兼容页。

规范：

- 高度约 `56px` / 小程序对应 rpx。
- 背景使用 `surface` 或透明暗色磨砂。
- 标题居中或左对齐取决于页面结构。
- 左侧返回/世界观图标，右侧通知/更多。
- 不在顶栏放过多操作。

### 10.3 底部导航

当前主导航只保留：

- 首页
- 社区
- 我的

规范：

- 使用微信原生 tabBar，不实现自定义 tabBar。
- `pages.json` 中为首页、社区、我的配置项目内 `iconPath` 和 `selectedIconPath`。
- 图标使用透明背景小尺寸像素 PNG：`static/tabbar/home.png`、`home-active.png`、`community.png`、`community-active.png`、`profile.png`、`profile-active.png`。
- 当前 tab 使用选中像素 icon 和主色文字，未激活 tab 使用未选中 icon 和弱文字。
- 不把补给铺、日报、账本作为底部主 tab。

### 10.4 Banner

适用：

- 首页世界观氛围。
- 社区顶部 Banner。
- 日报/世界观频道头图。
- 补给铺频道头图。

规范：

- 必须使用项目真实图片。
- 保留原图色彩，根据底图明度选择白字或深色字。
- 文字安全区固定在左侧或底部。
- 文字不得压在高亮复杂背景上。
- 不在正式 Banner 整图上叠扫描线或大面积遮罩。

### 10.5 信息流卡片

适用：

- 社区帖子列表。
- 我的帖子。
- 收藏帖子。
- 消息列表。
- 日报列表。
- 世界观文章列表。

卡片内容顺序：

1. 作者/来源信息。
2. 标题。
3. 摘要或正文片段。
4. 图片缩略图或引用。
5. 互动/状态行。

规则：

- 标题最多两行。
- 摘要最多两行。
- 有长连续字符必须断行。
- 图片缩略图固定尺寸，不能挤压文本列。
- 状态行用分割线隔开。

### 10.6 表单控件

适用：

- 工作档案设置。
- 发帖。
- 评论输入。
- 昵称修改。
- 补给后台以外的普通用户配置。

规范：

- 输入框背景 `surface`。
- 描边 `outline-variant`。
- 聚焦时使用紫色或青色边框。
- 保存/发布使用按钮橙，不使用咖啡金强调色。
- 危险操作使用错误色描边，不用金色。
- 输入提示简短，不写长说明。

### 10.7 状态面板

适用：

- 加载。
- 空状态。
- 错误。
- 未登录。
- 未建档。
- 未配置工作档案。
- 网络降级。

规范：

- 使用 `VisualStatePanel` 或同等统一组件。
- 图标/插画用项目已有 fallback 或真实资产。
- 文案短句化。
- 主操作使用按钮橙；非按钮状态强调按语义使用咖啡金、青色或错误色。
- 不注入假数据。

### 10.8 Toast / 成功反馈

规范：

- 背景深色半透明。
- 紫色或青色描边。
- 文字白色。
- 居中或顶部浮层。
- 自动消失。
- 不遮挡关键底部输入栏太久。

## 11. 页面级规范

### 11.1 首页

1. 身份区。
2. 今日已摸金额主卡。

6. 隐者日报入口。
7. 补给铺入口。

视觉规则：

- 今日已摸金额主卡是首页唯一重点像素框。

### 11.2 社区/论坛

社区已经采用营地 UI 方向，应作为全局信息流样板。

保留内容：

- 顶部世界观 Banner。
- 分区 tab。
- 最新/热门筛选。
- 消息入口。
- 发布/登录/建档按钮。
- 帖子列表。
- 图片附件。
- 赞、评、藏。
- 帖子、评论和回复作者公开个人页入口。
- 公开个人页。
- 关注列表和粉丝列表。

视觉规则：

- 不新增板块。
- 不更换插画。
- 不铺满强网格背景。
- 顶部世界观 Banner 可使用重点像素框。
- 帖子卡片为轻立体深色卡，分区 tab、筛选、互动统计和空状态保持低强调。
- 分区激活态为紫色。
- 引用日报为青色左边线。
- 发布按钮为金色。
- 可访问作者的头像和昵称可以进入公开个人页，并使用头像描边、昵称高亮和按压态表达轻量可点击感；无真实公开标识的种子或兼容作者只展示快照，不显示主页箭头、可点描边或假点击态。
- 公开个人页首屏使用隐者名片结构，展示头像、昵称、阵营、等级、称号、可用 `IP属地`、关注操作、关注数、粉丝数、公开帖数，并让公开帖子列表开头露出。
- `IP属地` 保留合规展示位，但使用弱层级文字，不使用主按钮色、强徽章、统计色或重点像素框。
- 关注按钮使用明确操作控件，不作为阵营、等级或认证徽章；关注成功使用轻量反馈，取消关注必须先确认或使用等价二次意图表达。
- 关注列表和粉丝列表使用轻量列表，头像、昵称、阵营、等级、称号和关注操作在同一列表项内稳定可读；列表项主体点击区和关注按钮点击区必须可区分。
- 关注列表和粉丝列表应覆盖加载、真实空、失败重试、分页加载和没有更多状态；空状态不填推荐用户、假用户或开发种子用户。
- 公开个人页、关注列表和粉丝列表不得展示手机号、微信身份字段、完整 IP、市/区县/街道、IP 来源 header、薪资、工作档案、隐币、能量、后台治理状态或内容安全内部风险信息。
- 320px、375px 和 414px 小屏下，长昵称、长称号、长统计数字和长按钮文案应使用省略、换行、固定容器或展示压缩处理，不得横向滚动、重叠或挤出关注按钮。

- 举报按钮低调，不抢主操作。

- 三项状态在长数字下必须保持各自容器内省略、收缩或 `9999+` 展示压缩，不得遮挡头像、徽章、签到按钮或后续内容；压缩只影响展示，不改变服务端和本地完整数值。

规则：

- 金额用金色，但退款/回滚用错误或弱化状态。

- 空订单与 API 错误必须区分。

### 11.8 工作档案设置

规则：

- 表单分组清晰：薪资、工作时间、休息段、工作日、发薪日、隐藏模式。
- 表单分组、字段和保存反馈使用轻立体或扁平表单样式，不使用重点像素框。
- 敏感字段展示要克制。
- 保存按钮固定或位于页面底部主位置。
- 表单错误直接标注字段，不用长段落解释。

## 14. 可访问性与可读性

### 14.1 对比度

- 正文必须使用 `#e4e1ed` 或足够接近的浅色。
- 次文字不得低于 `#9d8ba0` 一类可读灰紫。
- 小字号标签不能使用过暗颜色。
- 图片上叠字必须根据底图明度选择白字或深色字，并配合文字阴影、描边或必要时的局部文字底；不用整图滤镜或大面积遮罩。

### 14.2 触区

- 按钮最小高度不低于 `52rpx`，关键操作建议 `60rpx - 72rpx`。
- 底部导航 tab 有足够点击区域。
- 图标按钮不能只有图形无触区。

### 14.3 长文本

所有用户输入文本和服务端文本必须防溢出。尤其要覆盖：

- 连续数字。
- 长英文。
- URL。
- 无空格字符串。
- 表情和中英混排。

## 15. 全局验收标准

### 15.1 视觉验收

- 页面整体符合暗色像素 Material + 暗黑忍者 RPG。
- 紫色、青色、金色职责清晰。
- 全局背景干净，没有大面积格子噪音。
- 卡片边框、内高光和按压触感统一。
- 插画来自项目真实文件。
- 底部主导航只保留首页、社区、我的。

### 15.2 功能边界验收

- 纯视觉任务不得改接口、路由、状态推导和内容安全逻辑。
- 不新增无定义业务入口。
- 不更换用户明确要求保留的插画。
- 不展示普通用户禁用字段。
- 不注入假数据。

### 15.3 文本验收

- ISO 时间不直接展示。
- 同日时间只显示时分。
- 3 天内显示相对天数。
- 长连续字符不横向溢出。
- 标题、正文、评论、回复可读。

### 18.4 建议新增或变更的规格

建议能力名：

```text
miniapp-global-visual-system
```

建议规格场景：

```text
#### Scenario: Global visual polish preserves business capabilities
- WHEN the miniapp global UI is polished
- THEN all existing enabled user-facing page sections SHALL remain accessible
- AND existing API calls, route targets, feature keys, storage keys, auth checks, moderation checks, and order/statistics calculations SHALL remain unchanged
- AND the update SHALL only change visual presentation, layout density, typography, colors, borders, spacing, assets presentation, and display formatting.
```

```text
#### Scenario: Existing page sections remain complete
- WHEN a user opens home, community, profile, supply center, daily content, world intel, accounting ledger, or work profile pages
- THEN the page SHALL retain its existing business sections and primary actions
- AND visual polish SHALL NOT delete, hide, rename into unrelated meaning, or replace existing user-facing capabilities.
```

```text
#### Scenario: User-generated and service text remains readable
- WHEN user-generated content, comments, replies, article titles, order labels, or system messages contain long continuous digits, English, URLs, or mixed text
- THEN the UI SHALL wrap the text inside its container
- AND the page SHALL NOT horizontally overflow.
```

```text
#### Scenario: Time display is normalized
- WHEN community, comment, message, content, or order time is shown to users
- THEN same-day times SHALL display HH:mm
- AND one to three days old times SHALL display 1天前, 2天前, or 3天前
- AND older same-year times SHALL display M月D日 HH:mm
- AND cross-year times SHALL display YYYY年M月D日 HH:mm
- AND raw ISO datetime strings SHALL NOT be displayed.
```

```text
#### Scenario: User-facing pages do not expose internal commercial fields
- WHEN a normal user views supply center or accounting ledger pages
- THEN the UI SHALL NOT display CPS provider names, commission, source order ids, raw provider payloads, apikey, sid, brand_id, act_id, backend notes, or undesmoothed attribution fields.
```

```text
#### Scenario: Existing project assets are preserved
- WHEN global UI polish is applied
- THEN existing project-owned illustrations, avatars, badges, banners, and covers SHALL remain in use
- AND external image URLs SHALL NOT be introduced for primary visual assets.
```
