## Requirements

### Requirement: 微信登录第一版

系统 SHALL 支持微信小程序登录第一版，用于创建或识别应用内用户，并为本人资料、签到和后续本人接口提供登录态。

#### Scenario: 新用户通过微信登录

- **WHEN** 小程序端提交有效微信登录凭证
- **THEN** API SHALL 通过服务端微信登录流程识别微信用户
- **AND** 系统 SHALL 创建应用内用户记录
- **AND** API SHALL 返回应用自己的登录态和本人用户标识

#### Scenario: 已有用户再次登录

- **WHEN** 已绑定微信身份的用户再次提交有效微信登录凭证
- **THEN** API SHALL 识别同一个应用内用户
- **AND** API SHALL 返回可用于后续 `/me/*` 请求的应用登录态

#### Scenario: 微信登录失败

- **WHEN** 微信登录凭证无效、过期或微信服务返回错误
- **THEN** API SHALL 返回客户端可识别的登录失败响应
- **AND** 系统 SHALL NOT 创建未绑定有效微信身份的真实用户资料

### Requirement: 本人上下文使用登录态优先

系统 SHALL 在本人接口中优先通过应用登录态识别当前用户，并仅在明确开发环境中允许临时本人上下文 fallback。

#### Scenario: 登录态访问本人接口

- **WHEN** 请求携带有效应用登录态访问 `/me/profile` 或 `/me/daily-checkin`
- **THEN** API SHALL 使用该登录态对应的应用内用户作为当前用户

#### Scenario: 未登录访问本人接口

- **WHEN** 请求未携带有效应用登录态且不处于允许 fallback 的开发环境
- **THEN** API SHALL 拒绝访问本人资料和签到接口

#### Scenario: 开发 fallback 被明确标记

- **WHEN** 本地开发环境使用临时本人上下文 fallback
- **THEN** 系统 SHALL 将该来源明确标记为 `temporary-dev-placeholder`
- **AND** 该机制 SHALL NOT 表达为真实微信登录能力

### Requirement: 用户成长资料契约

系统 SHALL 为每个应用内用户维护一份当前有效的用户成长资料，包含职业类型、职业文本、当前阵营、显示名、头像 key、默认阵营徽章 key、等级、经验、隐币、能量、连续签到、称号和徽章展示字段。

#### Scenario: 资料字段可供跨端复用

- **WHEN** 小程序端或 API 需要读取用户成长资料
- **THEN** 系统 SHALL 使用共享包中定义的用户成长资料类型、职业类型、阵营类型和响应契约

#### Scenario: 默认成长资料初始化

- **WHEN** 系统创建用户成长资料
- **THEN** 系统 SHALL 初始化等级、经验、隐币、能量、连续签到、称号 key、徽章 key 和头像 key
- **AND** 初始化结果 SHALL 可转换为小程序首页或个人中心可消费的资料快照

#### Scenario: 敏感边界

- **WHEN** 用户成长资料被社区、功能入口或公开资料引用
- **THEN** 系统 SHALL NOT 默认暴露薪资、工作时间或工作档案敏感字段

### Requirement: 首次创建隐者档案

小程序端 SHALL 在用户首次打开“我的”且当前用户没有成长资料时，引导用户创建隐者档案。

#### Scenario: 未创建资料打开我的页面

- **WHEN** 已登录用户打开“我的”且 API 返回资料未创建状态
- **THEN** 小程序端 SHALL 展示创建隐者档案流程
- **AND** 小程序端 SHALL 要求用户选择职业类型后才能创建资料

#### Scenario: 已创建资料打开我的页面

- **WHEN** 已登录用户打开“我的”且已存在有效成长资料
- **THEN** 小程序端 SHALL 展示用户成长资料、资源统计、每日签到、本人状态摘要、菜单区域和阵营插画卡
- **AND** 小程序端 SHALL NOT 展示空成就卡、空徽章列表或未定义完整玩法的成就入口

#### Scenario: 未登录打开我的页面

- **WHEN** 用户打开“我的”但没有有效应用登录态
- **THEN** 小程序端 SHALL 展示微信登录入口
- **AND** 小程序端 SHALL NOT 在未登录状态下创建真实成长资料

### Requirement: 职业类型决定初始阵营

系统 SHALL 根据用户首次创建隐者档案时选择的职业类型分配初始阵营。

#### Scenario: 数字与技术类选择分配键影隐者

- **WHEN** 用户选择开发、测试、运维或等价数字与技术类职业类型
- **THEN** 系统 SHALL 将初始阵营展示为键影隐者

#### Scenario: 运营与商业类选择分配运影隐者

- **WHEN** 用户选择运营、商业、市场、增长或等价运营与商业类职业类型
- **THEN** 系统 SHALL 将初始阵营展示为运影隐者

#### Scenario: 创意与内容类选择分配策影隐者

- **WHEN** 用户选择内容、设计、创意、策划或等价创意与内容类职业类型
- **THEN** 系统 SHALL 将初始阵营展示为策影隐者

#### Scenario: 现实与执行类选择分配行影隐者

- **WHEN** 用户选择销售、客服、行政、项目执行或等价现实与执行类职业类型
- **THEN** 系统 SHALL 将初始阵营展示为行影隐者

### Requirement: 默认昵称和默认头像

系统 SHALL 在首次创建隐者档案时生成阵营默认昵称，并分配阵营默认头像 key 与阵营默认徽章 key。

#### Scenario: 自动生成默认昵称

- **WHEN** 用户完成职业类型选择并创建隐者档案
- **THEN** 系统 SHALL 生成包含阵营名和随机数字后缀的默认昵称
- **AND** 默认昵称 SHALL 可用于个人中心资料卡展示

#### Scenario: 自动分配默认头像 key

- **WHEN** 系统创建用户成长资料
- **THEN** 系统 SHALL 根据用户阵营分配默认头像 key
- **AND** 小程序端 SHALL 可在真实头像资产未完成时使用该 key 展示占位或默认阵营头像

#### Scenario: 自动分配默认徽章 key

- **WHEN** 系统创建用户成长资料
- **THEN** 系统 SHALL 根据用户阵营分配默认徽章 key
- **AND** 小程序端 SHALL 在顶部身份卡和成就徽章区域展示该阵营徽章或明确占位

### Requirement: 阵营默认徽章和插画资产

系统 SHALL 为每个用户阵营提供稳定的默认头像 key、默认徽章 key 和个人页阵营插画 key。

#### Scenario: 首次创建资料分配阵营资产

- **WHEN** 系统首次创建用户成长资料
- **THEN** 系统 SHALL 根据用户初始阵营分配默认头像 key
- **AND** 系统 SHALL 根据用户初始阵营分配默认徽章 key
- **AND** 系统 SHALL 在资料快照中提供可用于个人页展示的当前头像 key 和当前徽章 key

#### Scenario: 个人页读取阵营插画 key

- **WHEN** 小程序“我的”页面拥有合法成长资料快照
- **THEN** 小程序端 SHALL 能根据当前阵营解析对应的个人页阵营插画 key
- **AND** 插画资源尚未接入时 SHALL 展示明确的阵营插画占位，不得展示无关演示场景作为真实阵营插画

#### Scenario: 阵营资产 key 稳定

- **WHEN** 小程序端更新头像、徽章或插画文件
- **THEN** 系统 SHALL 保持用户成长资料中的阵营资产 key 稳定
- **AND** 系统 SHALL NOT 因替换本地图片文件而要求迁移用户资料记录

### Requirement: 用户头像来源限制

系统 SHALL 禁止普通用户上传自定义头像，用户头像 SHALL 来自项目内置头像资产。

#### Scenario: 个人页不提供上传头像

- **WHEN** 用户打开“我的”页面
- **THEN** 页面 SHALL NOT 展示上传头像按钮、上传角标或本地图片选择入口
- **AND** 点击头像 SHALL NOT 调起相册或相机选择

#### Scenario: 后续头像替换来自头像库

- **WHEN** 后续开放头像替换能力
- **THEN** 系统 SHALL 仅允许用户从项目自制头像库中选择已解锁头像
- **AND** 系统 MAY 使用隐币作为头像解锁成本
- **AND** 系统 SHALL NOT 接受用户上传的自定义头像文件作为个人头像

### Requirement: 本人资料编辑

系统 SHALL 支持当前用户编辑本人角色资料，包括昵称、职业文本、职业类型和当前阵营。

#### Scenario: 修改昵称

- **WHEN** 已登录且已创建成长资料的用户提交合法新昵称
- **THEN** API SHALL 更新当前用户成长资料中的显示名
- **AND** API SHALL 返回更新后的用户成长资料快照
- **AND** 小程序端 SHALL 更新本地 `moyuxia.userProfileSnapshot`

#### Scenario: 拒绝非法昵称

- **WHEN** 用户提交空昵称、过长昵称、纯符号昵称或命中禁止规则的昵称
- **THEN** API SHALL 拒绝保存
- **AND** 小程序端 SHALL 保持原昵称

#### Scenario: 填写职业文本

- **WHEN** 已登录且已创建成长资料的用户提交合法职业文本
- **THEN** API SHALL 保存该职业文本到本人用户成长资料
- **AND** 职业文本 SHALL 仅作为本人资料和推荐阵营依据之一
- **AND** 系统 SHALL NOT 将职业文本公开到社区作者快照

#### Scenario: 修改职业类型

- **WHEN** 用户修改职业类型
- **THEN** 系统 SHALL 根据新的职业类型计算推荐阵营
- **AND** API SHALL 返回当前阵营与推荐阵营是否一致的信息
- **AND** 系统 SHALL NOT 因职业类型变化强制切换当前阵营

#### Scenario: 手动切换阵营

- **WHEN** 用户在“我的角色”中选择合法阵营并确认切换
- **THEN** API SHALL 更新当前用户成长资料中的阵营
- **AND** 系统 SHALL 同步更新默认阵营资产展示所需的头像、徽章或插画解析结果
- **AND** 后续社区发帖阵营权限 SHALL 以更新后的当前阵营为准

### Requirement: 本人资料 API

系统 SHALL 提供本人用户成长资料 API，用于读取、创建和更新当前登录用户的成长资料。

#### Scenario: 读取已有资料

- **WHEN** 已登录用户请求 `GET /me/profile` 且已创建成长资料
- **THEN** API SHALL 返回包含用户成长资料快照的成功响应

#### Scenario: 读取未创建资料状态

- **WHEN** 已登录用户请求 `GET /me/profile` 但尚未创建成长资料
- **THEN** API SHALL 返回资料未创建状态
- **AND** API SHALL NOT 返回静态演示资料作为真实资料

#### Scenario: 创建本人资料

- **WHEN** 已登录用户提交合法职业类型创建本人资料
- **THEN** API SHALL 创建当前用户的成长资料
- **AND** API SHALL 返回创建后的用户成长资料快照

#### Scenario: 重复创建本人资料

- **WHEN** 已创建成长资料的用户再次提交创建资料请求
- **THEN** API SHALL NOT 创建第二份当前有效成长资料
- **AND** API SHALL 返回可识别的重复创建结果或当前已有资料

#### Scenario: 更新本人资料

- **WHEN** 已登录且已创建成长资料的用户提交合法昵称、职业文本、职业类型或当前阵营更新
- **THEN** API SHALL 保存允许更新的字段
- **AND** API SHALL 返回更新后的用户成长资料快照

### Requirement: 小程序资料快照缓存

小程序端 SHALL 缓存合法的本人用户成长资料快照，用于“我的”页面首屏展示和 API 不可用时的降级体验。

#### Scenario: 本地快照首屏展示

- **WHEN** 用户打开“我的”且本地存在合法 `moyuxia.userProfileSnapshot`
- **THEN** 小程序端 SHALL 可以先基于本地快照展示资料
- **AND** 小程序端 SHALL 在后台同步本人资料 API

#### Scenario: API 同步成功更新缓存

- **WHEN** 本人资料 API 返回合法成长资料快照
- **THEN** 小程序端 SHALL 更新 `moyuxia.userProfileSnapshot`
- **AND** “我的”页面 SHALL 使用最新快照刷新展示

#### Scenario: 无缓存且 API 不可用

- **WHEN** 用户打开“我的”且没有本地合法快照，同时本人资料 API 不可用
- **THEN** 小程序端 SHALL 展示可恢复的错误或重试状态
- **AND** 小程序端 SHALL NOT 展示静态演示资料作为真实资料

### Requirement: 我的页面成长资料展示

小程序“我的”页面 SHALL 使用真实用户成长资料或合法本地快照渲染资料卡、等级经验、关注数、粉丝数、连续签到天数、签到状态、称号、阵营徽章、基础身份徽章和阵营插画，不得使用无关演示数据替代真实个人状态。

#### Scenario: 展示基础身份

- **WHEN** “我的”页面拥有合法成长资料快照
- **THEN** 页面 SHALL 展示用户默认昵称或用户修改后的昵称、阵营、等级、经验进度、称号、头像和阵营徽章
- **AND** 页面 SHALL 在昵称右侧展示可点击的改名 icon，并进入“我的角色”二级页

#### Scenario: 展示三项状态数据

- **WHEN** “我的”页面拥有合法成长资料快照
- **THEN** 页面 SHALL 在顶部身份区域展示关注数、粉丝数和连续签到天数
- **AND** 关注数 SHALL 可点击进入本人关注列表
- **AND** 粉丝数 SHALL 可点击进入本人粉丝列表
- **AND** 连续签到天数 SHALL 保持既有展示语义，不被替换为关注关系入口
- **AND** 页面 SHALL 在每日签到成功后使用 API 返回的资料快照刷新连续签到天数

#### Scenario: 展示阵营插画

- **WHEN** “我的”页面拥有合法成长资料快照
- **THEN** 页面 SHALL 根据用户当前阵营展示对应阵营插画或对应阵营插画占位
- **AND** 页面 SHALL NOT 继续使用按时间变化的无关修行场景表达用户阵营身份

#### Scenario: 展示称号和徽章边界

- **WHEN** 用户尚未拥有真实称号商店、徽章解锁能力或完整成就系统
- **THEN** 页面 SHALL 仅展示当前称号、当前阵营徽章和当前已装备徽章作为基础身份资产
- **AND** 页面 SHALL NOT 表达称号购买、徽章商店、完整成就系统或复杂解锁能力已经开放

### Requirement: 隐币和能量普通 UI 暂时隐藏

小程序普通用户界面 SHALL 暂时隐藏隐币和能量展示，直到隐币和能量的获取来源、消耗场景、用途和展示位置由后续 change 明确定义。

#### Scenario: 我的页不展示隐币和能量

- **WHEN** 已登录且已创建隐者档案的用户打开“我的”页
- **THEN** 页面 SHALL NOT 展示隐币或能量统计
- **AND** 原资源统计区域 SHALL 展示关注数、粉丝数和连续签到天数

#### Scenario: 公开页面不展示隐币和能量

- **WHEN** 普通用户打开公开个人页、关注列表、粉丝列表、社区作者入口或其他公开资料区域
- **THEN** 页面和普通公开 API 响应 SHALL NOT 展示隐币、能量或其他未开放成长资源
- **AND** 系统 SHALL NOT 将隐币或能量作为公开身份、排行榜、关注关系或社区作者快照字段

#### Scenario: 内部成长资料可继续保留字段

- **WHEN** 系统维护用户成长资料、签到奖励或后续成长资源字段
- **THEN** 系统 MAY 继续在内部资料和本人接口中保留隐币与能量数据
- **AND** 普通 UI 是否恢复展示 SHALL 由后续 change 定义

### Requirement: 签到持久化验证边界

系统 SHALL 明确区分正式数据库持久化与开发期内存 fallback，避免把开发期内存状态表达为正式签到保存能力。

#### Scenario: 数据库配置下签到持久化

- **WHEN** API 配置有效 `DATABASE_URL` 且用户完成当日首次签到
- **THEN** 系统 SHALL 将签到记录和更新后的用户成长资料持久化到数据库
- **AND** API 重启后再次读取本人资料 SHALL 返回已持久化的经验、隐币、能量、连续签到和最后签到日期

#### Scenario: 未配置数据库使用内存 fallback

- **WHEN** API 未配置 `DATABASE_URL`
- **THEN** 系统 SHALL 仅使用开发期内存存储用户资料和签到记录
- **AND** 文档或日志 SHALL 明确该状态在 API 进程重启后不会持久保留

#### Scenario: mock 登录身份稳定

- **WHEN** 使用 mock 微信登录验证签到持久化
- **THEN** 测试环境 SHALL 使用稳定 mock openid 或稳定应用用户身份
- **AND** 系统 SHALL NOT 因每次登录 code 变化而把同一测试用户识别为不同用户

### Requirement: 每日签到

系统 SHALL 支持当前用户每日签到第一版，同一用户同一业务日期只能领取一次签到奖励。

#### Scenario: 当日首次签到

- **WHEN** 已登录且已创建成长资料的用户在当日首次请求签到
- **THEN** 系统 SHALL 发放第一版签到奖励
- **AND** 系统 SHALL 更新经验、隐币、能量、连续签到天数和最后签到日期
- **AND** API SHALL 返回更新后的用户成长资料快照和签到结果

#### Scenario: 当日重复签到

- **WHEN** 用户在同一业务日期重复请求签到
- **THEN** 系统 SHALL NOT 重复发放奖励
- **AND** API SHALL 返回已签到状态和当前用户成长资料快照

#### Scenario: 未创建资料签到

- **WHEN** 已登录但未创建成长资料的用户请求签到
- **THEN** API SHALL 拒绝签到
- **AND** API SHALL 提示需要先创建隐者档案

### Requirement: 等级经验第一版

系统 SHALL 在签到奖励或后续成长事件增加经验后，按第一版等级规则更新用户等级和经验进度。

#### Scenario: 经验未达到升级阈值

- **WHEN** 用户获得经验但总经验未达到下一等级阈值
- **THEN** 系统 SHALL 保持当前等级
- **AND** 系统 SHALL 返回当前经验进度

#### Scenario: 经验达到升级阈值

- **WHEN** 用户获得经验后达到或超过下一等级阈值
- **THEN** 系统 SHALL 更新用户等级
- **AND** 系统 SHALL 返回新的等级和经验进度

#### Scenario: 等级规则可复用

- **WHEN** API 或小程序端需要展示等级经验进度
- **THEN** 系统 SHALL 复用共享包中的等级规则或等级快照契约

### Requirement: 微信登录配置安全

系统 SHALL 通过环境变量或安全配置读取微信小程序登录所需密钥，并 SHALL NOT 在代码、样例或文档中写入真实密钥。

#### Scenario: 配置缺失

- **WHEN** 微信登录所需配置缺失
- **THEN** API SHALL 返回可识别的配置错误或使用明确允许的开发 fallback
- **AND** 系统 SHALL NOT 使用硬编码真实密钥

#### Scenario: 环境变量样例

- **WHEN** 项目提供微信登录相关环境变量样例
- **THEN** 样例 SHALL 使用占位值
- **AND** 样例 SHALL NOT 包含真实微信密钥

### Requirement: 首页身份读取用户成长资料

小程序首页 SHALL 使用真实用户成长资料或合法本地快照展示顶部身份状态，不得继续使用静态阵营和等级文案作为真实身份。

#### Scenario: 未登录首页身份

- **WHEN** 用户打开首页且没有有效应用登录态
- **THEN** 首页顶部身份区 SHALL 展示未登录或未进入隐者大陆的状态
- **AND** 首页 SHALL 提供登录或进入“我的”页登录的入口
- **AND** 首页 SHALL NOT 展示静态“键影隐者 · 一阶”作为真实用户身份

#### Scenario: 已登录未建档首页身份

- **WHEN** 用户已登录但尚未创建用户成长资料
- **THEN** 首页顶部身份区 SHALL 展示隐者档案未创建状态
- **AND** 首页 SHALL 提供创建隐者档案的入口
- **AND** 首页 SHALL NOT 展示默认阵营、等级或称号作为已创建身份

#### Scenario: 已建档首页身份

- **WHEN** 首页拥有远端返回或本地缓存的合法用户成长资料快照
- **THEN** 首页顶部身份区 SHALL 展示用户昵称、当前阵营和等级语义
- **AND** 展示内容 SHALL 与用户成长资料快照保持一致

#### Scenario: 用户资料同步失败但有快照

- **WHEN** 首页同步本人用户资料失败
- **AND** 本地存在合法 `moyuxia.userProfileSnapshot`
- **THEN** 首页 MAY 使用本地快照展示身份
- **AND** 首页 SHALL 提示身份资料同步失败或处于本地快照展示状态

### Requirement: 建档完成后的首跑下一步

小程序端 SHALL 在用户创建隐者档案成功后明确引导配置工作档案，而不是只停留在个人资料展示。

#### Scenario: 创建档案成功后展示下一步

- **WHEN** 用户在“我的”页成功创建隐者档案
- **THEN** 小程序端 SHALL 保存返回的合法用户成长资料快照
- **AND** 页面 SHALL 提供配置工作档案的明确入口或行动按钮

#### Scenario: 已建档用户再次进入我的页

- **WHEN** 已创建隐者档案的用户进入“我的”页
- **THEN** 页面 SHALL 继续展示真实成长资料、签到、资源、本人状态摘要和高价值入口列表
- **AND** 页面 SHALL NOT 将未定义完整玩法的“我的成就”作为常用入口或首屏占位展示

### Requirement: 个人中心成就占位下线

小程序“我的”页 SHALL 在成就系统玩法未定义完整前下线成就首屏占位和菜单入口。

#### Scenario: 成就玩法未定义时不展示

- **WHEN** 用户打开“我的”页
- **THEN** 页面 SHALL NOT 展示空成就卡、空徽章列表或 `我的成就` 菜单入口
- **AND** 页面 SHALL NOT 暗示完整成就系统、成就奖励或复杂解锁能力已经开放

#### Scenario: 保留基础徽章身份展示

- **WHEN** 用户拥有当前阵营徽章或当前装备徽章
- **THEN** 页面 MAY 在身份卡中展示该徽章作为身份资产
- **AND** 页面 SHALL NOT 将该基础徽章展示扩展表达为已上线成就系统

### Requirement: 个人中心本人状态摘要

小程序“我的”页 SHALL 使用真实可用信息替换原成就占位区域，展示本人状态摘要。

#### Scenario: 展示工作档案摘要

- **WHEN** “我的”页拥有合法工作档案快照或可读取本人工作档案
- **THEN** 本人状态摘要 SHALL 展示上班时段、发薪日或工作日规则等非月薪摘要
- **AND** 本人状态摘要 SHALL 提供进入工作设置的操作入口

#### Scenario: 展示今日生存摘要

- **WHEN** “我的”页可以读取本人今日生存消耗摘要
- **THEN** 本人状态摘要 SHALL 展示 `隐者食堂`、`下午续命` 和 `通勤` 的轻量金额摘要
- **AND** 本人状态摘要 SHALL 提供进入生存账本详情的操作入口

#### Scenario: 不展示敏感月薪

- **WHEN** “我的”页展示工作档案摘要
- **THEN** 页面 SHALL NOT 展示用户月薪金额
- **AND** 页面 SHALL NOT 将薪资字段公开到社区或其他用户可见区域

### Requirement: 个人中心下半区入口增强

小程序“我的”页 SHALL 保留左侧入口列表与右侧阵营插画卡的下半区排版，并在左侧入口中展示真实可用功能和语义 icon。

#### Scenario: 左侧入口列表展示真实功能

- **WHEN** 用户打开“我的”页且已创建用户成长资料
- **THEN** 下半区左侧入口列表 SHALL 展示 `我的角色`、`工作设置`、`我的帖子`、`收藏帖子`、`论坛消息` 和 `生存账本`
- **AND** 页面 SHALL NOT 在该列表中展示称号商店、皮肤商店、赛季排行或未定义完整玩法的成就入口

#### Scenario: 每个入口展示语义 icon

- **WHEN** 下半区左侧入口列表渲染
- **THEN** 每个入口 SHALL 在标题前展示与功能语义匹配的 icon
- **AND** 页面 SHALL NOT 使用完全相同的无语义圆点替代所有入口 icon

#### Scenario: 右侧阵营插画保留

- **WHEN** 下半区右侧阵营插画卡渲染
- **THEN** 页面 SHALL 保留当前阵营插画或对应阵营插画占位
- **AND** 页面 MAY 在插画上叠加当前阵营修行状态、今日状态或低调态短文案

### Requirement: 我的页视觉资产增强

我的页 SHALL 使用阵营资产、语义菜单 icon 和统一状态视觉表达身份、成长资源、入口列表、未登录、未建档和错误状态。

#### Scenario: 阵营资产保持身份一致

- **WHEN** 我的页拥有合法用户成长资料快照
- **THEN** 页面 SHALL 展示当前阵营头像、当前阵营徽章和当前阵营插画
- **AND** 页面 SHALL NOT 使用无关演示场景替代真实阵营插画

#### Scenario: 菜单入口展示语义 icon

- **WHEN** 我的页展示 `我的角色`、`工作设置`、`我的帖子`、`收藏帖子`、`论坛消息` 和 `生存账本`
- **THEN** 每个入口 SHALL 展示与功能语义匹配的 icon 或同风格 fallback
- **AND** 页面 SHALL NOT 使用完全相同的无语义圆点替代所有入口 icon

#### Scenario: 未登录展示登录引导视觉

- **WHEN** 用户未登录并打开我的页
- **THEN** 页面 SHALL 展示统一登录引导视觉和登录操作
- **AND** 页面 SHALL NOT 展示静态演示昵称、演示等级或演示阵营作为真实身份

#### Scenario: 未建档展示建档引导视觉

- **WHEN** 用户已登录但尚未创建隐者档案
- **THEN** 页面 SHALL 展示统一建档引导视觉和职业/阵营创建入口
- **AND** 页面 SHALL NOT 展示已创建用户才拥有的资源、签到或阵营插画卡

#### Scenario: 资料同步失败展示可恢复错误

- **WHEN** 我的页同步资料失败且没有合法本地快照
- **THEN** 页面 SHALL 展示统一错误状态视觉和重试操作
- **AND** 页面 SHALL NOT 使用开发态占位资料替代真实资料

### Requirement: 公开资料字段内容安全审核

系统 SHALL 对会进入公开作者快照或普通用户可见区域的本人资料字段执行内容安全审核，并 SHALL 在审核未通过时阻止保存或阻止公开展示。

#### Scenario: 修改昵称时审核公开字段

- **WHEN** 已登录且已创建成长资料的用户提交新昵称
- **THEN** 系统 SHALL 对昵称执行本地规则和微信文本内容安全或等价供应商审核
- **AND** 明确违规昵称 SHALL 被拒绝保存
- **AND** 无法确认的昵称 SHALL 被拒绝保存或进入人工处理边界

#### Scenario: 合规昵称保存并更新快照

- **WHEN** 用户提交的昵称通过内容安全审核
- **THEN** API SHALL 更新当前用户成长资料中的显示名
- **AND** API SHALL 返回更新后的用户成长资料快照
- **AND** 后续社区作者快照 SHALL 使用最新合规昵称

#### Scenario: 私密职业文本不公开

- **WHEN** 用户编辑职业文本
- **THEN** 系统 SHALL 继续将职业文本仅作为本人资料和推荐阵营依据之一
- **AND** 系统 SHALL NOT 将职业文本公开到社区作者快照
- **AND** 系统 SHALL NOT 因职业文本审核结果向其他用户暴露内部风险细节

### Requirement: 本人社区发布资格状态

系统 SHALL 为本人社区发布流程提供最小必要的发布资格状态，包含隐私同意是否满足当前版本、手机号是否已验证和社区写操作是否可继续。

#### Scenario: 本人读取发布资格状态

- **WHEN** 已登录用户进入发帖、评论或回复流程
- **THEN** 小程序 MAY 读取本人社区发布资格状态
- **AND** 状态 SHALL 区分未同意当前隐私政策、未完成手机号验证和已满足发布门槛
- **AND** 状态 SHALL NOT 包含完整手机号、手机号尾号、微信 openid、微信 unionid 或微信 sessionKey

#### Scenario: 本人页面不常驻展示手机号验证状态

- **WHEN** 用户打开“我的”页、我的角色页或个人资料页
- **THEN** 页面 SHALL NOT 将手机号验证状态作为常驻身份信息展示
- **AND** 页面 SHALL NOT 展示手机号尾号或手机号认证徽章

### Requirement: 公开个人主页 IP 属地边界

公开个人主页 SHALL 展示用户最近一次公开发布行为对应的 IP 属地，并 SHALL 继续过滤手机号验证状态、完整手机号、明文 IP 和微信身份字段。

#### Scenario: 公开个人主页展示 IP 属地

- **WHEN** 用户打开某个用户的公开个人主页
- **AND** 该用户存在可公开的发帖、评论或回复发布记录
- **THEN** 页面 SHALL 展示最近一次公开发布行为的 `IP属地`
- **AND** 页面 SHALL NOT 展示完整 IP、城市、区县或街道

#### Scenario: 公开个人主页没有可公开发布记录

- **WHEN** 用户打开某个用户的公开个人主页
- **AND** 该用户没有可公开的发帖、评论或回复发布记录
- **THEN** 页面 MAY 隐藏 IP 属地字段或展示 `IP属地：未知`
- **AND** 页面 SHALL NOT 根据登录、浏览或后台请求伪造属地

#### Scenario: 公开个人主页过滤手机号验证状态

- **WHEN** API 返回公开个人主页资料
- **THEN** 响应 SHALL NOT 包含手机号验证状态、完整手机号、手机号尾号、微信 openid、微信 unionid、微信 sessionKey 或用户登录态
- **AND** 响应 MAY 包含公开作者身份、阵营、等级、称号、徽章和降精度 IP 属地标签

#### Scenario: 最近公开发布 IP 属地不使用已移除内容

- **WHEN** 用户最近的发帖、评论或回复已被隐藏、移除或不再公开
- **THEN** 公开个人主页 SHALL 使用仍可公开内容中的最近一次发布 IP 属地
- **AND** 若不存在仍可公开内容，页面 MAY 隐藏 IP 属地字段或展示 `IP属地：未知`

### Requirement: 我的页社交三项数据排版 polish

小程序“我的”页顶部身份区域 SHALL 以稳定、紧凑且可读的方式展示关注数、粉丝数和连续签到天数。

#### Scenario: 三项数据稳定展示

- **WHEN** “我的”页拥有合法用户成长资料和关注统计
- **THEN** 页面 SHALL 在顶部身份区域展示关注数、粉丝数和连续签到天数
- **AND** 三项 SHALL 使用三列或等价稳定布局
- **AND** 页面 SHALL NOT 因隐藏隐币和能量留下空白占位、未开放文案或资源玩法暗示

#### Scenario: 关注和粉丝入口清晰

- **WHEN** 用户查看“我的”页三项数据
- **THEN** 关注数和粉丝数 SHALL 具备可点击入口暗示和按压反馈
- **AND** 点击关注数 SHALL 进入本人关注列表
- **AND** 点击粉丝数 SHALL 进入本人粉丝列表
- **AND** 连续签到 SHALL NOT 被展示为关注关系入口

#### Scenario: 连续签到保持成长反馈

- **WHEN** 用户查看连续签到数据或执行每日签到
- **THEN** 连续签到 SHALL 继续表达签到连续天数和签到反馈
- **AND** 页面 SHALL NOT 将连续签到替换为关注、粉丝、隐币、能量或其他未定义资源统计

#### Scenario: 长数字不破坏身份卡

- **WHEN** 关注数、粉丝数或连续签到天数为长数字
- **THEN** 页面 SHALL 使用固定容器、收缩、省略、换行或 `9999+` 等展示策略保持布局稳定
- **AND** 展示策略 SHALL NOT 遮挡头像、昵称、徽章、签到按钮、工作档案入口或后续内容
- **AND** 若使用缩写展示，服务端和本地状态 SHALL 保留完整数值

### Requirement: 我的页普通 UI 继续隐藏隐币和能量

小程序普通用户界面 SHALL 继续隐藏隐币和能量统计，除非后续 change 明确定义完整玩法和展示位置。

#### Scenario: 我的页不恢复隐币能量统计

- **WHEN** 用户打开“我的”页
- **THEN** 页面 SHALL NOT 在顶部三项数据、菜单、卡片、icon 或插画中展示隐币或能量统计
- **AND** 签到成功后 MAY 更新内部成长资料字段
- **AND** 页面 SHALL NOT 将内部隐币或能量字段表达为已开放玩法

#### Scenario: 公开社交页面不展示隐币能量

- **WHEN** 用户打开公开个人页、关注列表或粉丝列表
- **THEN** 页面 SHALL NOT 展示目标用户或当前用户的隐币、能量、资源余额、成长资源 icon 或资源玩法入口
- **AND** API 响应 SHALL NOT 将隐币或能量返回给普通公开社交页面
