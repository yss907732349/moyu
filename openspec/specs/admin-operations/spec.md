# admin-operations Specification

## Purpose

`admin-operations` 用于定义后台运营中心的聚合工作台、统一审核队列、审核详情、操作入口和敏感字段边界。

## Requirements

### Requirement: 后台运营中心工作台

系统 SHALL 提供后台运营中心工作台，用于集中展示社区审核、日报评论复核、举报处理和日报发布待办，并 SHALL 区分评论/回复待人工复核数量与自动处理数量。

#### Scenario: 读取工作台待办

- **WHEN** 携带有效 `x-admin-token` 的后台请求读取运营工作台
- **THEN** API SHALL 返回待审核帖子、待复核评论/回复、待处理举报、日报评论待复核和待发布日报的计数
- **AND** API SHALL 返回评论/回复自动通过、自动驳回和人工复核的统计摘要
- **AND** API SHALL 返回最近待办列表，供后台首页快速进入处理流程

#### Scenario: 工作台入口跳转队列

- **WHEN** 后台用户点击工作台中的社区、日报评论或举报待办卡片
- **THEN** 后台 SHALL 进入统一审核队列对应分组
- **AND** 社区帖子、社区评论、社区回复和日报评论待办 SHALL 默认进入内容审核队列
- **AND** 举报待办 SHALL 默认进入举报审核队列
- **AND** 后台 SHALL 自动应用与该卡片对应的来源、类型、状态或举报状态筛选

### Requirement: 统一审核队列

系统 SHALL 提供统一审核队列事实源，聚合社区帖子、社区评论、社区回复、社区举报和日报评论的后台处理事项，并 SHALL 在后台界面和 API 查询中区分内容审核队列、举报审核队列和全部待办。内容审核队列 SHALL 主要承载帖子、评论、回复和日报评论等需要人工判断的内容事项；举报审核队列 SHALL 独立承载社区举报案件，避免与正常发帖审核混杂。

#### Scenario: 读取统一审核队列

- **WHEN** 携带有效 `x-admin-token` 的后台请求读取统一审核队列
- **THEN** API SHALL 返回统一队列项列表
- **AND** 每个队列项 SHALL 包含来源、对象类型、业务对象 ID、状态、内容摘要、作者公开快照、创建时间、AI 审核摘要、低成本审核摘要、审核决策、用户风险原因、可用操作和详情入口标识
- **AND** API SHALL 支持 `reviewGroup` 查询分组，稳定取值至少包含 `content`、`report` 和 `all`

#### Scenario: 默认查看内容审核队列

- **WHEN** 后台用户首次进入审核队列或未显式选择 `reviewGroup`
- **THEN** 后台 SHALL 默认展示内容审核队列
- **AND** API SHALL 排除社区举报队列项
- **AND** 后台 SHALL 保留切换到举报审核和全部待办的入口

#### Scenario: 查看举报审核队列

- **WHEN** 后台用户切换到举报审核队列或请求 `reviewGroup=report`
- **THEN** API SHALL 只返回社区举报或举报案件队列项
- **AND** 后台 SHALL 展示举报状态、举报原因、目标类型、优先级和举报处理相关筛选
- **AND** 后台 SHALL NOT 将举报案件混入普通内容审核列表

#### Scenario: 按来源类型状态筛选

- **WHEN** 后台用户按来源、对象类型或状态筛选队列
- **THEN** API SHALL 只返回符合筛选条件的队列项
- **AND** 支持的来源 SHALL 至少包含社区和日报
- **AND** 筛选 SHALL 与当前 `reviewGroup` 共同生效

#### Scenario: 按 AI 风险筛选

- **WHEN** 后台用户按 AI 风险标签筛选队列
- **THEN** API SHALL 返回包含对应归一化风险标签的队列项
- **AND** API SHALL NOT 返回 AI 原始 prompt、供应商原始响应或密钥配置

#### Scenario: 自动处理评论不进入待办队列

- **WHEN** 社区评论或回复的审核决策为 `auto_approve` 或 `auto_reject`
- **THEN** API SHALL NOT 将该评论或回复作为待人工复核队列项返回
- **AND** 后台 SHALL 仍可在社区内容治理详情中查看自动处理摘要

### Requirement: 举报人工审核队列

后台运营中心 SHALL 将社区举报作为独立人工审核事项展示，并 SHALL 支持按举报状态、举报原因、目标类型和优先级查看待处理举报。举报队列 SHALL 优先展示可帮助运营判断的最小必要上下文，不得暴露普通后台不需要的敏感字段。

#### Scenario: 查看待处理举报队列

- **WHEN** 携带有效 `x-admin-token` 的后台用户读取举报审核队列
- **THEN** API SHALL 返回待处理举报或举报案件条目
- **AND** 每个条目 SHALL 包含举报原因分类、补充说明摘要、目标类型、目标 ID、目标内容摘要、当前举报状态、创建时间和可用操作
- **AND** API SHALL NOT 返回微信 openid、unionid、sessionKey、应用登录态、薪资、工作档案、生存账单、CPS 来源、供应商密钥或完整供应商原始响应

#### Scenario: 按优先级展示举报

- **WHEN** 举报原因是隐私泄露、违法违规，或同一目标在短时间内收到多名用户举报
- **THEN** 后台 SHALL 将该举报或举报案件标记为高优先级
- **AND** 高优先级 SHALL 只影响队列排序和提示，不 SHALL 自动隐藏公开内容

#### Scenario: 合并同一目标举报

- **WHEN** 多名用户举报同一帖子、评论或回复
- **THEN** 后台 SHALL 能够按同一目标聚合展示举报数量、原因分布、首次举报时间和最近举报时间
- **AND** 后台 SHALL 支持进入聚合目标的举报明细和内容上下文

### Requirement: 举报审核详情

后台 SHALL 提供社区举报审核详情，使运营人员在处理前查看举报理由、被举报内容、关联上下文、作者治理状态、同目标举报明细和可用操作。

#### Scenario: 查看举报详情

- **WHEN** 后台用户打开社区举报详情
- **THEN** API SHALL 返回举报原因分类、补充说明、举报对象类型、举报对象 ID、目标内容摘要、目标内容当前状态、被举报作者公开快照、举报状态和可用处理操作
- **AND** API SHALL 支持后台进入被举报对象所在帖子的治理详情查看上下文

#### Scenario: 查看举报目标上下文

- **WHEN** 被举报对象是评论或回复
- **THEN** API SHALL 返回足够定位上下文的帖子 ID、帖子标题、父评论 ID 和评论区摘要
- **AND** API SHALL NOT 要求运营人员只能通过公开小程序页面查找上下文

#### Scenario: 查看已处理举报详情

- **WHEN** 后台用户打开已处理举报详情
- **THEN** API SHALL 返回处理动作、处理人、处理时间、处理备注和内容治理结果
- **AND** API SHALL NOT 允许后台重复处理状态已经变化的举报

### Requirement: 举报人工处理操作

后台 SHALL 支持对待处理举报执行保留内容、隐藏内容、移除内容和标记误报等人工处理操作，并 SHALL 在状态已变化时拒绝重复处理。

#### Scenario: 保留被举报内容

- **WHEN** 后台用户处理待处理举报并选择保留内容
- **THEN** API SHALL 将举报状态更新为内容保留
- **AND** API SHALL 保留被举报内容的当前状态
- **AND** API SHALL 记录处理备注

#### Scenario: 隐藏被举报内容

- **WHEN** 后台用户处理待处理举报并选择隐藏内容
- **THEN** API SHALL 将举报状态更新为已隐藏处理
- **AND** API SHALL 将对应帖子、评论或回复从公开侧移除
- **AND** API SHALL 创建对应社区治理审计记录

#### Scenario: 移除被举报内容

- **WHEN** 后台用户处理待处理举报并选择移除内容
- **THEN** API SHALL 将举报状态更新为已移除处理
- **AND** API SHALL 将对应帖子、评论或回复更新为 `removed`
- **AND** API SHALL 保留原始内容、举报记录和治理审计记录

#### Scenario: 标记误报

- **WHEN** 后台用户处理待处理举报并选择标记误报
- **THEN** API SHALL 将举报状态更新为误报
- **AND** API SHALL 保留被举报内容的当前状态
- **AND** API SHALL NOT 将该举报计入被举报作者的有效举报

#### Scenario: 重复处理举报

- **WHEN** 后台用户处理状态已经变化的举报
- **THEN** API SHALL 拒绝该操作
- **AND** API SHALL 返回后台可理解的状态已变化提示

### Requirement: 统一审核详情

系统 SHALL 提供统一审核详情数据，使后台用户可以在操作前查看原文、上下文、风险原因、审核决策和可用操作。

#### Scenario: 查看内容详情

- **WHEN** 后台用户打开社区帖子、社区评论、社区回复或日报评论的审核详情
- **THEN** API SHALL 返回原文全文、作者公开快照、当前状态、创建时间、AI 风险标签、AI 简短原因、低成本审核摘要、审核决策、用户风险原因、人工复核原因和可用操作
- **AND** API SHALL 返回足够定位上下文的关联对象信息

#### Scenario: 查看举报详情

- **WHEN** 后台用户打开社区举报详情
- **THEN** API SHALL 返回举报原因分类、举报对象类型、举报对象 ID、被举报内容摘要、当前举报状态和可用处理操作
- **AND** API SHALL 支持后台进入被举报对象的详情查看上下文

### Requirement: 后台统一操作体验

后台 SHALL 在统一审核队列和详情中提供通过、驳回、隐藏、处理举报和查看详情等操作入口，并 SHALL 只展示当前对象允许的操作。

#### Scenario: 内容审核操作

- **WHEN** 队列项是可人工处理的帖子、评论、回复或日报评论
- **THEN** 后台 SHALL 基于队列项 `availableActions` 展示通过、驳回、隐藏或查看详情入口
- **AND** 后台执行操作时 SHALL 携带 `x-admin-token`

#### Scenario: 举报处理操作

- **WHEN** 队列项是举报审核队列中的待处理举报
- **THEN** 后台 SHALL 基于队列项 `availableActions` 展示保留内容、隐藏内容、标记误报和查看详情入口
- **AND** 后台 SHALL NOT 将举报误表达为普通内容的通过或驳回操作

### Requirement: 后台令牌权限

后台运营中心的聚合读取接口和相关操作入口 SHALL 继续使用 `x-admin-token` 校验，不引入完整账号系统。

#### Scenario: 无令牌读取运营中心

- **WHEN** 请求未携带有效 `x-admin-token` 访问后台运营中心接口
- **THEN** API SHALL 拒绝请求
- **AND** API SHALL NOT 返回待审内容、举报内容、日报草稿或任何后台待办数据

#### Scenario: 有效令牌读取运营中心

- **WHEN** 请求携带有效 `x-admin-token` 访问后台运营中心接口
- **THEN** API SHALL 返回对应后台运营数据
- **AND** API SHALL 只返回后台第一阶段允许展示的字段

### Requirement: 后台敏感字段边界

后台运营中心响应 SHALL 过滤与运营处理无关的敏感字段和内部供应商细节。

#### Scenario: 队列响应不暴露用户私密数据

- **WHEN** 后台读取工作台、审核队列或审核详情
- **THEN** 响应 SHALL NOT 包含薪资、工作档案、上班时间、隐藏模式、生存账单、消费统计、CPS 平台来源、微信 openid、微信 unionid 或用户登录态
- **AND** 作者信息 SHALL 使用公开作者快照或后台允许展示的最小字段

#### Scenario: 队列响应不暴露 AI 内部细节

- **WHEN** 后台读取包含 AI 审核结果的队列项或详情
- **THEN** 响应 SHALL 可以包含归一化风险标签、AI 简短原因、审核来源、置信度和人工复核原因
- **AND** 响应 SHALL NOT 包含 AI 原始 prompt、供应商完整原始响应、供应商密钥或代理配置

### Requirement: 后台运营验证脚本

项目 SHALL 提供后台运营中心验证脚本，覆盖入口、队列、操作按钮、后台令牌和敏感字段边界。

#### Scenario: 运行后台运营验证

- **WHEN** 开发者运行 `pnpm verify:admin-operations`
- **THEN** 验证 SHALL 检查后台运营中心入口、内容审核队列、举报审核队列、审核操作入口、举报处理入口和 `x-admin-token` 请求头
- **AND** 验证 SHALL 检查后台聚合响应不包含敏感字段黑名单

### Requirement: 社区媒体资产风险控制

社区媒体资产入口 SHALL 在第一版提供基础大小限制、类型限制、上传频率限制和孤儿资产清理能力，避免未来论坛图片能力开放后无限占用服务端资源。

#### Scenario: 上传媒体资产时校验大小和类型

- **WHEN** 已登录且已创建档案的用户上传社区媒体资产
- **THEN** API SHALL 只接受受支持的图片 MIME 类型
- **AND** API SHALL 拒绝超过单图大小上限的媒体资产

#### Scenario: 上传媒体资产时限流

- **WHEN** 同一用户在单个业务日内上传社区媒体资产超过配置上限
- **THEN** API SHALL 拒绝继续上传
- **AND** API SHALL 返回可理解的限流提示

#### Scenario: 清理未绑定媒体资产

- **WHEN** 后台或定时任务触发社区媒体资产清理
- **THEN** 系统 SHALL 将超过保留时间且未绑定帖子的已上传媒体资产标记为隐藏或不可公开
- **AND** 系统 SHALL 返回本次清理的资产数量，供后台或日志记录

### Requirement: 后台社区内容治理入口

后台运营中心 SHALL 提供社区内容治理入口，并 SHALL 将其与统一审核队列区分。

#### Scenario: 工作台展示社区内容治理入口

- **WHEN** 后台用户打开运营中心工作台
- **THEN** 后台 SHALL 展示进入社区内容治理总览的入口
- **AND** 该入口 SHALL NOT 被表达为仅处理待审核事项的统一审核队列

#### Scenario: 从审核详情进入治理详情

- **WHEN** 后台用户在统一审核队列中查看社区帖子、评论、回复或举报
- **THEN** 后台 MAY 提供进入社区内容治理详情的入口
- **AND** 治理详情 SHALL 能展示该对象所在帖子的完整后台上下文

### Requirement: 社区治理后台令牌权限

后台社区内容治理接口 SHALL 继续使用 `x-admin-token` 校验，不引入完整后台账号系统。

#### Scenario: 无令牌访问社区内容治理

- **WHEN** 请求未携带有效 `x-admin-token` 访问社区内容治理总览、搜索、详情或治理操作接口
- **THEN** API SHALL 拒绝请求
- **AND** API SHALL NOT 返回历史帖子、评论、回复、举报、作者治理状态或审计记录

#### Scenario: 有效令牌访问社区内容治理

- **WHEN** 请求携带有效 `x-admin-token` 访问社区内容治理接口
- **THEN** API SHALL 允许读取或执行对应后台治理能力
- **AND** API SHALL 只返回后台第一阶段允许展示的字段

### Requirement: 社区治理敏感字段边界

后台社区内容治理响应 SHALL 过滤与社区治理无关的用户私密数据、内部身份字段和供应商细节。

#### Scenario: 社区治理响应不暴露用户私密数据

- **WHEN** 后台读取社区内容总览、搜索结果、帖子治理详情、作者治理详情或治理审计记录
- **THEN** 响应 SHALL NOT 包含薪资、工作档案、上班时间、隐藏模式、生存账单、消费统计、CPS 平台来源、微信 openid、微信 unionid、微信 sessionKey 或用户登录态
- **AND** 作者信息 SHALL 使用公开作者快照和后台治理所需的最小用户 ID

#### Scenario: 社区治理响应不暴露内部审核细节

- **WHEN** 后台读取包含内容审核信息的社区治理响应
- **THEN** 响应 SHALL 可以包含归一化风险标签、命中字段、命中词摘要、AI 简短原因、低成本审核建议、置信度和人工复核原因
- **AND** 响应 SHALL NOT 包含 AI 原始 prompt、供应商完整原始响应、供应商密钥或代理配置

#### Scenario: 社区治理响应不暴露明文 IP

- **WHEN** 后台读取社区治理响应
- **THEN** 响应 SHALL NOT 包含明文 IP
- **AND** 如展示来源风险，只能展示脱敏值、哈希摘要或风险标签

### Requirement: 后台运营验证扩展

后台运营验证 SHALL 覆盖社区内容治理入口、令牌校验和敏感字段边界。

#### Scenario: 运行后台运营验证

- **WHEN** 开发者运行 `pnpm verify:admin-operations` 或后台社区治理验证命令
- **THEN** 验证 SHALL 检查后台运营中心存在社区内容治理入口
- **AND** 验证 SHALL 检查社区内容治理请求携带 `x-admin-token`
- **AND** 验证 SHALL 检查社区治理响应不包含敏感字段黑名单

### Requirement: 日报后台板块独立工作区

后台日报运营界面 SHALL 将 `今日参悟` 和 `离谱卷宗` 作为独立内容记录管理，而不是作为同一期日报下的子表单管理。大陆新闻 SHALL 从日报后台运营中移除，并通过独立大陆新闻管理页面维护。

#### Scenario: 查看日报运营界面

- **WHEN** 后台用户进入日报运营模块
- **THEN** 后台 SHALL 在列表中展示日报板块记录及其所属板块、业务日期和状态
- **AND** 后台 SHALL 提供新建按钮并要求先选择 `今日参悟` 或 `离谱卷宗`
- **AND** 后台 SHALL NOT 将 `大陆新闻` 作为日报板块新建选项展示

#### Scenario: 工作区操作不混用

- **WHEN** 后台用户选择某条 `离谱卷宗` 记录
- **THEN** 后台 SHALL 只展示该记录的 `离谱卷宗` 编辑器
- **AND** 后台 SHALL NOT 同时展示或提交 `今日参悟`、`大陆新闻` 内容

### Requirement: 日报后台板块独立保存体验

后台日报运营界面 SHALL 为每个日报板块提供独立保存操作，并 SHALL 明确反馈当前保存的是哪个板块。

#### Scenario: 保存单个板块

- **WHEN** 后台用户点击 `离谱卷宗` 记录的保存按钮
- **THEN** 后台 SHALL 调用板块级保存能力只保存 `absurd_casefile`
- **AND** 后台 SHALL 展示 `离谱卷宗` 已保存或等价反馈

#### Scenario: 保存后刷新当前板块

- **WHEN** 某个板块保存成功
- **THEN** 后台 SHALL 使用 API 返回结果刷新该板块编辑状态和发布检查
- **AND** 后台 SHALL NOT 重置、覆盖或删除其他板块记录

### Requirement: 日报后台板块独立发布体验

后台日报运营界面 SHALL 为每个日报可编辑板块提供独立发布入口，并 SHALL 将板块发布与大陆新闻独立发布区分展示。

#### Scenario: 发布当前板块

- **WHEN** 后台用户点击 `今日参悟` 工作区的发布按钮
- **THEN** 后台 SHALL 只提交 `daily_reflection` 的保存与发布流程
- **AND** 后台 SHALL 不触发大陆新闻发布或其他板块发布

#### Scenario: 其他板块存在问题

- **WHEN** 后台用户发布 `离谱卷宗` 且 `离谱卷宗` 通过板块级发布检查
- **AND** `今日参悟` 仍存在空内容或草稿未完成问题
- **THEN** 后台 SHALL 允许继续发布 `离谱卷宗`
- **AND** 后台 SHALL 保持 `今日参悟` 记录原状态不变

#### Scenario: 不展示整期发文入口

- **WHEN** 后台用户编辑某个日报板块记录
- **THEN** 后台 SHALL 只展示该板块记录的保存、预览、发布、定时发布、取消定时和归档操作
- **AND** 后台 SHALL NOT 将日报板块表达为同一期日报的整期发布操作

### Requirement: 后台大陆新闻独立入口

后台运营中心 SHALL 提供大陆新闻独立管理入口，并 SHALL 将其与隐者日报运营入口区分。

#### Scenario: 工作台展示大陆新闻入口

- **WHEN** 后台用户打开运营中心工作台或主导航
- **THEN** 后台 SHALL 展示进入大陆新闻管理页面的入口
- **AND** 该入口 SHALL NOT 被表达为隐者日报某个板块的子入口

#### Scenario: 从日报运营进入大陆新闻

- **WHEN** 后台用户在日报运营页面寻找大陆新闻能力
- **THEN** 后台 MAY 展示跳转到大陆新闻管理页的提示或入口
- **AND** 后台 SHALL NOT 在日报运营页内直接编辑大陆新闻文章列表

### Requirement: 后台大陆新闻管理页面

后台 SHALL 提供大陆新闻管理页面，支持列表检索、批量新增、编辑已发布内容、发布和下线。

#### Scenario: 查看大陆新闻管理页面

- **WHEN** 后台用户进入大陆新闻管理页面
- **THEN** 后台 SHALL 展示大陆新闻文章列表、状态、发布时间、更新时间和可用操作
- **AND** 后台 SHALL 支持按状态或关键词筛选文章

#### Scenario: 批量新增文章

- **WHEN** 后台用户在大陆新闻管理页面一次新增多篇文章
- **THEN** 后台 SHALL 调用大陆新闻独立管理接口提交这些文章
- **AND** 后台 SHALL NOT 因文章数量超过 10 条而阻止提交

#### Scenario: 编辑已发布文章

- **WHEN** 后台用户打开已发布大陆新闻文章并保存修改
- **THEN** 后台 SHALL 调用大陆新闻独立管理接口保存修改
- **AND** 后台 SHALL 展示修改成功和最新更新时间

#### Scenario: 下线文章

- **WHEN** 后台用户下线某篇大陆新闻文章
- **THEN** 后台 SHALL 调用大陆新闻独立管理接口更新文章状态
- **AND** 后台 SHALL 在列表中展示该文章已下线或隐藏

### Requirement: 后台大陆新闻令牌和敏感字段边界

后台大陆新闻管理接口 SHALL 继续使用 `x-admin-token` 校验，并 SHALL 过滤不属于大陆新闻运营所需的敏感字段。

#### Scenario: 无令牌访问大陆新闻管理

- **WHEN** 请求未携带有效 `x-admin-token` 访问大陆新闻后台列表、详情、创建、编辑、发布或下线接口
- **THEN** API SHALL 拒绝请求
- **AND** API SHALL NOT 返回草稿文章、后台备注或未公开内容

#### Scenario: 后台响应不暴露无关敏感数据

- **WHEN** 后台读取大陆新闻管理列表或详情
- **THEN** 响应 SHALL NOT 包含薪资、工作档案、上班时间、隐藏模式、生存账单、消费统计、CPS 平台来源、微信 openid、微信 unionid 或用户登录态
- **AND** 响应 SHALL NOT 包含 AI 原始 prompt、供应商完整原始响应、供应商密钥或代理配置

### Requirement: 后台补给铺运营配置

后台运营中心 SHALL 提供补给铺配置能力，用于维护补给项、所属板块、展示文案、封面、排序、状态、有效期和内部活动来源。

#### Scenario: 查看补给项列表

- **WHEN** 携带有效 `x-admin-token` 的后台用户进入补给铺运营配置
- **THEN** 后台 SHALL 展示补给项列表
- **AND** 列表 SHALL 支持查看补给项标题、所属板块、状态、排序、有效期和最近更新时间

#### Scenario: 新增或编辑补给项

- **WHEN** 后台用户新增或编辑补给项
- **THEN** 后台 SHALL 允许配置用户可见标题、描述、所属板块、封面、行动文案、排序、上下架状态和有效期
- **AND** 后台 SHALL 允许配置聚推客活动标识、活动来源、默认生存分类和内部备注

#### Scenario: 校验补给项板块

- **WHEN** 后台保存补给项配置
- **THEN** 系统 SHALL 校验所属板块只能是 `隐者食堂`、`下午续命` 或 `通勤补给`
- **AND** 系统 SHALL 拒绝保存不属于第一期补给铺范围的板块

### Requirement: 后台补给铺同步排查

后台运营中心 SHALL 提供补给点击和聚推客订单同步排查能力，用于定位转链失败、订单未匹配和订单入账状态。

#### Scenario: 查看补给点击记录

- **WHEN** 授权后台用户查看补给点击记录
- **THEN** 后台 SHALL 展示点击时间、补给项、补给板块、活动标识、跳转状态和脱敏 `sid` 摘要
- **AND** 后台 SHALL NOT 展示微信 openid、unionid、手机号或用户登录态

#### Scenario: 查看订单同步记录

- **WHEN** 授权后台用户查看聚推客订单同步记录
- **THEN** 后台 SHALL 展示订单号、活动标识、品牌标识、订单状态、付款金额、付款时间、匹配状态、入账状态和失败原因
- **AND** 后台 SHALL NOT 展示聚推客 `apikey` 或完整原始响应作为默认列表字段

#### Scenario: 排查未匹配订单

- **WHEN** 聚推客订单无法通过 `sid` 匹配用户点击
- **THEN** 后台 SHALL 能看到该订单的未匹配原因
- **AND** 系统 SHALL NOT 自动把该订单归入任意用户生存账本

### Requirement: 后台补给铺令牌和敏感字段边界

后台补给铺运营接口 SHALL 继续使用 `x-admin-token` 校验，并过滤不属于运营所需的用户私密数据和密钥字段。

#### Scenario: 无令牌访问补给铺运营

- **WHEN** 请求未携带有效 `x-admin-token` 访问补给铺配置、点击记录或订单同步排查接口
- **THEN** API SHALL 拒绝请求
- **AND** API SHALL NOT 返回补给项内部来源、点击归因、订单同步记录或后台备注

#### Scenario: 后台响应不暴露密钥

- **WHEN** 后台读取补给铺配置或同步排查数据
- **THEN** 响应 SHALL NOT 包含聚推客 `apikey`、服务端环境变量、供应商密钥或代理配置

#### Scenario: 后台响应最小化用户私密数据

- **WHEN** 后台读取补给点击或订单同步排查数据
- **THEN** 响应 SHALL 使用内部用户 ID、脱敏值或状态标签支撑排查
- **AND** 响应 SHALL NOT 包含薪资、工作档案、上班时间、隐藏模式、微信 openid、微信 unionid、手机号或用户登录态

### Requirement: 后台补给铺验证

项目 SHALL 提供后台补给铺运营验证，覆盖配置校验、令牌校验和敏感字段边界。

#### Scenario: 运行后台补给铺验证

- **WHEN** 开发者运行后台运营或补给铺相关验证命令
- **THEN** 验证 SHALL 检查补给项配置只能使用第一期三类板块
- **AND** 验证 SHALL 检查无效 `x-admin-token` 无法读取补给铺运营数据
- **AND** 验证 SHALL 检查后台响应不包含聚推客密钥、微信身份字段或用户登录态

### Requirement: 后台实时待办事件流

后台运营中心 SHALL 提供实时待办事件流，用于提示新审核、新举报、日报评论复核和工作台计数变化，并 SHALL 继续使用后台令牌校验。

#### Scenario: 连接实时待办事件流

- **WHEN** 后台前端携带有效 `x-admin-token` 连接实时待办事件流
- **THEN** API SHALL 建立只面向后台运营中心的事件流连接
- **AND** 事件流 SHALL 只返回后台待办提示所需的最小事件字段

#### Scenario: 无效令牌连接事件流

- **WHEN** 请求未携带有效 `x-admin-token` 连接实时待办事件流
- **THEN** API SHALL 拒绝连接
- **AND** API SHALL NOT 返回待审内容、举报内容、日报草稿、补给铺排查数据或任何后台待办事件

#### Scenario: 事件流不暴露敏感字段

- **WHEN** 后台实时事件包含社区、日报、举报或补给铺相关变化
- **THEN** 事件 payload SHALL NOT 包含薪资、工作档案、上班时间、隐藏模式、生存账单、微信 openid、微信 unionid、用户登录态、AI 原始 prompt、供应商完整原始响应、聚推客 `apikey`、完整订单同步原始响应或服务端环境变量
- **AND** 事件 payload SHALL NOT 作为审核详情或补给铺排查详情的替代响应

### Requirement: 实时待办可控刷新

后台前端 SHALL 将实时事件和轮询结果表达为可控的新待办提示，而不是自动覆盖当前队列、详情或编辑表单。

#### Scenario: 收到新待办事件

- **WHEN** 后台前端收到新的待办事件
- **THEN** 后台 SHALL 在顶部展示新待办数量或等价提示
- **AND** 后台 SHALL NOT 在用户点击刷新前强制替换当前审核队列、详情面板或编辑表单

#### Scenario: 点击新待办提示刷新

- **WHEN** 后台用户点击新待办提示
- **THEN** 后台 SHALL 重新读取当前工作台、审核队列或当前模块事实源
- **AND** 后台 SHALL 清除已经被本次刷新消费的新待办提示

#### Scenario: 编辑中不自动覆盖

- **WHEN** 后台用户正在填写审核备注、治理原因、日报正文、补给项配置或处理详情
- **AND** 后台收到实时事件或轮询发现新待办
- **THEN** 后台 SHALL 保留用户当前输入
- **AND** 后台 SHALL 只展示新待办提示或连接状态变化

### Requirement: 实时连接降级轮询

后台前端 SHALL 在实时连接不可用时自动降级为轮询，并 SHALL 在实时连接恢复后停止降级轮询。

#### Scenario: 实时连接断开

- **WHEN** 后台实时事件流断开、超时或连接失败
- **THEN** 后台 SHALL 展示已降级轮询或等价连接状态
- **AND** 后台 SHALL 以低频轮询读取待办摘要或队列变化

#### Scenario: 轮询发现变化

- **WHEN** 降级轮询发现待办计数或队列摘要发生变化
- **THEN** 后台 SHALL 展示新待办提示
- **AND** 后台 SHALL NOT 自动覆盖正在编辑的内容

#### Scenario: 实时连接恢复

- **WHEN** 后台实时事件流恢复可用
- **THEN** 后台 SHALL 停止降级轮询
- **AND** 后台 SHALL 展示实时连接已恢复或等价状态

### Requirement: 审核队列局部处理反馈

统一审核队列 SHALL 为每个可操作队列项展示处理状态，并 SHALL 在操作后局部更新、移出当前队列或展示状态已变化反馈。

#### Scenario: 审核操作处理中

- **WHEN** 后台用户对队列项执行通过、驳回、隐藏、处理举报或发布操作
- **THEN** 当前队列项 SHALL 展示处理中状态
- **AND** 后台 SHALL 防止同一队列项在处理中被重复提交

#### Scenario: 审核操作成功

- **WHEN** 审核操作成功
- **THEN** 后台 SHALL 展示操作成功反馈
- **AND** 当前队列项 SHALL 按当前筛选条件局部移出队列或更新为最新状态和可用操作

#### Scenario: 审核操作失败

- **WHEN** 审核操作失败
- **THEN** 后台 SHALL 展示后台用户可理解的失败原因
- **AND** 失败原因 SHALL 区分令牌失效、网络失败、状态不允许、目标不存在和服务端校验失败

#### Scenario: 队列项已被处理

- **WHEN** 后台用户操作的队列项已经被其他流程处理或状态已变化
- **THEN** 后台 SHALL 展示状态已变化或重复处理提示
- **AND** 当前队列项 SHALL 标记为需要刷新或从当前待办队列移出

### Requirement: 审核队列信息密度和操作层级

统一审核队列 SHALL 更清晰地区分内容审核、举报审核和全部待办，并 SHALL 按操作风险和使用频率组织按钮层级。

#### Scenario: 查看统一审核队列

- **WHEN** 后台用户查看统一审核队列
- **THEN** 每个队列项 SHALL 清晰展示来源、对象类型、状态、风险标签、审核来源、用户风险原因、内容摘要、作者公开快照和创建时间
- **AND** 帖子、评论、回复、举报和日报评论 SHALL 使用可扫读的类型标签或等价视觉区分

#### Scenario: 切换内容与举报队列

- **WHEN** 后台用户在内容审核、举报审核和全部待办之间切换
- **THEN** 后台 SHALL 保留各分组适用的筛选和刷新上下文
- **AND** 内容审核分组 SHALL 不展示举报原因筛选作为主筛选
- **AND** 举报审核分组 SHALL 优先展示举报状态、举报原因、目标类型和优先级筛选

#### Scenario: 展示内容审核操作

- **WHEN** 队列项是帖子、评论、回复或日报评论
- **THEN** 后台 SHALL 将通过、驳回、隐藏和查看详情按当前可用操作展示
- **AND** 后台 SHALL NOT 展示当前对象状态不允许执行的操作

#### Scenario: 展示举报处理操作

- **WHEN** 队列项是社区举报
- **THEN** 后台 SHALL 展示保留内容、隐藏内容、标记误报和查看上下文等举报处理操作
- **AND** 后台 SHALL NOT 将举报处理误表达为普通内容的通过或驳回

### Requirement: 后台运营统一状态体验

后台运营中心 SHALL 统一工作台、审核队列、日报运营、大陆新闻、社区治理和补给铺排查的加载态、错误态、空状态、无结果状态、刷新入口和操作反馈。

#### Scenario: 模块加载中

- **WHEN** 后台模块正在读取列表、详情或表单数据
- **THEN** 后台 SHALL 展示与当前模块范围一致的加载状态
- **AND** 后台 SHALL NOT 使用空列表误表达仍在加载的数据

#### Scenario: 模块读取失败

- **WHEN** 后台模块读取数据失败
- **THEN** 后台 SHALL 展示可理解的错误状态和重试入口
- **AND** 令牌失效 SHALL 被明确表达为需要检查后台令牌

#### Scenario: 空状态和无结果状态

- **WHEN** 后台模块没有任何待办或配置数据
- **THEN** 后台 SHALL 展示真实空状态
- **AND** 当筛选条件导致无结果时，后台 SHALL 展示无结果状态并保留当前筛选条件

#### Scenario: 筛选和返回保留上下文

- **WHEN** 后台用户在列表、详情和模块入口之间切换
- **THEN** 后台 SHALL 尽量保留当前模块筛选条件、选中项和刷新上下文
- **AND** 返回列表时 SHALL NOT 无故清空用户刚设置的筛选条件

### Requirement: 后台实时待办验证

项目 SHALL 扩展后台运营验证，覆盖实时事件、降级轮询、可控刷新、局部处理反馈和敏感字段边界。

#### Scenario: 运行后台运营验证

- **WHEN** 开发者运行 `pnpm verify:admin-operations`
- **THEN** 验证 SHALL 检查后台实时事件接口需要 `x-admin-token`
- **AND** 验证 SHALL 检查实时事件 payload 不包含敏感字段黑名单
- **AND** 验证 SHALL 检查审核队列操作反馈、状态已变化提示和新待办可控刷新入口

### Requirement: 后台补给活动运营深化

后台运营中心 SHALL 提供更完整的补给活动运营能力，用于批量维护活动分组、标签、优先级、有效期、上下线和排序。

#### Scenario: 批量启停补给活动

- **WHEN** 授权后台用户选择多个补给活动执行批量上架或下线
- **THEN** 系统 SHALL 更新所选活动状态
- **AND** 后台 SHALL 展示批量操作成功、部分失败或全部失败的反馈
- **AND** 普通用户公开响应 SHALL 只展示仍处于可见状态的补给活动

#### Scenario: 复制补给活动

- **WHEN** 授权后台用户复制已有补给活动
- **THEN** 系统 SHALL 生成新的草稿活动
- **AND** 新活动 SHALL 继承标题、描述、板块、标签、默认分类、转链配置、有效期和内部备注
- **AND** 新活动 SHALL 使用新的稳定活动 ID，避免覆盖原活动统计和归因记录

#### Scenario: 配置活动标签和优先级

- **WHEN** 授权后台用户新增或编辑补给活动
- **THEN** 后台 SHALL 允许配置活动分组、用户可见标签、展示优先级、推荐时段、工作日规则和排序值
- **AND** 后台 SHALL 校验普通用户可见标签不得包含 CPS、佣金、返利、聚推客字段、结算状态或内部活动 ID

#### Scenario: 预览普通用户可见效果

- **WHEN** 授权后台用户预览补给活动配置
- **THEN** 后台 SHALL 展示按当前配置生成的普通用户可见标题、描述、标签、排序、可点击状态和同步提示
- **AND** 预览 SHALL NOT 展示聚推客 `apikey`、完整 `sid`、佣金、订单源 ID 或服务端环境变量

### Requirement: 后台 CPS 排查中心

后台运营中心 SHALL 提供 CPS 排查中心，将补给活动、点击记录、转链记录、订单同步记录和生存账本入账记录串成可追踪链路。

#### Scenario: 查看订单链路详情

- **WHEN** 授权后台用户查看一条 CPS 订单排查详情
- **THEN** 后台 SHALL 展示来源活动、补给板块、脱敏 `sid`、点击时间、转链状态、订单状态、匹配状态、入账状态、账单 ID 和失败原因
- **AND** 后台 SHALL NOT 展示聚推客 `apikey`、完整 `sid`、完整原始订单响应、微信 openid、微信 unionid、手机号或用户登录态

#### Scenario: 查看点击链路详情

- **WHEN** 授权后台用户查看一条补给点击排查详情
- **THEN** 后台 SHALL 展示补给项、活动标识、点击时间、转链尝试、跳转目标类型、转链失败原因、匹配订单数量和入账结果摘要
- **AND** 后台 SHALL NOT 将点击详情作为普通用户消费明细公开

#### Scenario: 查询为什么没入账

- **WHEN** 授权后台用户查看未入账订单
- **THEN** 后台 SHALL 展示明确失败原因码和用户可读解释
- **AND** 失败原因 SHALL 至少区分 `sid` 缺失、`sid` 未匹配、归因过期、活动不匹配、订单状态无效、订单未支付、重复订单已更新、分类异常和账本入账失败

#### Scenario: 按排查条件筛选

- **WHEN** 授权后台用户使用排查中心筛选数据
- **THEN** 后台 SHALL 支持按时间范围、补给活动、补给分类、转链状态、订单状态、入账状态和失败原因筛选
- **AND** 筛选结果 SHALL 保持分页或等价限制，避免一次返回无限制历史数据

### Requirement: 后台 CPS 异常池

后台运营中心 SHALL 提供 CPS 异常池，用于集中查看无归因、过期归因、活动不匹配、状态未知、分类异常和入账失败的订单。

#### Scenario: 查看异常订单

- **WHEN** 授权后台用户进入 CPS 异常池
- **THEN** 后台 SHALL 展示异常订单的时间、来源活动、脱敏 `sid`、订单状态、金额、异常类型、失败原因和最近更新时间
- **AND** 后台 SHALL NOT 默认展示完整原始响应或密钥字段

#### Scenario: 重试可恢复异常

- **WHEN** 异常订单属于转链记录延迟、账本入账失败或状态映射补充后可恢复的类型
- **THEN** 后台 SHALL 提供受控重试入口或标记待重试状态
- **AND** 重试 SHALL 继续遵守来源订单幂等规则

#### Scenario: 不允许强行归属无归因订单

- **WHEN** 异常订单缺少有效归因或归因无法匹配用户
- **THEN** 后台 SHALL NOT 提供把该订单手动分配给任意用户的操作
- **AND** 系统 SHALL 保持该订单不进入任何用户生存账本

### Requirement: 后台 CPS 经营指标

后台运营中心 SHALL 提供 CPS 经营指标，用于评估补给展示、点击、转链、订单回流、有效订单和入账结果。

#### Scenario: 查看今日指标

- **WHEN** 授权后台用户查看 CPS 指标面板
- **THEN** 后台 SHALL 展示今日点击数、独立点击用户数、转链成功率、订单回流数、有效订单数、入账金额和异常订单数
- **AND** 指标 SHALL 支持按 `隐者食堂`、`下午续命` 和 `通勤补给` 分类查看

#### Scenario: 查看活动指标

- **WHEN** 授权后台用户按补给活动筛选指标
- **THEN** 后台 SHALL 展示该活动的点击数、转链成功数、订单回流数、有效订单数、入账金额和异常原因分布
- **AND** 指标 SHALL 基于点击、转链、订单同步和账本入账事实记录聚合

#### Scenario: 指标不展示佣金和返利承诺

- **WHEN** 后台展示 CPS 经营指标
- **THEN** 后台 SHALL NOT 将佣金、返利、用户可获得收益或结算承诺作为普通用户可见字段
- **AND** 指标响应 SHALL NOT 被普通用户接口复用或公开

### Requirement: 后台 CPS 敏感字段边界

后台 CPS 运营、排查、异常池和指标接口 SHALL 使用字段白名单，过滤密钥、完整身份、完整归因 token 和不必要的原始响应。

#### Scenario: 后台响应不暴露完整 sid

- **WHEN** 后台读取补给点击、转链记录、订单同步、异常池或排查详情
- **THEN** 响应 SHALL 只返回脱敏 `sid` 或不可逆摘要
- **AND** 响应 SHALL NOT 返回完整原始 `sid`

#### Scenario: 后台响应不暴露供应商密钥

- **WHEN** 后台读取补给活动、转链配置、订单同步或指标数据
- **THEN** 响应 SHALL NOT 包含聚推客 `apikey`、服务端环境变量、代理配置、供应商密钥或完整签名材料

#### Scenario: 后台响应最小化用户数据

- **WHEN** 后台读取匹配用户相关的 CPS 排查数据
- **THEN** 响应 SHALL 仅返回排查所需的内部用户标识、公开昵称快照或脱敏值
- **AND** 响应 SHALL NOT 包含微信 openid、微信 unionid、手机号、用户登录态、薪资、工作档案、隐藏模式或完整消费明细导出

### Requirement: 后台 CPS 验证

项目 SHALL 提供后台 CPS 验证，覆盖活动运营、排查中心、异常池、经营指标和敏感字段边界。

#### Scenario: 验证活动运营

- **WHEN** 开发者运行后台补给铺或后台运营验证命令
- **THEN** 验证 SHALL 检查批量启停、复制活动、标签校验、排序和普通用户预览字段
- **AND** 验证 SHALL 检查普通用户可见字段不包含 CPS 内部信息

#### Scenario: 验证排查中心

- **WHEN** 验证数据包含转链失败、无归因订单、活动不匹配订单、退款回滚和入账失败记录
- **THEN** 验证 SHALL 检查后台排查中心返回链路详情和失败原因
- **AND** 验证 SHALL 检查无归因订单不能被强行归属到用户

#### Scenario: 验证指标与敏感字段

- **WHEN** 验证后台 CPS 指标响应
- **THEN** 验证 SHALL 检查点击数、转链成功率、订单回流数、有效订单数、入账金额和异常订单数
- **AND** 验证 SHALL 检查响应不包含聚推客密钥、完整 `sid`、微信身份、用户登录态、完整原始同步响应或佣金返利承诺

### Requirement: 后台内容安全摘要展示

后台运营中心 SHALL 在统一审核队列、审核详情和工作台摘要中展示归一化内容安全结果，帮助管理员区分本地规则、微信内容安全、mock、供应商错误和人工复核来源。

#### Scenario: 队列展示内容安全摘要

- **WHEN** 后台用户读取统一审核队列
- **THEN** 每个相关队列项 SHALL 展示内容安全来源、归一化风险标签、简短原因、审核决策、用户风险原因和人工复核原因
- **AND** 队列项 SHALL 能区分本地规则、微信文本审核、微信图片审核、mock 和供应商降级

#### Scenario: 工作台统计内容安全分流

- **WHEN** 后台用户读取运营中心工作台
- **THEN** API SHALL 返回自动通过、自动驳回、微信待回调、供应商无法确认和人工复核的统计摘要
- **AND** 摘要 SHALL 支持社区、日报评论和大陆新闻评论的来源区分

#### Scenario: 审核详情展示图片审核状态

- **WHEN** 后台用户查看带图社区帖子的审核详情
- **THEN** API SHALL 返回每张图片的归一化审核状态、供应商来源、脱敏任务标识和可理解原因
- **AND** 后台 SHALL 可以据此人工通过、驳回或隐藏帖子

### Requirement: 后台内容安全敏感字段边界

后台运营中心 SHALL 过滤微信内容安全和供应商审核的敏感字段，只返回后台处理所需的归一化摘要。

#### Scenario: 队列响应不暴露微信敏感字段

- **WHEN** 后台读取工作台、审核队列或审核详情
- **THEN** 响应 SHALL NOT 包含微信 `secret`、`access_token`、`sessionKey`、完整 openid、完整 unionid、完整供应商原始响应或服务端环境变量
- **AND** 如需展示任务标识，响应 SHALL 使用脱敏 `trace_id` 或不可逆摘要

#### Scenario: 后台筛选风险不依赖原始响应

- **WHEN** 后台用户按内容安全来源、风险标签或人工复核原因筛选队列
- **THEN** API SHALL 基于归一化字段完成筛选
- **AND** API SHALL NOT 需要返回供应商完整原始响应来支撑筛选

### Requirement: 运营队列身份与 IP 合规边界

后台统一审核队列、实时待办和运营摘要 SHALL 可以展示社区内容的脱敏身份与 IP 合规摘要，但 SHALL NOT 暴露完整手机号、明文 IP 或微信身份字段。

#### Scenario: 审核队列展示 IP 属地摘要

- **WHEN** 后台用户查看社区帖子、评论或回复审核队列
- **THEN** 队列项 MAY 展示发布时 IP 属地标签、手机号验证布尔状态和隐私同意版本摘要
- **AND** 队列项 SHALL NOT 展示完整手机号、手机号尾号、明文 IP、微信 openid、微信 unionid、微信 sessionKey 或用户登录态

#### Scenario: 实时待办不暴露敏感字段

- **WHEN** 社区内容审核、举报或治理相关事件进入实时待办提示
- **THEN** 事件数据 MAY 包含降精度 IP 属地标签和必要合规状态
- **AND** 事件数据 SHALL NOT 包含完整手机号、明文 IP、微信身份、用户登录态或 IP 来源 header

#### Scenario: 后台敏感字段黑名单扩展

- **WHEN** 开发者运行后台运营验证命令
- **THEN** 验证 SHALL 将完整手机号、手机号尾号、明文 IP、IP 来源 header、微信 openid、微信 unionid、微信 sessionKey 和用户登录态纳入敏感字段黑名单
- **AND** 验证 SHALL 检查统一审核队列、实时待办和运营摘要均不返回这些字段
