## Context

小程序用户端当前以暗黑像素忍者 RPG 视觉作为默认外观，且大量页面在自身 `<style>` 中重新声明 `--camp-*` 暗色变量、深色渐变、白字、暗色边框和暗色阴影。全局 `visual-system.css` 已经提供了 `vs-page`、`vs-panel`、`camp-*` 等通用视觉语言，但主链路页面和历史页面仍存在较多局部硬编码。

本 change 的产品方向已经从“给主链路补一套浅色主题”调整为“工位模式成为小程序用户端全局默认外观”。因此实现不能只处理首页、社区、我的等主路径，也不能只替换页面背景；所有已注册用户端页面里的可见组件、状态和弹层都要适配默认工位模式，并保留隐者模式作为可切换暗黑 RPG 体验。

## Goals / Non-Goals

**Goals:**

- 将工位模式作为小程序用户端默认视觉模式。
- 保留隐者模式，并通过首页右上角入口切换。
- 使用本地存储保存 `workplace | hermit` 选择。
- 建立模式化 token，让背景、卡片、文字、边框、按钮、状态色、弹层和导航栏/tabBar 可统一切换。
- 覆盖 `apps/miniapp/src/pages.json` 中所有用户端页面和公共视觉组件。
- 对页面内每一个可见 UI 单元做组件级细节适配，包括普通态、选中态、禁用态、加载态、错误态、骨架屏和弹层态。
- 保持 `stealth-workbench/*` 办公表格视觉例外，不把它强行改成暗黑 RPG。
- 不改变业务逻辑、接口、路由、审核、安全、订单、账本和工作价值计算。

**Non-Goals:**

- 不覆盖 `apps/admin` 后台。
- 不新增服务端主题同步、用户偏好表或远端配置。
- 不跟随系统主题。
- 不重做信息架构、底部主 tab 或页面层级。
- 不重新设计一整套白版插画、头像或 icon 资产。
- 不修改 webview 内部第三方页面视觉。

## Decisions

### 1. 使用 `visualMode` 而不是 `theme`

使用 `workplace` 和 `hermit` 表达产品语义，避免“白昼/暗夜”这类自然时间词。用户使用场景主要发生在白天工位，因此浅色默认不是“白天主题”，而是更低暴露的工位外观。

备选方案是 `light/dark` 或 `day/night`。这些命名虽然技术直观，但会把产品语义带向系统主题或自然昼夜，不适合当前世界观和办公场景。

### 2. 默认值为 `workplace`

没有本地存储时默认进入工位模式。这样用户即使从未发现首页右上角切换按钮，也会处在更适合办公环境的界面里。

备选方案是继续默认隐者模式，只把工位模式作为伪装开关。这个方案会把“发现按钮”变成关键路径，和用户担心按钮不易发现的问题冲突。

### 3. 首页右上角只放主入口

首页右上角入口显示目标动作：工位模式下显示“隐者模式”，隐者模式下显示“返回工位”。不在“我的”页新增设置项，因为“我的”页已经承载角色、工作、社区和账本入口，继续加设置会造成信息拥挤。

入口必须呈现为可点击按钮或胶囊按钮，不能复用原有身份徽章样式导致用户误以为只是标签。

### 4. Token 先行，页面逐页收敛

先在全局样式和 `packages/ui-tokens` 中建立工位/隐者两套语义 token，再逐页清理页面内的暗色硬编码。页面应尽量引用 `--camp-*`、`--vs-*` 或新增模式变量，而不是继续散落 `#050508`、`#13131b`、白字和暗色渐变。

备选方案是逐页直接写浅色覆盖样式。这个方案短期快，但后续新增页面和组件会继续分叉，也难以保证所有状态一致。

### 5. 原生 tabBar 保持，运行时切色

继续使用微信原生 tabBar，不引入自定义 tabBar。`pages.json` 默认配置应偏向工位模式，运行时根据本地 visualMode 调用导航栏和 tabBar 样式更新。

这样既保持既有三主 tab 结构，也避免自定义 tabBar 带来的状态同步和小程序兼容风险。

### 6. `stealth-workbench/*` 仍是视觉例外

摸鱼模式工作台已经被 OpenSpec 定义为办公表格视觉例外。它和本 change 的工位模式不是同一个东西：工位模式是全局默认浅色皮肤，摸鱼模式是特定页面组的办公表格信息架构。

因此 `stealth-workbench/*` 不跟随隐者模式改成暗黑 RPG，但需要检查其导航栏、页面背景、表格组件和状态提示与新的默认工位模式不冲突。

### 7. webview 只适配小程序外壳

`pages/webview/index` 只能控制小程序自身导航栏、加载态、错误态、外壳背景和提示文案。外部 webview 内容来自第三方页面，不能承诺全局主题化。

## Risks / Trade-offs

- 页面内硬编码暗色较多 → 先建立 token 清单和页面组件清单，再逐页替换；验收时用搜索检查暗色硬编码是否仍裸露在工位模式路径。
- scoped 样式和页面级 `.vs-page` 变量可能覆盖全局 token → 每个页面都要检查根节点变量声明，避免页面重新把工位模式染回暗色。
- 首页右上角入口可发现性有限 → 默认工位模式降低发现成本；入口文案使用“隐者模式/返回工位”这类动作表达，并增加按钮边框、按压态和切换 toast。
- 隐者模式可能被浅色改造削弱 → 工位模式和隐者模式都使用模式变量，暗色旧质感不能被直接删除，只能收敛为隐者模式。
- 全量页面组件适配工作量大 → 按页面组拆任务，先公共 token/组件，再核心 tab，再内容页、社区子页、兼容页和工作台例外。
- 图片资产在浅色背景下可能不清晰 → 不重画资产，优先用外框、底座、遮罩、文字安全区和占位状态解决可读性。
- webview 内部页面无法控制 → 规格明确只控制小程序外壳，不把第三方内容纳入验收。

## Migration Plan

1. 新增本地视觉模式读取、保存和应用逻辑，默认值为 `workplace`。
2. 将 `pages.json` 的默认导航栏和 tabBar 色彩调整为工位模式。
3. 在全局视觉系统中新增或改造工位/隐者模式 token。
4. 更新公共组件和共享视觉类。
5. 逐页清理页面内暗色硬编码，并补齐组件状态。
6. 更新视觉设计文档和规格事实源。
7. 运行类型检查、小程序构建、包体检查和人工页面清单验收。

如需回退，保留 `hermit` token 和原有暗色资产，可将默认值临时改回 `hermit` 并隐藏首页切换入口；但规格归档后默认事实源应仍以工位模式为准。

## Open Questions

- 首页右上角按钮是否需要配套一个小图标，还是只使用文字和边框即可。
- 原生 tabBar 的现有像素 icon 在浅色背景下是否足够清晰；若不清晰，是否只做描边/明度调整版小图。
