## Context

小程序用户端已经具备暗黑像素忍者 RPG 的基础 token、页面结构和插画预留清单。当前已入库的正式图片主要集中在个人中心阵营头像、徽章、阵营插画、角色页横幅和日报栏目封面；首页、补给铺、社区状态、生存账本和多类空/错/加载状态仍大量依赖 `vs-asset-placeholder`、纯文本提示或无语义圆点。

这次 change 的核心不是重做业务链路，而是把现有视觉方向收束为可执行系统：资产有尺寸和安全区，页面有统一状态模型，动态信息由原生 UI 渲染，缺图时也使用同风格 fallback。

## Goals / Non-Goals

**Goals:**

- 建立插画资产矩阵，覆盖 `banner`、`artwork`、`scene`、`icon`、`avatar`、`badge` 和状态图。
- 为首页、补给铺、社区列表、我的页、生存账本和首跑引导定义统一视觉状态。
- 替换高频页面里的开发态占位和无语义圆点，使入口和状态具备明确语义。
- 保持移动端信息密度克制，确保金额、倒计时、按钮、列表和状态标签可读。
- 同步 OpenSpec 和项目文档，给后续插画生成、替换和验收提供稳定依据。

**Non-Goals:**

- 不改后台运营中心 UI。
- 不改 CPS、补给铺、社区、用户成长资料或生存账本的数据模型。
- 不新增独立漫画 tab、成就商店、皮肤商店或排行榜玩法。
- 不把整页 UI 切成静态图片，也不把动态业务文案画进插画。
- 不要求一次性产出所有最终美术资源；允许先用符合规范的 fallback。

## Decisions

### 1. 以资产注册表和规格矩阵作为插画入口

继续沿用 `apps/miniapp/src/visual-system/illustration-registry.ts` 的稳定 asset key，并补充页面实际使用的尺寸、比例、安全区和 fallback 规则。页面不得直接散落临时文件名或展示开发态占位文本。

备选方案是按页面各自维护图片路径，但这会让后续替换、压缩、验收和风格收束变难。

### 2. 将状态视觉抽象为少量共享状态类型

加载、空数据、未登录、未建档、配置缺失、网络失败、降级展示、补给不可用和订单未同步应映射到统一状态类型，再由页面选择对应插画、标题、说明和操作按钮。

备选方案是每个页面自己写一套状态文案和布局。短期快，但会导致小程序主链路看起来像不同产品。

### 3. 先打磨用户主链路，不先碰后台

优先页面为首页、补给铺、社区列表、我的页和生存账本。这些页面构成用户内测时最高频的首屏体验和核心闭环。后台仍保持已有运营可用性，不在本 change 内做视觉重构。

### 4. 插画只承载氛围和语义，动态信息保持原生 UI

金额、倒计时、等级、经验、订单金额、按钮、tabs、筛选、状态标签、帖子正文和业务说明都必须由原生 UI 渲染。插画可包含符号、道具、卷轴、工位、阵营元素和无关键语义的小装饰，但不得承载会变化的业务文案。

### 5. 缺失正式插画时使用生产级 fallback

正式图缺失时，使用暗色图案、剪影、像素 icon、阵营符号或状态 scene fallback。fallback 不展示 `placeholder`、文件名、尺寸、TODO 或无意义圆点。

### 6. 首页首屏文案和按钮层级收敛

首页工作价值主卡在未登录、未建档、未配置工作档案和 ready 状态之间切换时，应保持相近尺寸和稳定插画区域。未完成首跑或未配置工作档案时，只保留状态、占位金额/倒计时和下一步入口，不展示“登录后才能保存”“填写后首页会显示”等重复解释型长文案。

小程序真实按钮统一采用厚像素文字按钮风格，通过粗暗色外框、内高光、底部压暗阴影和按压下沉反馈表达暗黑像素 RPG 质感。按钮不得为了像素风强行加入无关图标；只有图标具有明确业务语义并来自当前视觉系统时才允许进入按钮。

### 7. 全局字体统一为方正黑体字体栈

小程序用户端展示字体、正文、按钮、金额、倒计时、badge 和短标签统一使用方正黑体字体栈。当前仅声明字体名称和中文系统 fallback，不提交字体文件、不通过 `@font-face` 内嵌字体，也不使用在线 Web Font；如后续取得覆盖微信小程序嵌入或在线调用的授权，再接入字体文件。

## Risks / Trade-offs

- 插画过多导致小程序包体增长 → 优先使用少量可复用状态图和 icon，图片压缩后再入库，非首屏资产按页面引用。
- 视觉装饰压过工具可读性 → 主链路仍以原生 UI 和 RPG 面板承载核心信息，插画只做背景或语义辅助。
- 生成式插画含文字、品牌或平台标识 → 在规范中明确禁止动态 UI 文案、真实 CPS 平台名、二维码和未经授权品牌。
- 字体授权边界误判 → 先使用字体名称和系统 fallback，不把方正字体文件打包进小程序，避免误入嵌入授权范围。
- 页面状态太多导致实现分散 → 先定义共享状态类型和通用状态面板，再按页面接入。
- 当前已有图片风格存在明暗差异 → 保留已有可用资产，但新增资产必须遵守暗黑像素忍者 RPG、克制辉光和安全区规则。

## Migration Plan

1. 补充插画资产矩阵和状态视觉文档。
2. 建立或完善小程序端资产解析和状态展示 helper。
3. 按优先级打磨首页、补给铺、社区列表、我的页、生存账本。
4. 统一首跑、空状态、错误状态和降级状态。
5. 执行类型检查、相关验证脚本和小程序构建验证。

回滚策略：页面仍保留原业务数据和路由结构，若单个视觉资产或状态组件异常，可回退到对应页面的暗色面板 fallback，不影响业务接口。

## Open Questions

- 正式插画是否由人工生成后入库，还是先全部使用同风格 fallback 等待后续美术批次？
- 小程序发布包体预算是否需要设定单张图片和总静态资源上限？
- 角色页横幅和日报栏目封面现有明亮像素图是否保持，还是后续统一调整为更暗的夜幕版本？
