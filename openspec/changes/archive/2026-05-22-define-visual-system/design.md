## Context

`摸鱼侠` 的产品定位是面向年轻打工人的微信小程序，核心体验是把上班时间转化为可见金额、倒计时和 RPG 成长反馈。视觉方向已经明确为暗黑忍者 RPG：用户身份是 `隐者`，世界观是 `隐者大陆`，小程序需要同时具备工具效率、游戏反馈和内容社区属性。

用户提供的 UI 图包含三个关键页面：

- 首页：已摸金额主卡、上班状态场景、设置入口、日报/餐厅入口、倒计时卡、底部辅助工具横幅和四 tab 导航。
- 社区页：漫画更新 Banner、分区导航、筛选 chips、发布按钮、帖子卡列表和互动统计。
- 个人中心页：资料卡、等级进度、资源统计、功能宫格、成就栏、菜单列表和个人场景卡。

该图不是最终生产切图，而是视觉系统基准。实现时应通过 token、原生 UI 组件和像素资产组合复现体验。

## Goals / Non-Goals

**Goals:**

- 固化暗黑忍者 RPG 的视觉身份，形成后续页面和组件实现的事实来源。
- 明确 `packages/ui-tokens` 应承载的 token 类型和语义层级。
- 定义首页、社区页、个人中心页的页面布局模板，避免后续页面各自发明结构。
- 定义第一批小程序共享 UI 组件及其状态。
- 定义像素资产分类、用途和使用边界，避免整屏静态图。
- 定义首页底部辅助工具横幅，将当前视觉实例从签到入口调整为记账入口。
- 约束移动端可读性、触控尺寸、信息密度和状态表达。

**Non-Goals:**

- 不生产最终角色、场景、图标、徽章、漫画等美术资产。
- 不实现完整小程序 UI，不替代后续业务 change 的页面实现任务。
- 不实现首页已摸金额、倒计时、薪资设置、隐藏模式或节假日逻辑。
- 不实现记账业务，包括账单模型、分类、统计、接口、存储或页面流程。
- 不实现社区发帖、审核、成长、签到、活动、小游戏、漫画解锁或管理后台功能。
- 不确定最终字体授权、生产插画风格供应商或完整品牌手册。

## Decisions

### Decision: Treat the mockup as a component system, not static screen art

小程序 SHALL 使用原生 UI 组件承载文本、金额、倒计时、按钮、列表、标签、进度条和交互状态。像素图 SHALL 作为角色、场景、图标、徽章、Banner、漫画和锁定状态资产使用。

Rationale: 当前 UI 图信息密度高且包含大量动态数据。如果整屏切成图片，会导致数据更新、响应式布局、审核态、无障碍和后续迭代困难。

Alternatives considered:

- 整屏静态图片切片：短期最接近视觉稿，但无法承载真实动态业务。
- 纯通用组件库样式：实现快，但会丢失 `摸鱼侠` 的 IP 感和 RPG 反馈。

### Decision: Use semantic visual tokens as the cross-page contract

`packages/ui-tokens` SHALL 不只导出基础颜色，还应提供语义 token，例如页面背景、面板背景、浮起面板、描边、高亮描边、主要文本、次要文本、收益金、能量蓝、稀有紫、成功青、警示橙、危险红、锁定态和禁用态。

Rationale: 后续首页、社区、个人中心和管理配置入口都会复用同一套状态和层级。语义 token 可以降低页面直接引用具体色值造成的漂移。

Alternatives considered:

- 页面内直接写颜色：起步快，但会造成同类状态不一致。
- 只保留品牌色：不足以支持 RPG 面板、资源、等级、阵营和状态。

### Decision: Define primary page layout templates before full business pages

首页、社区页和个人中心页 SHALL 有明确布局模板。后续业务 change 可以替换数据和交互，但 SHOULD 保持核心结构一致，除非新的 OpenSpec change 明确调整。

Rationale: 用户提供的 UI 图已经确定第一版信息架构。先固化布局模板可以减少后续每个业务页面重新讨论视觉结构。

Alternatives considered:

- 等每个业务 change 分别定义页面：灵活，但容易视觉割裂。
- 只定义组件不定义页面：组件可复用，但页面信息层级仍不稳定。

### Decision: Home bottom area is a configurable utility banner

首页底部 tab 上方 SHALL 提供一个横向辅助工具横幅。该横幅可以承载签到、记账、活动或其他高频轻操作入口；当前视觉实例 SHALL 定位为记账入口，而不是每日签到入口。

Rationale: 用户已确认该区域从签到变为记账。视觉系统需要更新区域语义，但记账属于独立业务能力，不应在视觉 change 中定义数据模型。

Alternatives considered:

- 仍固定为签到横幅：与最新 UI 决策冲突。
- 在本 change 中定义完整记账功能：会扩大范围，混淆视觉系统和业务能力。

### Decision: Keep dense RPG panels readable on mobile

视觉系统 SHALL 明确移动端可读性和触控约束，包括最小字号、行高、点击区域、文字截断、卡片内层级数量和小屏安全间距。

Rationale: 暗黑 RPG 面板容易在小屏上过密。可读性约束是视觉系统能否落地的关键。

Alternatives considered:

- 完全跟随视觉稿比例：可能在真实设备上过小或重叠。
- 大幅降低信息密度：会削弱工具面板和 RPG 仪表盘的特色。

## Risks / Trade-offs

- [Risk] 视觉规范过细导致后续业务页面调整困难 -> Mitigation: 规范核心结构和状态语义，允许业务 change 在明确理由下扩展组件变体。
- [Risk] 像素资产缺失导致实现阶段只能用占位图 -> Mitigation: 资产分类先定义占位和命名规则，生产资产可后续替换。
- [Risk] 暗黑背景和高密度信息降低可读性 -> Mitigation: 在 spec 中约束最小字号、对比度、行高、间距和截断规则。
- [Risk] 记账入口被误解为本 change 要实现记账功能 -> Mitigation: proposal、design 和 spec 均明确只定义横幅视觉和入口语义，不定义记账业务。

## Open Questions

- 首页辅助工具横幅的正式命名使用 `摸鱼账本`、`隐者账本` 还是 `今日记账`。
- 是否需要在第一版视觉系统中引入正式像素字体；若引入，需要确认字体授权和微信小程序兼容性。
- 阵营颜色是否只在个人身份和社区标签中使用，还是也影响首页角色资产和主题强调色。
