# 《摸鱼隐者》暗黑像素忍者 UI 设计规范

本规范用于固化小程序用户端的暗黑像素忍者 RPG 视觉体系。当前事实来源已同步至 `refine-miniapp-layout-and-entry-hierarchy`：用户端以深色夜幕背景、暗色 RPG 面板、像素插画、语义 icon、正式字体层级和原生组件承载动态信息，不再沿用早期开发态占位框、无语义圆点 icon、功能宫格堆叠或手动记账横幅。

## 视觉基调

- 应用视觉身份：暗黑像素忍者 RPG + 上班工具。
- 世界观表达：`隐者大陆`、`隐者`、阵营身份、上班生存补给和 RPG 成长反馈应统一到同一套暗色视觉系统。
- 默认背景：主 tab 和主要二级页使用深色夜幕、黑蓝或深紫黑作为整页背景。
- 面板风格：使用暗色 RPG 面板、细描边、稳定阴影和克制辉光，避免每个模块都抢主角。
- 强调色：金、紫、蓝、青、绿、橙等只用于金额、能量、等级、奖励、风险、状态和关键行动。
- 局部浅色：纸张、卷轴、吐槽气泡、图标高光或插画道具可以使用浅色，但不得把整页恢复为浅色背景。

## 信息层级

- 首页第一屏优先表达今日已摸金额、工作状态、距离下班秒级倒计时和今日进度条。
- 首页隐藏模式是主金额旁的小眼睛即时开关，用于临时遮挡金额类敏感信息并切换低调视觉状态。
- 首页快捷入口只保留真实可用的核心入口，移除低价值、重复或未来态按钮。
- 首页今日生存消耗是一张大卡，展示 `隐者食堂`、`下午续命`、`通勤` 三类金额和 `查看详情` 入口，不表达为 `记一笔`、签到、任务或商城入口。
- 社区页保持高密度论坛体验，优先保证分区、筛选、发布、消息和帖子列表的扫读效率。
- 个人中心移除未定义完整玩法的成就首屏占位和空徽章入口，改为展示本人状态摘要、真实可用入口和阵营插画卡。
- 补给铺表达为上班生存补给工具，不做商城首页、券墙、价格墙、购物车或收货地址。

## 页面布局

### 首页

首页应包含顶部身份区域、已摸金额主卡、上班状态场景、主金额隐藏开关、核心功能入口、必要倒计时、今日生存消耗大卡和底部 tab。

首页主卡要求：

- 主信息为今日已摸金额、工作状态、距离下班倒计时和今日进度条。
- 已工作时长、约等于时间、进度百分比等解释型信息不得抢占主信息层级。
- 主 banner 为普通状态和隐藏状态预留稳定插画位。
- 金额、倒计时、进度条和按钮必须由原生组件渲染，不得整屏切图。

### 社区

社区页应包含顶部标题、漫画/IP 或隐者大陆内容 Banner、分区 tabs、筛选控件、消息入口、发布入口、帖子卡列表和底部 tab。

社区页要求：

- Banner 必须承载可理解内容入口、世界观入口、活动或后续内容入口，不得只显示“敬请期待”。
- 帖子卡保持固定信息密度，图片缩略图使用稳定比例和裁切，不因原图比例撑高列表。
- 发布、消息、分区和互动统计使用语义 icon，不使用无语义圆点。

### 个人中心

个人中心应包含资料卡、等级经验、资源统计、每日签到、本人状态摘要、精简菜单区域、我的角色入口、阵营插画卡和底部 tab。

个人中心要求：

- 顶部身份卡展示当前头像、昵称、改名 icon、阵营、称号、等级经验条和当前阵营徽章。
- 头像区域不得展示上传角标、上传按钮、本地相册入口或微信头像授权暗示。
- 本人状态摘要展示非月薪的工作档案摘要和今日生存消耗摘要。
- 菜单保留 `我的角色`、`工作设置`、`我的帖子`、`收藏帖子`、`论坛消息` 和 `生存账本`。
- 菜单移除“称号收藏”、“我的评论”、“隐者大陆百科”、“设置”、“我的成就”、称号商店、皮肤商店和赛季排行等未实现或未定义入口。
- 下半区保留左侧入口列表和右侧阵营插画卡排版。

### 补给铺

补给铺应围绕 `隐者食堂`、`下午续命` 和 `通勤补给` 三类上班生存场景组织。

补给铺要求：

- 每个板块展示短标题、场景说明、精选补给项和明确行动按钮。
- 页面不展示购物车、收货地址、价格墙、券墙、平台促销瀑布流、佣金、返利、聚推客字段或后台同步状态。
- 转链失败、空板块和加载态使用风格化状态图或语义 icon。

### 主要二级页

工作设置、生存账本、日报、大陆新闻、WebView 回退页、登录/建档、空状态和错误态应统一使用暗色背景、正式字体层级、RPG 面板、语义 icon 和必要插画预留位。

## 字体规范

字体分层如下：

- 品牌标题、金额数字、等级、倒计时、badge、入口标题和短标签可使用正式像素字体。
- 长正文、帖子内容、表单说明、错误提示和账本明细优先使用系统字体或可读中文字体。
- 像素字体未覆盖字符、加载失败或低端设备表现异常时，必须使用配置好的 fallback，不得出现乱码、方块字、不可读字形或明显排版跳变。
- 字体资源必须记录名称、来源、授权和使用范围，不得提交来源不明或不可商用字体。

当前字体选型记录位于 `apps/miniapp/src/static/fonts/FONT_SELECTION.md`。如正式字体文件尚未完整入库，页面应先使用安全 fallback，不能因缺失字体阻断构建。

## 插画与 icon 规范

- 插画资产使用稳定 asset key、稳定文件命名、固定比例、最小高度和文字叠加安全区。
- 首页主 banner、隐藏模式 banner、社区 banner、个人中心阵营插画、补给铺板块图、日报/大陆新闻封面、生存账本空状态、登录/建档和错误态应有稳定预留位。
- 正式插画缺失时，使用同风格剪影、图案底纹、暗色场景、色块面板或默认图标作为 fallback。
- 不得把 `px-* 200×200`、`placeholder`、`badge_placeholder`、TODO 文案、文件名或无语义圆点 icon 展示为正式 UI 内容。
- 金额、倒计时、等级、经验、按钮、tabs、列表、表单、标签和状态文本必须由原生 UI 承载，不能作为静态图片切片。

## 组件与交互

- 页面应由可复用组件和稳定样式 token 组合，不依赖整屏静态图。
- 功能入口、菜单项、tabs、chips、按钮和底部 tab 的可点击区域应满足移动端触控尺寸。
- 高密度卡片必须有稳定行高、间距、截断和换行策略，避免小屏文字重叠。
- 强发光、稀有色和主按钮样式只用于关键状态或主行动。
- 未解锁、敬请期待、禁用和隐藏状态必须通过 feature registry 或统一状态视觉表达，不在页面内散落硬编码。

## 工程约束

- 页面优先使用 `packages/ui-tokens` 的基础 token 和语义 token。
- 新增资产优先通过 `apps/miniapp/src/visual-system/illustration-registry.ts` 或等价稳定映射引用。
- 新增页面应同步检查 `openspec/specs/visual-system/spec.md`、当前 change 的 `specs/visual-system/spec.md` 和 `AGENTS.md` 的 UI 原则。
- 提交前至少运行 `corepack pnpm exec prettier --check <changed-files>`、`corepack pnpm lint`、`corepack pnpm typecheck` 和受影响的小程序验证/构建命令。
