## Context

当前 `work-value-tracker` 正式规格已经定义工作档案结构、校验规则、本地计算、未配置状态和隐私边界。`packages/shared/src/work-profile.ts` 已经提供类型、校验、快照和计算函数。

小程序端首页仍使用视觉演示金额和静态倒计时，`salary_work_time_settings` 默认功能入口目前指向个人中心页。后端 API 只有健康检查和功能入口读取接口，Prisma schema 也尚未定义业务模型。项目尚未实现真实微信登录，但工作档案契约已经要求“本人读取”和敏感字段不公开。

这个 change 是工作档案契约落地的第一步：让用户可以配置、保存、回读自己的薪资和上班节奏，并为后续首页真实计算提供 `WorkProfileSnapshot`。

## Goals / Non-Goals

**Goals:**

- 新增小程序工作档案设置页，覆盖第一版工作档案必填字段和隐藏模式开关。
- 通过小程序端统一服务读取和保存工作档案，页面不直接散落请求和回退逻辑。
- 新增后端本人工作档案读取与保存 API，复用共享包中的类型和校验规则。
- 新增 Prisma 工作档案持久化模型，保存薪资、工时、工作日、发薪日和隐藏模式。
- 更新 `salary_work_time_settings` 功能入口，让首页“薪资时间”进入设置页。
- 使用明确标记的临时本人上下文支持开发验证，后续可替换为真实微信登录用户身份。

**Non-Goals:**

- 不实现真实微信登录、openid 绑定、session 或完整用户资料体系。
- 不替换首页演示金额和倒计时为真实计算结果。
- 不接入法定节假日、调休、请假、大小周或复杂排班。
- 不实现税前/税后、个税、社保、公积金、奖金或真实工资结算。
- 不实现后台工作档案管理、运营统计或公开展示工作档案。
- 不实现轻量记账的账单模型或记账流程。

## Decisions

### Decision: 设置页使用独立非 tab 页面

新增工作档案设置页，例如 `pages/work-profile/settings`，并通过 `salary_work_time_settings` 入口导航进入。个人中心可以后续增加入口，但不把表单直接嵌入个人中心。

Rationale: 工作档案包含敏感字段和较多表单项，独立页面能降低个人中心复杂度，也便于从首页配置引导、功能入口和后续个人设置菜单复用同一路由。

Alternatives considered:

- 直接在个人中心页内展开设置表单。实现少一个页面，但会把敏感薪资设置混入资料展示页，后续维护成本更高。
- 使用首页弹窗配置。路径短，但表单项较多，不适合第一版完整配置。

### Decision: API 使用 `/me/work-profile` 语义

后端新增本人工作档案接口：`GET /me/work-profile` 读取，`PUT /me/work-profile` 保存。响应结构复用 `GetWorkProfileResponse` 和 `SaveWorkProfileResponse`。

Rationale: 工作档案只属于本人敏感设置，`/me` 语义能避免暴露可传入用户 ID 的接口，也和未来登录态自然衔接。

Alternatives considered:

- 使用 `/users/:userId/work-profile`。更通用，但第一版容易误导客户端可以读取他人工作档案。
- 只做本地存储，不做 API。速度最快，但无法验证已定义的本人读取、保存和隐私边界。

### Decision: 第一版使用临时本人上下文

在真实微信登录实现前，API 使用明确标记的临时开发用户 ID 作为本人上下文，例如来自配置或服务内常量。该机制只用于本地开发和端到端验证，后续由微信登录 change 替换。

Rationale: 当前没有用户身份系统，但工作档案设置需要 API 和数据库链路才能真正落地。临时本人上下文可以推进设置页能力，同时避免把微信登录塞进本 change。

Alternatives considered:

- 在本 change 中实现微信登录。契约更完整，但会显著扩大范围并牵连用户资料、授权和 session。
- 要求客户端传 `userId`。实现简单，但会削弱“本人工作档案”的安全语义。

### Decision: 持久化模型按契约字段存储

Prisma 新增工作档案模型，以 `userId` 唯一约束保存一份工作档案。月薪、币种、薪资模式、上班开始/结束、工作日规则、发薪日和隐藏模式使用明确字段；休息段和自定义星期可使用 JSON 字段保存契约结构。

Rationale: 核心字段直接建列便于后续查询和迁移，数组类字段保留契约结构，避免为第一版午休场景过早拆分多张表。

Alternatives considered:

- 将整个工作档案作为单个 JSON 存储。实现快，但字段边界、迁移和局部校验不清晰。
- 为休息段单独建表。结构更规范，但第一版只需要少量休息段，成本偏高。

### Decision: 小程序端服务负责请求、校验响应和错误归一

新增 `work-profile` 小程序服务，负责调用 API、校验响应结构、处理网络错误和校验错误。设置页只处理表单状态、展示错误和导航反馈。

Rationale: 现有功能入口已经采用服务层封装远端读取和本地回退，工作档案也应保持相同边界，避免页面直接耦合 API 细节。

Alternatives considered:

- 页面直接调用 `uni.request`。短期代码更少，但请求错误、响应校验和后续缓存会散落在页面内。

### Decision: 本 change 只保存快照，不替换首页计算

保存成功后返回 `WorkProfileSnapshot`，小程序可在设置页展示保存成功或返回首页，但首页真实金额替换留给后续 change。

Rationale: 设置页、API、持久化已经跨越小程序、后端和数据库。首页替换涉及演示逻辑清理、计时器、隐藏模式视觉和倒计时展示，适合单独验证。

Alternatives considered:

- 同时替换首页真实计算。用户价值更完整，但 change 范围过大，回归风险更高。

## Risks / Trade-offs

- [Risk] 临时本人上下文被误认为生产登录能力 -> Mitigation: 代码、文档和任务中明确标记为开发占位，后续由微信登录 change 替换。
- [Risk] 工作档案包含敏感薪资和工时，接口或日志可能泄露 -> Mitigation: API 只提供 `/me` 读取/保存，不实现公开读取，不在功能入口或公开资料中返回工作档案字段。
- [Risk] JSON 字段存储休息段和星期数组导致后续查询不便 -> Mitigation: 第一版不做后台分析和复杂查询；未来运营分析需要时再迁移为更细表结构。
- [Risk] 不替换首页计算会让保存后的价值感不完整 -> Mitigation: 设置页保存后返回快照，并在任务中保留后续首页消费衔接验证。
- [Risk] 表单字段较多影响移动端体验 -> Mitigation: 页面按“薪资、时间、工作日、发薪日、隐私”分组，使用移动端原生选择器和明确错误文案。

## Migration Plan

1. 新增 Prisma 工作档案模型和迁移，生成 Prisma Client。
2. 新增后端工作档案服务与 `/me/work-profile` 控制器，接入临时本人上下文。
3. 新增小程序端工作档案服务，封装读取、保存和响应校验。
4. 新增小程序设置页和 `pages.json` 路由。
5. 更新默认功能注册表中 `salary_work_time_settings` 的公开路由。
6. 增加验证脚本或测试，覆盖 API 保存读取、敏感字段边界、feature route 和小程序服务行为。

Rollback strategy: 如果 API 或数据库迁移出现问题，可以暂时把 `salary_work_time_settings` 路由恢复到个人中心或配置为不可用，同时保留共享契约不变；数据库迁移按 Prisma 本地迁移流程回滚或重新生成。
