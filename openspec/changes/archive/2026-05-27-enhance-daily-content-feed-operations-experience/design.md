## Context

隐者日报当前已经具备公开摘要、栏目文章、文章详情、点赞、评论、引用发帖和后台草稿审核发布能力。现阶段的主要问题不在公开接口是否存在，而在运营编辑链路不够完整：后台无法自然地增删、排序和校验文章；来源完整性更多依赖发布时校验而不是编辑时反馈；数据库模式下栏目和文章内容需要明确持久化更新策略；日报评论也需要从内存状态升级为可持久审核资产。

本 change 聚焦日报功能成熟化，不包含测试种子、一键测试发布或绕过审核的测试工具。

## Goals / Non-Goals

**Goals:**

- 后台可完整编辑今日参悟、大陆新闻和离谱卷宗，包括增删、排序、正文、摘要、来源和配图字段。
- 后台编辑时可看到来源完整性、配图缺失、正文长度、栏目文章数量等问题提示。
- 后台提供发布前预览，预览结果与小程序公开响应结构一致，但不绕过审核和发布状态。
- 数据库模式下保存日报时同步持久化栏目和文章内容。
- 日报文章评论、审核状态、AI 审核痕迹和人工复核结果持久化。
- 小程序日报页面在无配图、缺来源、评论待审核、引用不可用等状态下有清晰表达。
- 保持官方日报人工审核发布原则，AI 只做辅助处理，不自动发布日报。

**Non-Goals:**

- 不提供测试种子、一键测试发布或本地测试绕审入口。
- 不恢复第三方来源自动采集作为本 change 的核心范围。
- 不实现复杂 CMS 权限系统、多人协同编辑、版本差异对比或定时任务调度器。
- 不实现日报评论回复、评论通知或评论点赞；这些可在社区通知成熟后另行扩展。

## Decisions

### 决策 1：将后台编辑契约显式化

后台不再只提交整份 `DailyContentIssue` 的模糊结构，而是使用稳定的编辑契约表达栏目、文章、排序、来源字段和删除/新增后的结果。服务端仍以整份 issue 校验作为最终防线，但数据库更新要显式同步 section 和 item。

替代方案：继续让前端提交完整对象并由主表更新兜底。该方案在内存模式可用，但数据库模式下容易遗漏子表更新，且难以给出字段级错误。

### 决策 2：发布前预览复用公开响应转换

后台预览接口使用与公开接口一致的转换函数生成摘要、栏目列表和文章详情预览，但只允许后台令牌访问，并标记为预览结果。这样可以减少“后台看到的内容”和“小程序看到的内容”不一致。

替代方案：后台前端自行拼预览。该方案实现快，但会复制公开响应逻辑，容易遗漏来源过滤和内部字段隐藏边界。

### 决策 3：评论持久化独立建模

新增日报文章评论持久化模型，保存文章 ID、用户 ID、作者公开快照、评论正文、状态、AI 审核痕迹、人工复核信息和时间字段。公开文章详情只返回 `approved` 评论。

替代方案：复用社区评论表。日报评论和社区帖子评论的目标对象、审核后台和展示场景不同，强行复用会让查询和隐私边界变复杂。

### 决策 4：来源完整性在编辑期和发布期双重校验

编辑期给出问题提示，不阻断保存草稿；发布期对 `world_intel` 和 `absurd_casefile` 执行强校验，缺少来源 URL 或来源标题/站点时不得公开发布。

替代方案：只在发布时校验。该方案会让运营在最后一步才发现大量问题，编辑体验差。

## Risks / Trade-offs

- 子表同步更新复杂度提升 → 采用事务处理 issue、section、item 的更新，必要时先删除重建当前 issue 的子表，保持排序和字段一致。
- 预览接口可能误暴露内部字段 → 预览转换复用公开过滤函数，并由后台接口额外包装后台状态。
- 评论表新增后历史内存评论无法迁移 → 当前内存评论不视为生产事实，不迁移；上线后以数据库评论为准。
- 编辑器能力扩大后后台 UI 更复杂 → 第一版采用单页编辑分区布局，后续再拆分为独立文章编辑抽屉或详情页。
- 发布期严格来源校验会阻止无来源内容公开 → 今日参悟不要求来源，两个文章栏目必须有来源，这是日报可信度底线。

## Migration Plan

1. 扩展共享契约，增加日报编辑请求、编辑校验结果、预览响应、评论持久化字段。
2. 新增日报评论 Prisma 模型，补充迁移和服务层读写。
3. 调整数据库模式下日报 issue 保存逻辑，同步更新 `DailyContentSection` 和 `DailyContentItem`。
4. 增加后台预览、编辑校验和评论审核接口。
5. 更新后台页面，支持文章增删排序、来源提示、预览和持久评论审核。
6. 更新小程序日报页面空状态、来源展示、评论状态反馈和引用不可用提示。
7. 更新日报相关验证，覆盖编辑保存、预览、评论持久化和公开响应边界。
