## Context

当前用户成长资料已经支持微信登录、首次创建档案、职业类型到阵营映射、默认昵称、默认头像 key、等级经验、隐币、能量、连续签到和本地快照缓存。小程序个人页已经能读取 `/me/profile`、创建资料和调用 `/me/daily-checkin`，但页面仍包含占位成就、无效菜单、可上传本地头像预览和未接入真实资产的 key 文本展示。

本次变更跨越共享契约、API、Prisma、小程序页面和静态资产目录，因此需要把“个人页身份中心”的数据模型与视觉呈现先定清楚。

## Goals / Non-Goals

**Goals:**

- 让个人页的隐币、能量、经验、连续签到和签到按钮状态都来自真实用户成长资料或合法本地快照。
- 为四个阵营建立默认头像、默认徽章和个人页阵营插画的稳定 key 与小程序资产映射。
- 删除自定义上传头像能力，明确后续头像只能来自项目自制头像库，并可通过隐币解锁和装备。
- 新增本人资料编辑边界：昵称修改、职业文本填写、职业类型调整、阵营推荐和手动切换阵营。
- 清理个人页无效入口，使“我的角色”成为角色详情与资料编辑入口。
- 明确开发测试环境下签到不持久化的原因和验证方式，避免把内存 fallback 误判为正式数据丢失。

**Non-Goals:**

- 不实现完整头像商店、头像购买记录、运营头像后台或头像资产审核。
- 不开放用户自定义上传头像。
- 不回刷历史社区帖子、评论和回复中的作者快照。
- 不新增完整后台账号体系、用户封禁体系或复杂成就系统。
- 不把职业文本用于强制阵营迁移。

## Decisions

### 1. 阵营资产使用稳定 key，前端映射到本地静态文件

共享包维护阵营到默认资产 key 的映射，例如：

- `avatar_key_shadow_default`
- `badge_key_shadow_default`
- `scene_key_shadow_default`

小程序端维护 key 到实际静态文件路径的映射。用户后续补充的根目录插画应整理到小程序资产目录，例如 `apps/miniapp/src/static/profile/factions/<faction>/`，避免长期散落在仓库根目录。

原因：用户成长资料和 API 响应只需要稳定 key，不需要暴露小程序文件路径；后续换图不应触发数据库迁移。

备选方案是后端直接返回图片 URL，但当前默认资产属于小程序内置资源，URL 化会增加本地开发和小程序包构建复杂度。

### 2. 当前装备头像和徽章保存在成长资料，解锁列表先预留

第一阶段继续使用 `avatarKey` 表示当前头像，并将默认 `equippedBadgeKeys` 改为阵营默认徽章 key。可在共享契约中预留 `unlockedAvatarKeys` 或以后单独 change 扩展，不要求本次实现完整商店。

原因：本次核心是删除上传头像并保证默认阵营资产正确；头像解锁涉及货币扣减、库存、价格和幂等购买，适合后续独立 change。

### 3. 改名、职业填写和阵营切换走本人资料更新接口

新增或扩展本人资料更新接口，接受有限字段：

- `displayName`
- `jobTitle`
- `professionType`
- `faction`

服务端负责校验昵称、职业文本、职业类型和阵营合法性。职业类型变更后系统返回推荐阵营；当用户选择不切换阵营时，服务端必须保留当前阵营。

原因：个人资料是本人私有数据，必须由服务端持久化并返回统一快照，不能只停留在小程序本地缓存。

### 4. 阵营推荐不强制切换

职业类型仍用于计算推荐阵营，但不再等同于当前阵营的唯一来源。首次创建档案时职业类型决定初始阵营；之后用户可以手动保留或切换阵营。

原因：产品设定中职业是现实身份，阵营是用户的隐者身份。推荐可以降低选择成本，但强制迁移会削弱角色归属感。

### 5. 签到持久化以数据库配置为准，mock 登录身份需要稳定

API 当前在没有 `DATABASE_URL` 时使用内存存储，这种状态只适合开发预览。若 `WECHAT_LOGIN_MOCK_ENABLED=true` 且 mock openid 由每次登录 code 生成，测试中可能出现“每次都是新用户”的现象。实现和验收时应提供稳定 mock 用户方式，或直接使用 MySQL + Prisma 验证持久化。

原因：这能区分真实保存缺陷和开发环境身份不稳定问题。

## Risks / Trade-offs

- [风险] 修改用户成长资料字段可能需要 Prisma migration，已有测试数据缺少新字段。
  缓解：迁移提供默认值；读取旧记录时用阵营默认资产和空职业文本兜底。

- [风险] 小程序静态资产路径和构建产物路径不一致导致图片不显示。
  缓解：集中封装资产 key 到路径的解析函数，并在小程序个人页验证四个阵营默认头像、徽章和插画。

- [风险] 删除上传头像后，用户短期内无法自定义头像。
  缓解：用阵营默认头像和后续头像库方向补足，并避免展示“上传”入口造成误导。

- [风险] 职业类型和阵营可不一致后，部分社区阵营权限逻辑可能误用职业类型。
  缓解：社区发帖权限继续只读当前 `faction`，职业文本和职业类型仅用于个人资料与推荐提示。

- [风险] 顶部改名入口过小导致误触或不可发现。
  缓解：使用清晰编辑 icon，并保证整个 icon 按钮满足移动端点击区域。

## Migration Plan

1. 扩展共享契约，增加阵营默认徽章、阵营插画 key、职业文本和资料更新请求/响应。
2. 增加 Prisma migration，为用户成长资料补充必要字段，并为旧资料提供默认值。
3. 扩展 API，支持资料更新、昵称修改、职业资料保存、阵营推荐返回和手动阵营切换。
4. 整理根目录头像、徽章和后续插画到小程序静态资产目录，接入统一资产解析。
5. 改造个人页 UI，删除无效板块和上传头像入口，新增改名 icon、我的角色详情和阵营插画展示。
6. 补充验证，分别覆盖数据库持久化、内存 fallback 提示、默认资产、资料编辑和小程序展示。

回滚时可保留数据库新增字段不使用，前端恢复到读取基础 `UserGrowthProfileSnapshot`；新增接口不影响已有 `/me/profile` 和 `/me/daily-checkin` 语义。

## Open Questions

- 昵称是否需要消耗隐币改名，还是第一版免费不限次数。
- 职业文本是否需要低成本内容审核，还是仅做长度和敏感词基础校验。
- 阵营切换是否需要冷却时间或隐币成本。
- 阵营插画最终原图比例是否统一为 `5:7`，小程序展示是否固定为 `300rpx × 420rpx`。
