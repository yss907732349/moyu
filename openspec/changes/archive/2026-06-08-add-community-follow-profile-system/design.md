## Context

社区当前已经具备微信登录、隐者档案、作者公开快照、发帖、评论、回复、点赞、收藏、举报、内容安全、作者自见和后台治理。作者信息目前主要服务于内容展示，用户从帖子、评论或回复中看到作者后，缺少进入公开个人页、查看公开帖子和建立轻量关注关系的路径。

当前活跃的 `harden-community-identity-and-ip-attribution` 正在补齐社区发布身份门槛、手机号验证和 IP 属地。这个 change 不重复定义手机号验证或 IP 解析，只消费其公开侧允许展示的 `IP属地` 口径：若公开个人页可获得最近一次公开发布行为的降精度属地，则展示；不得展示完整 IP、手机号验证状态或后台合规状态。

## Goals / Non-Goals

**Goals:**

- 为社区作者提供稳定可点击的公开个人页入口。
- 提供公开个人页，展示公开身份、关注/粉丝/公开发帖统计和公开帖子列表。
- 提供单向关注、取消关注、关注列表和粉丝列表。
- 将“我的”页顶部三项状态数据调整为关注数、粉丝数和连续签到天数。
- 暂时隐藏隐币和能量的普通用户 UI 展示，避免未定义用途的成长资源提前承诺玩法。
- 保持社区公开响应、公开个人页和关注列表的敏感字段过滤边界。

**Non-Goals:**

- 不做私信、拉黑、好友确认、动态流、关注流、推荐算法、排行榜、用户搜索或新增粉丝通知。
- 不改变社区帖子、评论、回复的审核生命周期和作者自见规则。
- 不新增头像上传、头像商店、称号商店、隐币/能量获取消耗玩法或完整成就系统。
- 不开放后台关注关系运营面板，不在后台展示关注关系作为治理依据。
- 不公开手机号、真实姓名、完整 IP、微信身份字段、登录态、薪资、工作时间、工作档案、隐币、能量、生存账单、CPS 来源、后台治理状态或内容安全内部风险信息。

## Decisions

### 1. 使用稳定公开标识进入个人页

社区作者公开快照新增或关联 `publicProfileId` 一类稳定公开标识，小程序通过该标识进入公开个人页。该标识不应使用微信 `openid`、手机号、昵称，也不应直接暴露可枚举的内部自增 ID。

Alternatives considered:

- 直接用内部 `userId` 路由：实现简单，但可枚举风险更高，也会把内部身份语义带入公开 API。
- 用昵称路由：可读性好，但昵称可修改、可重复，也会制造历史内容跳转不稳定问题。

### 2. 作者快照保持历史展示，公开个人页展示当前资料

帖子、评论和回复继续使用发布或渲染时的作者公开快照展示内容流中的作者信息；用户点击后进入公开个人页，公开个人页展示该用户当前允许公开的资料、统计和公开帖子列表。历史帖子上的昵称或头像与当前主页不完全一致时，这是可接受的快照差异。

Alternatives considered:

- 所有历史内容动态读取当前资料：可以保持一致，但会让历史内容被资料修改影响，且增加列表查询成本。
- 公开个人页展示历史快照：跳转后的资料可能过旧，不符合用户对个人主页的预期。

### 3. 关注关系为单向幂等关系

关注关系使用 `followerUserId + followingUserId` 的唯一约束或等价机制保存。关注自己必须拒绝；重复关注返回已关注状态，不重复加计数；取消未关注用户返回未关注状态，不把计数减成负数。粉丝数和关注数可以冗余存储以提升读取性能，但关系表或关系事实应作为最终事实源。

Alternatives considered:

- 只做实时聚合不冗余计数：一致性更直接，但公开个人页和“我的”页高频读取会反复聚合。
- 关注后必须对方确认：会扩展成好友关系，不符合第一版轻关注范围。

### 4. 访问门槛沿用社区完整访问边界

未登录或未创建隐者档案的用户点击作者头像、昵称、关注数或粉丝数时，小程序先引导登录和建档。已登录且已建档用户可以查看公开个人页、关注列表和粉丝列表；关注/取消关注也要求登录且已建档，但不强制手机号验证，因为它不是发帖、评论或回复类公开发布行为。

Alternatives considered:

- 未登录也可看公开个人页：更开放，但当前社区详情和互动已经采用登录建档边界，先保持一致更稳。
- 关注也强制手机号验证：合规更重，但会把发布门槛扩大到非发布互动，增加不必要摩擦。

### 5. 关注不驱动分发、不发通知

第一版关注只影响关注状态、关注/粉丝统计和列表，不新增关注流，不改变推荐分区排序，不产生新增粉丝通知，也不影响社区消息未读数。后续若要做关注流、推荐或通知，需要新建 change。

Alternatives considered:

- 关注后新增通知：反馈更强，但会牵动 `community-notification`、消息未读、聚合和打扰控制。
- 关注影响推荐排序：更像社交产品，但会引入推荐算法和冷启动策略，超出本次范围。

### 6. “我的”页三项数据保留紧凑布局

“我的”页顶部身份区域保持三项状态数据布局，依次展示关注数、粉丝数和连续签到天数。关注数、粉丝数可点击进入对应列表；连续签到保留既有签到语义，不变成关注关系入口。隐币和能量继续作为内部数据或后续玩法字段存在，但当前普通 UI 不展示。

Alternatives considered:

- 只展示关注和粉丝两项：会让原三项布局出现空缺，也会误删已有连续签到反馈。
- 继续展示隐币和能量：当前缺少完整用途和消耗场景，容易让用户以为未开放玩法已经上线。

### 7. 公开列表只展示最小公开身份

公开个人页、关注列表和粉丝列表只返回头像、昵称、阵营、等级、称号、公开标识、关注状态和必要统计。公开个人页的帖子列表只包含 `approved` 且未 `hidden/removed` 的公开帖子；总发帖数也只统计这类帖子。

Alternatives considered:

- 列表展示更多资料和关注时间：信息更丰富，但会扩大公开字段范围，第一版不需要。
- 公开主页展示评论和回复：能体现活跃度，但审核、上下文和隐私解释成本更高。

## Risks / Trade-offs

- [Risk] 计数和关注关系不一致。→ Mitigation: 关注/取消关注保持幂等，验证脚本覆盖重复点击、取消未关注和计数不为负；必要时以后端关系事实重算。
- [Risk] 公开标识可枚举导致批量爬取。→ Mitigation: 使用不可预测公开标识，分页限制列表读取数量，公开响应过滤敏感字段。
- [Risk] 关注按钮快速点击造成竞态。→ Mitigation: 服务端唯一约束和事务保护，客户端展示处理中状态并以后端返回为准。
- [Risk] 公开个人页被理解为完整社交主页。→ Mitigation: 页面只展示公开身份、关注关系和公开帖子，不放动态流、私信、排行榜或推荐入口。
- [Risk] 隐币/能量隐藏影响后续成长玩法。→ Mitigation: 仅隐藏普通 UI，不删除内部字段；后续通过新 change 恢复展示和定义玩法。
- [Risk] 与 IP 属地 change 落地顺序不同。→ Mitigation: 公开个人页对 `IP属地` 做能力兼容：有合法降精度属地则展示，无可公开发布记录或字段未落地时隐藏或展示未知，不伪造属地。

## Migration Plan

1. 扩展共享契约，定义公开个人页、公开用户列表项、关注状态、关注统计和公开作者路由标识。
2. 扩展数据模型，新增关注关系持久化结构和必要索引；为用户生成稳定公开标识。
3. 扩展社区公开响应，使帖子、评论和回复作者头像/昵称具备可跳转公开个人页的标识。
4. 新增公开个人页、关注/取消关注、关注列表、粉丝列表和公开个人帖子列表 API。
5. 调整小程序“我的”页顶部三项数据为关注、粉丝、连续签到，并隐藏隐币/能量普通 UI。
6. 新增小程序公开个人页和关注/粉丝列表，接入社区详情作者点击。
7. 更新验证脚本和人工内测清单，覆盖敏感字段过滤、计数幂等、分页、空状态和小屏布局。

Rollback: 新增关注关系和公开个人页 API 可先不在入口暴露；若关注能力异常，小程序可临时隐藏关注按钮和列表入口，但不得回退为展示隐币/能量或伪造统计。数据模型新增字段应保持可空或可延迟生成，避免阻断既有社区内容读取。

## Open Questions

- `publicProfileId` 的具体生成策略使用短随机 ID、UUID 派生还是内部 snowflake 映射，需结合现有用户模型和数据库约束在实现时确定。
- 公开个人页是否在第一版展示 `IP属地：未知`，还是无合法属地时隐藏该行，需要与正在推进的社区身份/IP change 最终视觉口径保持一致。
