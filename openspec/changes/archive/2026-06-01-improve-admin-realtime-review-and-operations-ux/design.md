## Context

后台运营中心当前通过 `GET /admin/operations/workbench`、`GET /admin/operations/review-queue` 和各模块详情接口聚合社区审核、举报、日报评论复核、日报运营和补给铺排查。前端在进入后台时执行整页同步，操作后也较多依赖重新拉取完整数据。

现有后台权限边界是 `x-admin-token`，尚未引入完整后台账号系统。统一审核队列已经要求过滤薪资、工作档案、微信身份、用户登录态、AI 原始 prompt、供应商原始响应和聚推客密钥等敏感字段。本变更需要在保持这些边界的前提下增加实时待办能力和更稳的后台交互。

## Goals / Non-Goals

**Goals:**

- 为后台运营中心提供实时待办事件流，使新审核、新举报、日报评论复核和工作台计数变化能被及时感知。
- 在实时连接不可用时自动降级轮询，并在连接恢复后停止轮询。
- 将实时事件作为“有变化”的提示，不直接替代工作台、审核队列和详情接口的事实源。
- 优化统一审核队列的信息密度、按钮层级、操作反馈和局部更新。
- 保护运营人员正在填写的备注、治理原因、日报正文和补给配置表单，不被后台自动刷新覆盖。
- 统一社区治理、日报运营和补给铺排查的刷新、返回、筛选保留、加载态、错误态、空状态和操作反馈。
- 继续执行后台令牌校验和敏感字段过滤。

**Non-Goals:**

- 不引入完整后台账号系统、角色权限系统或多运营人员协同锁定。
- 不实现 WebSocket 双向协同；本期只需要服务端到后台的待办事件提示。
- 不把实时事件推送给普通小程序用户。
- 不改变普通用户端社区、日报、补给铺或生存账本能力。
- 不承诺跨多 API 实例的强一致实时广播；事件流只作为提示，轮询和事实源接口负责收敛。

## Decisions

### 使用 `fetch` 消费 SSE，而不是原生 `EventSource`

后台新增 `GET /admin/operations/events`，响应 `text/event-stream`。前端使用 `fetch` 发起请求并携带 `x-admin-token`，通过 `ReadableStream` 解析 SSE 数据。

原因：当前后台统一使用 `x-admin-token`。原生 `EventSource` 无法稳定携带自定义请求头，如果把长期后台令牌放在 query 参数中，容易被浏览器历史、代理日志或服务端访问日志记录。`fetch` 流式读取可以延续现有令牌边界，并避免新增完整账号系统。

备选方案是新增短期 SSE ticket：先用 `x-admin-token` 换取短期票据，再用原生 `EventSource` 连接。该方案更接近标准浏览器 SSE，但需要新增票据签发、过期和撤销逻辑，本期收益不足。

### 实时事件是提示，不是事实源

事件 payload 只包含最小必要字段，例如：

```text
eventId
eventType
source
targetType
targetId
createdAt
pendingCount
queueDelta
```

前端收到事件后只累加“有 X 条新待办”并更新连接状态；只有用户点击提示、切换筛选或执行操作完成后，才通过现有事实源接口刷新工作台、队列或详情。

原因：审核队列项可能包含原文、作者摘要、风险信息和可用操作。直接在 SSE 中推送完整队列项会增加敏感字段泄漏面，也容易和前端正在编辑的详情发生覆盖冲突。

备选方案是在事件中推送完整队列项，前端直接插入列表。该方案看起来更实时，但会复制一套字段过滤和状态冲突处理逻辑，风险更高。

### 断线降级为低频轮询

前端维护连接状态：

```text
connecting -> live -> degraded_polling -> offline
```

当 SSE 连接失败或超时时，后台进入降级轮询，定期读取工作台摘要或队列轻量状态。轮询发现计数变化时仍只展示新待办提示。SSE 恢复后停止轮询。页面隐藏时降低轮询频率或暂停非必要刷新。

原因：后台实时能力不能成为单点体验依赖。运营人员至少要能通过轮询感知新待办，不因代理、浏览器或本地网络问题完全失去刷新提示。

### 自动刷新需要尊重“脏编辑状态”

前端为审核备注、治理原因、日报编辑器、补给项配置表单和详情处理面板维护本地脏状态。存在脏状态时，实时事件和轮询不得自动覆盖当前模块数据，只展示顶部提示。用户显式点击刷新时，后台先提示可能覆盖未保存内容，再执行刷新或保留当前编辑。

原因：后台运营操作往往伴随备注、原因和正文编辑。强刷会造成内容丢失，也会让运营人员难以判断当前操作是否已经提交。

### 队列操作以局部状态收敛为优先

审核队列项在前端增加处理态：`idle`、`processing`、`succeeded`、`failed`、`stale`。操作发起后当前行进入 `processing`；成功后根据当前筛选移出、更新状态或更新可用操作；如果服务端返回状态已变化，则标记为 `stale` 并提示刷新。

原因：整页刷新会打断当前上下文，也会让运营人员失去刚处理事项的反馈。局部状态能提升处理效率，并和实时提示形成更清楚的节奏。

### 一致化后台模块交互，不重写业务能力

社区治理、日报运营和补给铺排查沿用现有业务接口和敏感字段边界。本变更只统一以下后台交互：

- 返回入口和当前模块刷新入口。
- 筛选条件保留。
- 列表、详情、表单的加载态、空状态、错误态和无结果状态。
- 操作成功、失败、令牌失效、状态已变化和重复处理反馈。

原因：这个 change 的主线是运营效率和实时感知，不应顺手重写日报、社区治理或补给铺业务流程。

## Risks / Trade-offs

- **SSE 在部分环境中不可用** → 使用降级轮询，并把事实源保持在普通 HTTP 接口。
- **单实例内存事件流无法跨进程广播** → 本期事件只作提示；轮询和重新拉取事实源负责最终收敛。后续如部署多实例，再引入 Redis pub/sub 或消息队列。
- **实时提示和队列事实源短暂不一致** → 前端展示“有新待办”而不是直接插入完整数据，用户点击后从事实源刷新。
- **操作被重复处理或状态已变化** → 服务端返回可理解错误，前端将行标记为 `stale` 并提示刷新。
- **模块统一交互导致单文件后台继续膨胀** → 实现时优先抽取后台请求、实时连接、反馈状态和列表状态管理，不把所有新逻辑继续堆在页面模板中。
- **敏感字段通过事件流泄漏** → 事件流使用独立 DTO 和验证脚本黑名单，不复用包含详情原文或内部排查字段的响应对象。

## Migration Plan

1. 先扩展共享类型和 API 事件 DTO，不影响现有后台页面。
2. 新增后台事件流接口和最小事件发布机制；未连接时不影响现有接口。
3. 前端接入连接状态和顶部提示，默认仍可通过手动刷新使用后台。
4. 接入降级轮询和新待办计数提示。
5. 优化审核队列局部操作状态和反馈。
6. 统一社区治理、日报运营、补给铺排查的状态展示和刷新语义。
7. 扩展验证脚本，确认令牌、敏感字段、降级轮询和不打断编辑规则。

回滚时可关闭前端实时连接入口，保留原有工作台、审核队列和模块接口；服务端事件流接口即使存在也不影响旧后台使用。

## Open Questions

- 是否需要为后台事件增加保留窗口和 `Last-Event-Id` 恢复能力，还是第一版只依赖轮询收敛？
- 页面隐藏时轮询频率采用固定 60 秒，还是完全暂停到页面重新可见？
- 审核队列处理成功后是否自动聚焦下一条待办，还是仅作为可选偏好保留？
