## Context

社区评论和回复已经具备低成本内容审核、本地敏感词库、文本归一化、重复内容判断、频率限制和后台人工复核队列。现有规格允许可信用户低风险短评论快速公开，但仍使用 `MAY` 语义，实际实现容易保守地把大量正常评论送入人工复核。

本 change 的核心不是取消审核，而是把评论/回复从“人工为主”调整为“自动分流为主”：明显安全自动公开，明显违规自动驳回，只有灰区和风险用户进入人工复核。帖子仍保持人工审核主导。

## Goals / Non-Goals

**Goals:**

- 降低社区评论和回复进入人工复核队列的比例。
- 明确评论/回复审核决策结果：`auto_approve`、`auto_reject`、`manual_review`。
- 为可信用户建立可实现的快速通过条件，并为新用户、异常用户、被举报用户和受限用户建立收紧策略。
- 保留公开侧只展示 `approved` 内容的边界。
- 后台审核队列只聚焦需要人工判断的灰区内容，并展示分流原因和风险摘要。
- 提供验证脚本，确保低风险可信评论/回复不再默认进入人工复核。

**Non-Goals:**

- 不让帖子自动公开。
- 不引入付费内容安全供应商或新的外部依赖。
- 不实现机器学习风控、复杂信誉模型或自动封禁。
- 不开放评论图片。
- 不改变微信身份、工作档案、生存账单等隐私边界。

## Decisions

### 评论/回复使用三通道审核决策

新增或扩展共享审核决策契约：

```text
auto_approve   自动通过，内容状态直接为 approved
auto_reject    自动驳回，内容状态为 rejected 或拒绝公开
manual_review  人工复核，内容状态为 pending
```

决策应记录来源、原因、风险标签、用户风险摘要、命中字段和是否命中快速通过条件。公开响应不得返回内部命中词细节；后台可以展示归一化摘要。

备选方案是继续沿用 `pass`、`reject`、`review`。该方案实现改动小，但难以表达“文本低风险但用户不可信，因此仍需人工复核”的复合原因。三通道决策更适合后台解释和验证。

### 快速通过是评论/回复默认目标，不适用于帖子

评论和回复是高频、短文本、互动型内容，符合条件时应自动公开。帖子包含标题、正文、图片和更高传播面，仍创建 `pending` 并等待人工审核。

低风险可信评论/回复应满足：

```text
已登录并已创建隐者档案
内容长度在短文本阈值内
本地硬规则无明确违规和灰区命中
未触发重复内容或频率限制
用户近期无被驳回、有效举报、禁言、封禁或 limited 治理状态
用户满足基础信任门槛，例如账号已完成冷启动期或已有通过记录
```

### 用户风险作为快速通过门禁

快速通过不只看文本，也看用户风险。建议第一版使用确定性规则而不是复杂评分：

```text
new_user        冷启动用户，前 N 条评论/回复进入人工复核或更严格分流
trusted_user    有一定通过历史且无近期违规，可走快速通过
limited_user    治理状态为 limited，取消快速通过
violated_user   近期有驳回或有效举报，取消快速通过
rate_limited    触发频率或重复规则，拒绝或进入人工复核
```

这样可以在不引入复杂风控系统的前提下降低误放风险。

### 后台队列只承接人工复核项

`auto_approve` 和 `auto_reject` 不应进入“待人工处理”队列，但后台治理总览和内容详情仍可查看其审核摘要。统一审核队列中评论/回复项应主要是 `manual_review`，并支持按风险标签、用户风险原因和来源筛选。

### 小程序提示跟随最终状态

提交评论/回复后，小程序应基于 API 返回状态展示：

```text
approved  已发布
pending   审核中
rejected  未通过
```

不要把所有评论提交都提示为“待审核”，否则用户会感知不到快速通过能力。

## Risks / Trade-offs

- [Risk] 自动通过误放违规内容。→ Mitigation: 仅对短文本、无命中、低频、无近期违规的可信用户开放；举报和后台隐藏后自动收紧用户后续快速通道。
- [Risk] 新用户互动延迟仍存在。→ Mitigation: 冷启动期只限制前几条评论/回复，通过后快速转入可信通道。
- [Risk] 自动驳回误伤。→ Mitigation: 明确违规才自动驳回；灰区进入人工复核，用户侧展示可理解原因。
- [Risk] 队列计数口径变化影响后台理解。→ Mitigation: 工作台区分待人工复核数量和自动处理数量，自动处理只作为统计或治理详情信息。
- [Risk] 规则过严导致减负效果不明显。→ Mitigation: 验证脚本覆盖可信低风险评论/回复必须自动公开，并覆盖风险用户不得快速通过。

## Migration Plan

1. 扩展共享契约，增加评论/回复审核决策、用户风险原因和后台展示字段。
2. 调整 API 评论/回复创建流程，把本地审核结果、用户风险和频率去重组合为三通道决策。
3. 更新后台队列，只把 `manual_review` 评论/回复作为待处理项，保留自动处理摘要展示。
4. 更新小程序提交反馈，按返回状态展示已发布、审核中或未通过。
5. 增加验证脚本，覆盖可信用户自动通过、风险用户人工复核、明显违规自动驳回和帖子不自动公开。
6. 回滚时可将快速通过开关关闭，使评论/回复重新进入 `manual_review` 路径；数据状态仍兼容 `approved`、`pending` 和 `rejected`。
