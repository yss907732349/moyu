## Context

现有日报后台以 `DailyContentIssue` 为主要编辑对象。前端在一个编辑面板中维护 `selectedIssue.sections`，通过 tab 切换 `daily_reflection`、`world_intel`、`absurd_casefile`。保存时调用整期 `PUT /admin/daily-content/issues/:issueId`，发布当前板块前也会先执行整期保存。

后端已经存在 `publish-sections` 能力，但它依赖前端先保存整期草稿，且校验与预览仍以整期为单位。结果是运营人员即使只想更新一个板块，也会经过整期数据提交、整期校验和整期按钮组，操作边界不清晰。

## Goals / Non-Goals

**Goals:**

- 将后台日报运营的主工作单元从整期 issue 拆到单个板块记录。
- 提供板块级保存接口，确保保存一个板块不会覆盖其他板块草稿。
- 提供板块级校验和预览，让运营只看到当前板块的阻断项、提示项和预览结果。
- 保留已有表结构的兼容读取，但后台新建、编辑和发布以单板块记录为准。
- 保持小程序公开侧日报结构兼容，不要求新增小程序页面或路由。

**Non-Goals:**

- 不引入完整后台账号和权限系统。
- 不改变日报公开侧“今日参悟 + 两个栏目入口 + 文章列表/详情”的结构。
- 不引入复杂栏目版本管理、撤回单篇文章、多人协同编辑或差异合并。
- 不改变日报评论审核、社区引用和 AI 辅助的基础规则。

## Decisions

### 1. 增加板块级保存接口，而不是继续复用整期 PUT

新增或等价实现：

```text
PUT /admin/daily-content/issues/:issueId/sections/:sectionKey
```

请求体只包含目标板块的 `summary`、`illustrationKey`、`items` 等可编辑字段。新建草稿时后台先选择 `sectionKey`，服务端创建只包含该板块的记录；保存时只替换该记录内的目标 section。

理由：继续复用整期 PUT 会让前端必须提交完整 `sections`，仍然存在覆盖其他板块未同步草稿的风险。板块级接口可以把“只更新一个板块”变成后端明确保证的行为。

备选方案是前端本地 merge 后继续调整期 PUT。这个方案改动小，但不能从 API 契约上防止误覆盖，不适合这次“完全独立”的目标。

### 2. 板块级发布使用“保存当前板块后发布当前板块”

单板块发布流程调整为：

```text
当前板块编辑状态
  -> PUT /issues/:issueId/sections/:sectionKey
  -> GET 或服务端内部执行板块级校验
  -> POST /issues/:issueId/publish-sections { sectionKeys: [sectionKey] }
```

`publish-sections` 不再复制 issue，也不归档同日其他记录。它只把当前板块记录设为 `published`，并保留其它板块草稿或已发布内容不变。

理由：隐者日报不是一期期发的内容，而是三个独立板块各自更新。发布一个板块不得删除、清空或归档另外两个板块。

### 3. 板块级校验复用现有校验模型并按 section 过滤

共享层可以保留现有 `DailyContentEditValidationResult`，并扩展为支持 `sectionKey` 参数的校验函数或服务端过滤逻辑。前端展示当前板块时只展示该板块相关问题；整期检查仍展示所有问题。

理由：现有校验 issue 已包含 `sectionKey` 和 `itemId` 字段，优先复用可以减少契约扩散。对于 `title`、`homeSummary` 这类整期字段，归入整期设置检查，不混入单板块发布检查。

### 4. 后台 UI 使用单板块编辑器

日报运营页面保留左侧记录列表。新建按钮先选择要创建的板块，选中记录后主编辑区只展示该记录所属板块，包含：

- 内容编辑表单
- 新增/删除/排序文章能力
- 保存板块
- 预览板块或预览文章
- 发布板块
- 当前板块发布检查

理由：tab 或三板块同屏仍会暗示它们属于同一期日报。单板块编辑器能让运营明确知道当前只在处理一个板块。

### 5. 公开侧聚合最新已发布板块

公开侧读取日报摘要时，服务端按 `daily_reflection`、`world_intel`、`absurd_casefile` 分别选择最新已发布记录并组合成一个公开响应。某个板块未发布时展示对应空状态，不影响其它板块公开。

理由：小程序仍需要一个日报入口，但后台运营不再以“期”为发布单位。

## Risks / Trade-offs

- [Risk] 复用 `DailyContentIssue` 表承载单板块记录，命名上仍有 issue 概念。
  Mitigation: API 和后台文案统一表达为板块记录，公开侧通过组合函数隐藏底层兼容细节。

- [Risk] 整期发布与单板块发布并存后，运营可能不清楚哪个是最终公开状态。
  Mitigation: UI 将整期操作与板块操作分区展示，并在板块面板显示最近保存/发布状态。

- [Risk] 板块级校验过滤不完整，可能隐藏外部采编来源缺失等阻断项。
  Mitigation: 校验函数按 `sectionKey` 和 `itemId` 精准过滤，外部采编来源完整性必须在当前板块发布前检查。

- [Risk] 前端状态管理仍共享 `selectedIssue.sections`，局部保存后容易展示旧状态。
  Mitigation: 板块保存成功后用服务端返回的 issue 或 section 回写本地状态，并刷新对应板块校验。

## Migration Plan

1. 先补充共享契约和后端板块级接口，保持现有整期接口可用。
2. 重构后台日报 UI，使三个板块工作区调用板块级保存和发布。
3. 更新验证脚本，覆盖只保存一个板块时其他板块不变、只发布一个板块时其他板块问题不阻塞。
4. 若发布后发现问题，可临时回退前端入口到整期保存/发布按钮；后端新增接口保持向后兼容，不需要数据迁移。

## Open Questions

- 单板块发布后，公开侧是否需要把同一天已发布的不同板块聚合成一个日报摘要，还是沿用“发布副本”策略只展示最新发布副本。实现前需要以现有公开读取逻辑为准做最小兼容。
