## Context

当前小程序社区页只提供暗黑忍者 RPG 风格的视觉演示和本地模拟互动，不具备真实帖子、评论、审核、举报或身份边界。`user-growth-profile` 已提供微信登录、隐者档案、阵营、等级、称号和头像 key，可作为社区公开身份来源；`work-value-tracker` 和 `accounting-ledger` 已明确薪资、工作时间、生存账单、消费统计等信息不得公开暴露。

`community-lite` 第一阶段需要在真实实现前固化契约：未登录用户只能看到推荐流预览，登录并创建隐者档案后才能进入完整社区；阵营专区强化身份归属；社区内容公开前必须人工审核。

## Goals / Non-Goals

**Goals:**

- 定义帖子、评论、互动、举报和审核生命周期的第一阶段契约。
- 定义社区分区、阵营专区发帖规则和未登录/未建档访问边界。
- 定义公开作者身份快照，复用用户成长资料中的公开字段。
- 定义普通公开接口只返回已人工审核通过的内容。
- 为后续 API、小程序页面、共享契约和最小管理审核能力提供实现拆分。

**Non-Goals:**

- 不实现私信、关注关系、实时聊天、在线状态或复杂社交网络。
- 不实现完整推荐算法、完整运营后台、复杂权限系统或无人工参与的自动发布。
- 不实现图片上传资产流、图片审核供应商接入或生产美术资产。
- 不将工作档案、薪资、工作时间、生存账单、CPS 平台来源或消费统计接入社区公开展示。
- 不把阵营专区设计为浏览权限墙；非本阵营用户仍可浏览和互动。

## Decisions

### 1. 社区访问以隐者档案为完整身份门槛

未登录或未创建隐者档案的用户只能浏览推荐分区帖子列表预览。用户点击帖子详情、切换到推荐以外分区、发帖、评论、点赞、收藏或举报时，小程序跳转到“我的”页，引导完成微信登录和隐者档案创建。

Rationale: 社区需要稳定的作者身份、阵营和审核追踪。只依赖匿名浏览会让互动、举报和阵营专区规则不可执行。保留推荐预览可以降低冷启动门槛。

Alternative considered: 完全禁止未登录访问社区。实现更简单，但新用户无法先感知社区内容，转化阻力更高。

### 2. 阵营专区只限制发帖，不限制浏览和互动

键影、运影、策影、行影专区只允许对应阵营用户发帖；所有已登录且已创建隐者档案的用户都可以浏览、查看详情、评论、点赞、收藏和举报。推荐和魔王吐槽不限制发帖阵营。

Rationale: 阵营专区需要有身份归属感，但第一阶段不应把内容浏览做得过窄。互动开放可以保持社区流动性。

Alternative considered: 阵营专区只允许本阵营浏览和互动。归属感更强，但会放大冷启动内容不足，并引入更多权限和提示分支。

### 3. 帖子和评论都采用审核状态机

帖子和评论提交后进入 `pending`；只有人工审核通过后成为 `approved`，才允许进入公开列表、公开详情和公开评论区。`rejected` 表示审核驳回，`hidden` 表示已公开内容因管理处理或举报风险被隐藏。

Rationale: 职场吐槽和社区内容有明显合规与内容安全风险。公开前人工审核是 MVP 的安全边界，规则检测或 AI 风险评分只能辅助人工判断。

Alternative considered: 先发后审或 AI 自动审核通过。体验更流畅，但与“社区内容公开前必须经过审核生命周期”的项目原则冲突，且风险不可控。

### 4. 作者公开身份使用发帖时快照

帖子和评论公开展示的作者信息使用用户成长资料中的公开字段生成快照，包括昵称、头像 key、阵营、等级、称号和可选徽章 key。工作档案、薪资、工作时间、隐藏模式、生存账单、消费统计、微信 openid/unionid 和内部敏感标识不得进入公开响应。

Rationale: 快照可以避免用户未来改阵营、改称号或资料缺失导致旧帖展示漂移，也能让公开接口不需要额外读取敏感本人资料。

Alternative considered: 每次公开展示实时联查用户成长资料。数据更新更即时，但旧帖身份语义会漂移，接口更容易误暴露非公开字段。

### 5. API 以服务端校验为准

小程序端负责引导跳转、隐藏不可用操作和展示提示，但 API 仍必须校验登录态、隐者档案、阵营发帖权限、内容状态和互动对象状态。普通公开列表和详情只能返回 `approved` 内容。

Rationale: 客户端限制无法作为安全边界。服务端统一校验可避免绕过小程序页面直接调用接口。

Alternative considered: 仅在小程序端限制未登录和阵营发帖。实现快，但无法防止接口绕过，也不利于后续管理后台和多端复用。

## Risks / Trade-offs

- 人工审核会降低发帖和评论的即时反馈 → 提交后明确展示“待审核”状态，后续可增加本人内容状态列表或消息通知。
- 推荐分区预览可能让未登录用户误以为可以完整浏览社区 → 点击详情和切换分区时统一跳转“我的”页，并用明确提示说明需要登录并创建隐者身份。
- 阵营专区限制发帖可能让用户选错阵营后体验受限 → 第一阶段沿用用户成长资料的初始阵营，后续如开放改阵营需单独 OpenSpec change。
- 评论也需要审核会增加管理成本 → 第一阶段只定义最小审核队列和状态，不要求复杂运营工作台。
- 作者身份快照可能与用户当前资料不一致 → 社区内容重视发布时身份语境，接受快照一致性优先于实时资料同步。
