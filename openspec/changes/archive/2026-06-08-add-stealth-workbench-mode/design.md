## Context

小程序用户侧当前主视觉已经稳定为暗黑忍者 RPG 和隐者营地信息流，首页、社区、日报和个人中心均依赖插画、icon、像素面板、头像徽章和深色层级建立产品识别。但“摸鱼隐者”的使用场景天然发生在办公室，用户需要一个可快速进入、低暴露、像工作表的浏览界面，用来查看今日已摸金额、下班倒计时、日报和论坛轻互动。

现有工程已经具备本 change 可复用的数据能力：

- 首页通过工作档案快照和 `calculateWorkValueState({ snapshot, now })` 计算今日金额、工作状态和倒计时。
- 日报已有公开摘要、栏目列表、文章详情和点赞/评论等客户端服务；摸鱼模式只需要读取和展示，不需要新增日报发布能力。
- 社区已有帖子列表、帖子详情、评论/回复、点赞和收藏客户端服务；摸鱼模式只复用列表、详情、点赞和收藏，不复用发帖、评论、回复、举报和消息链路。
- feature registry 已能按展示区域返回功能入口，并按入口状态决定导航、拦截或隐藏。

## Goals / Non-Goals

**Goals:**

- 新增一个独立的摸鱼模式，不替换现有首页、社区、日报和个人中心 UI。
- 以办公表格视觉展示核心工作数据、日报和论坛内容，形成“像工作表”的低暴露体验。
- 让摸鱼模式页面严格禁用插画、icon、头像、徽章、像素框、RPG 发光、深色营地背景和游戏化按钮。
- 复用现有工作价值计算、日报服务和社区服务，不新增后端 API 或数据库模型。
- 论坛支持浏览、详情、评论/回复只读、点赞和收藏；不允许用户在摸鱼模式提交新的社区内容或举报。
- 图片附件默认只显示数量，用户显式查看时才打开单张图片，并且查看层无动画、点击任意位置立即关闭。
- 建立统一移动端工作表布局规则，保证各板块行列均衡且小屏可读。

**Non-Goals:**

- 不做全站双 UI 皮肤系统。
- 不重做完整社区链路，不支持发帖、评论、回复、举报、消息中心、我的帖子或收藏管理。
- 不改变微信原生 tabBar 结构，不新增主 tab。
- 不接入真实 Excel、Office、WPS 或其他办公软件品牌、商标和样式资源。
- 不改变日报、社区、工作档案、内容安全、举报、审核、CPS、生存账本或用户成长资料的服务端契约。
- 不展示补给铺、生存账本、角色身份、签到、阵营、成就或商业转链能力。

## Decisions

### 1. 摸鱼模式使用独立页面，而不是全局主题开关

采用新增页面组的方式，例如：

- `pages/stealth-workbench/index`
- `pages/stealth-workbench/daily-list`
- `pages/stealth-workbench/daily-detail`
- `pages/stealth-workbench/forum`
- `pages/stealth-workbench/forum-detail`

理由：现有页面体量较大，社区详情包含评论输入、回复、举报、图片和审核提示等复杂状态。全局主题开关会要求每个页面维护双 UI 分支，风险高且容易破坏已确认的主视觉。独立页面能把业务范围收紧为表格版浏览层。

备选方案是给首页、社区、日报和帖子详情加 `stealth` 样式分支。该方案复用路由更多，但会把普通模式和摸鱼模式的样式、交互、输入边界混在同一组件里，后续维护成本更高。

### 2. 数据能力复用现有客户端服务

摸鱼模式首页读取现有本地工作档案快照并调用共享计算函数；日报页面复用 `daily-content-feed` 服务；论坛页面复用 `community-lite` 服务中的列表、详情、点赞和收藏能力。

理由：本 change 本质是 UI 和入口体验，不需要改变后端事实源。复用现有服务可以避免重复契约、降低验证范围，并保持内容安全、作者自见、公开过滤和互动统计与普通模式一致。

### 3. 论坛为“只读 + 轻互动”

摸鱼模式论坛支持：

- 帖子列表
- 帖子详情
- 评论/回复只读展示
- 点赞帖子
- 收藏帖子
- 查看图片附件

摸鱼模式论坛不支持：

- 发帖
- 评论
- 回复
- 举报
- 图片上传
- 消息中心
- 我的帖子
- 我的收藏管理

理由：发帖、评论、回复、举报和图片上传都会带来内容安全、审核提示、作者自见、举报去重限流和通知等链路。第一版为了保持办公表格伪装和实现风险可控，只保留不会产生新公开内容的互动。

### 4. 图片默认隐藏为附件数量

帖子列表不展示缩略图。帖子详情以字段行展示 `附件数量 | 3 | 查看`。点击 `查看` 后展示附件清单，再点击单个附件打开全屏图片查看层。

全屏图片查看层要求：

- 不显示标题栏、关闭按钮、保存按钮、转发入口或说明文案。
- 不使用打开/关闭动画和淡入淡出。
- 点击屏幕任意位置立即关闭。
- 不支持滑动切图，避免与任意点击关闭冲突。

理由：图片是最容易破坏伪装感的内容，但完全不可看会损失论坛详情完整性。附件数量和按需查看能兼顾安全感与可用性。

### 5. 建立“移动端工作表”而不是完全复制桌面 Excel

摸鱼模式视觉采用工作表隐喻，但不复制任何真实办公软件品牌。布局规则为：

- 页面使用浅色办公背景和白色/极浅灰表格面。
- 板块标题行、表头行、普通字段行、列表行使用固定高度。
- 同类页面复用统一列宽规则，操作列、数字列、时间列保持固定宽度。
- 长标题最多两行截断。
- 正文、日报详情、评论正文使用合并单元格，可以自然换行，但必须遵守最大宽度、防溢出和合理间距。
- 小屏不做横向滚动，优先压缩字段、隐藏低优先级列或把行内容拆成两行。

理由：移动端屏幕无法承载桌面表格的完整列宽和横向滚动体验。移动端工作表规则能保留“行列均衡”的视觉感，同时保证正文和评论可读。

### 6. 操作文案伪装为办公语义

摸鱼模式中可将普通社区文案替换为低暴露办公语义：

- `点赞` 显示为 `标记`
- `收藏` 显示为 `归档`
- `隐者日报` 显示为 `内部日报`
- `论坛` 显示为 `讨论记录`
- `距下班` 显示为 `剩余时长`

底层仍调用原有点赞、收藏和日报/社区接口，不改变服务端语义。

理由：办公语义能强化伪装感，并减少页面一眼看出娱乐社区的概率。

### 7. 摸鱼模式作为视觉系统的明确例外

现有 `visual-system` 规格要求主要用户端页面保持暗色 RPG 视觉。摸鱼模式需要在规格中明确作为经 OpenSpec 定义的例外：它不得套用普通视觉基线中的插画、icon、像素框和暗色营地样式，但仍必须满足中文可读、小屏防溢出、普通用户内部字段隐藏和业务能力边界。

理由：如果不在视觉规格中写明例外，后续实现或验收容易被误判为“不符合暗黑忍者 RPG 基线”。

## Risks / Trade-offs

- [风险] 表格风过于朴素，被误认为未完成 UI。 → [缓解] 明确细网格、统一列宽、固定行高、表头、字段表、操作列和状态反馈，让它看起来是刻意设计的办公工作表。
- [风险] 移动端固定列宽导致长标题或正文不可读。 → [缓解] 列表标题最多两行，正文/评论使用合并单元格，小屏隐藏低优先级列并禁止横向溢出。
- [风险] 用户期待摸鱼模式也能发帖或评论。 → [缓解] 第一版将其定义为办公室安全浏览模式，只开放不会产生新公开内容的轻互动；后续如要开放输入，应另起 change 处理内容安全和审核提示。
- [风险] 图片查看层自定义实现可能与平台图片能力相比少了缩放或滑动。 → [缓解] 本模式优先“快和易关闭”，不追求相册能力；普通社区详情仍保留完整图片浏览体验。
- [风险] 新增入口可能挤压首页核心入口层级。 → [缓解] 通过 feature registry 控制 `stealth_workbench_mode` 展示区域和排序，默认只作为首页低调入口，不新增底部主 tab。
- [风险] 表格文案过度伪装导致用户不理解。 → [缓解] 入口仍可叫 `摸鱼模式` 或 `隐身工作台`，进入后页面标题和字段使用办公语义；关键金额和倒计时保持可识别。

## Migration Plan

1. 在共享功能入口配置中新增 `stealth_workbench_mode`，默认配置为可进入摸鱼模式页面的已开放入口。
2. 新增摸鱼模式页面组和专用样式，不改现有首页、社区、日报页面结构。
3. 首页或首页快捷入口接入该功能入口，按 feature registry 状态处理导航、拦截或隐藏。
4. 复用现有客户端服务接入工作数据、日报和论坛数据。
5. 增加验证脚本或现有小程序验证项，检查路由注册、入口配置、禁用能力和 UI 禁用资产规则。
6. 如需回滚，隐藏或禁用 `stealth_workbench_mode` 入口即可，不影响普通模式页面。

## Open Questions

- 摸鱼模式入口最终展示文案使用 `摸鱼模式`、`隐身工作台` 还是 `工作表模式`，实现时可结合首页入口层级确定。
- 首页中金额字段在摸鱼模式内显示为 `今日已摸金额` 还是更伪装的 `今日累计`，建议默认用 `今日累计`，但可在实现前再确认。
