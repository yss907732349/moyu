## Context

`build-cps-supply-center-foundation` 已经建立补给铺、服务端点击归因、聚推客转链、订单同步和生存账本自动入账的第一版链路。当前小程序补给铺主要是三类补给项列表，后台主要是补给项配置、点击记录和订单同步记录；这些能力足够证明链路可走通，但还不足以支撑长期运营、异常排查、投放调优和对账。

本变更在现有 `supply-center`、`accounting-ledger` 和 `admin-operations` 上深化，不新建独立商业系统。普通用户仍只看到“上班生存补给”和本人账本结果；聚推客、CPS、佣金、订单源 ID、完整 `sid`、原始同步响应和平台内部字段继续只在服务端或受控后台排查边界内使用。

## Goals / Non-Goals

**Goals:**

- 把补给铺用户侧重构为可复访的“今日补给面板”，围绕上班时段、工作日场景、标签和三类生存消耗组织内容。
- 让运营可以配置活动分组、标签、优先级、有效期、上下线、排序、复制和批量启停，并预览普通用户侧效果。
- 明确 `sid` 生命周期、转链有效期、订单归因窗口、重复点击、跨活动匹配和备用链接边界。
- 让聚推客订单同步具备幂等、状态历史、有效入账、退款回滚、异常池和失败原因。
- 建立后台 CPS 排查中心和指标面板，能够解释“为什么这单没入账”，并支撑点击到入账的漏斗分析。
- 继续保护用户隐私和合规边界，不承诺返利，不展示佣金，不公开消费数据。

**Non-Goals:**

- 不建设泛电商商城、购物车、收货地址、平台支付、返利钱包或用户提现能力。
- 不引入完整后台账号和权限分层系统，继续沿用现有后台令牌边界。
- 不把无归因订单人工强行分配给用户。
- 不向普通用户展示聚推客、CPS 平台、佣金、结算状态、订单源 ID 或后台排查字段。
- 不把后台经营指标公开到社区、排行榜或用户资料。

## Decisions

### 1. 用户侧以“场景补给面板”为主，而不是活动货架

小程序补给铺继续保留 `隐者食堂`、`下午续命` 和 `通勤补给` 三类板块，但首屏改为 `今日补给面板`。面板根据当前时间、工作日状态和活动配置展示主推荐位，例如午饭前突出食堂、下午突出咖啡茶饮、上下班前后突出通勤。活动卡片展示用户语义标题、场景描述、标签、可点击状态、行动文案和同步提示，不展示平台内部信息。

选择这个方案，是因为补给铺的复访理由来自“现在上班需要什么”，不是泛电商式“这里有很多券”。替代方案是继续扩展三栏列表，但它会越来越像券墙，既弱化项目世界观，也难以提升日常使用心智。

### 2. 活动运营模型在补给项上增量扩展

在现有补给项基础上扩展活动分组、标签、展示优先级、推荐时段、工作日规则、归因窗口、转链有效期和备用链接策略。后台操作围绕同一补给项模型提供新增、复制、批量启停、排序、有效期和预览，不另建一套与补给铺割裂的“广告投放系统”。

这样可以复用当前公开列表、点击归因和后台配置入口，同时给运营足够的可控性。替代方案是新建 campaign/ad placement 模型，但当前规模下会增加迁移和后台复杂度，适合后续多供应商、多渠道投放时再抽象。

### 3. 归因拆成点击、归因 token 和转链尝试

一次用户点击 SHALL 产生点击记录；点击记录关联一个不可反推用户身份的归因 token。归因 token 对外传递时使用原始 `sid`，服务端持久化和匹配优先使用 `sidDigest`，后台展示只使用 `sidMasked`。转链调用单独记录尝试状态、失败原因、返回目标和过期时间。

重复点击不通过“猜最近一次用户行为”来归因。订单携带的精确 `sid` 能匹配到有效 token 时才归属；`sid` 缺失、过期、活动不一致或 token 不存在时进入异常池。这样牺牲了一部分宽松匹配能力，但换来归因可信和隐私边界清晰。

### 4. 备用链接必须显式标记 attribution 语义

聚推客转链失败时，系统可以返回受控降级结果，但必须区分两种情况：可归因备用链接和不可归因备用链接。不可归因备用链接不得承诺订单会进入生存账本，也不得制造已转链成功的运营指标。没有合规备用链接时，小程序只展示补给暂不可用和重试提示。

这个设计避免为了短期转化把用户送到无法回流的入口，同时允许在供应商短暂不可用时保留用户体验。

### 5. 订单同步以来源订单幂等和状态机为核心

聚推客订单以 `sourceProvider + sourceOrderId` 作为幂等键。每次同步 SHALL 保存状态历史或等价审计信息，并基于当前状态更新账本：有效订单创建或更新账单；未支付、未知、风控、无效、退款或取消订单不计入统计；已入账订单后续变成退款或无效时，账本 SHALL 回滚为不计入。

替代方案是只保存最新同步记录，但无法解释状态变更、退款回滚和重复同步问题，不适合商业闭环的对账要求。

### 6. 后台排查中心以“链路追踪”组织

后台 CPS 排查中心围绕订单或点击生成链路视图：补给项、点击、归因 token、转链尝试、订单同步、状态映射、账本入账、失败原因和更新时间。排查视图同时提供原因码，例如 `sid_missing`、`sid_not_found`、`sid_expired`、`activity_mismatch`、`order_status_invalid`、`duplicate_order_updated`、`ledger_import_failed`。

这种视图直接回答运营最常见的问题：“为什么这单没入账？” 替代方案是继续分散在点击表和订单表中查看，成本低但排查效率差，且难以发现转化漏斗断点。

### 7. 指标来自同一事实链路

CPS 指标从补给展示、点击、转链、订单回流、有效订单和账本入账记录聚合，不单独维护不可追溯的计数。后台指标包含今日点击数、独立点击用户数、转链成功率、订单回流数、有效订单数、入账金额、异常订单数和三类生存分类统计。

第一版指标可以按请求实时聚合或基于轻量缓存计算；当数据规模增大后再引入异步汇总表。

## Risks / Trade-offs

- 用户误解为返利或补贴承诺 -> 小程序文案只表达补给、领券和账本自动收纳，不出现返利、佣金、到账收益或结算承诺。
- `sid` 脱敏和匹配模型迁移不当 -> 采用新增字段兼容旧记录，先双读旧 `sid` 和新 `sidDigest`，再逐步收敛到摘要匹配。
- 聚推客状态枚举变化导致误入账 -> 未识别状态默认进入异常池或不计入，验证覆盖未知状态。
- 备用链接提升体验但降低归因确定性 -> 备用链接必须标记是否可归因，不可归因时不得计入转链成功和入账预期。
- 指标实时聚合带来性能压力 -> 第一版限制日期范围和分页，必要时只聚合最近周期；后续再加汇总表。
- 后台字段过多导致敏感信息外泄 -> 后台响应使用字段白名单，验证脚本检查密钥、完整 `sid`、微信身份、用户登录态和完整原始响应黑名单。

## Migration Plan

1. 扩展共享契约和数据模型，新增活动标签、分组、优先级、归因窗口、转链尝试、失败原因、状态历史和指标响应字段，字段默认保持兼容。
2. 对现有补给项补齐默认分组、标签和优先级；现有 `隐者食堂`、`下午续命`、`通勤补给` 继续可用。
3. 对现有点击记录生成 `sidDigest` 和 `sidMasked`，保留旧 `sid` 兼容未完成回流订单；后台响应只返回脱敏值。
4. 先部署 API 和共享契约，再部署后台排查中心和小程序新面板；旧公开列表和点击接口保持兼容。
5. 扩展验证脚本覆盖旧数据兼容、重复同步、退款回滚、异常池、备用链接和敏感字段过滤。
6. 回滚时保留新增字段但恢复旧补给铺列表展示；订单同步继续按来源订单幂等处理，避免重复入账。

## Open Questions

- 第一版活动推荐是否只基于本地时间和工作日规则，还是同时读取用户工作档案的上班/下班时间来增强推荐？
- 备用链接是否允许配置不可归因的官方入口，还是第一版只允许可归因备用链接和不可用提示？
- 指标第一版是否需要导出 CSV 对账，还是只在后台页面展示筛选结果和汇总数？
