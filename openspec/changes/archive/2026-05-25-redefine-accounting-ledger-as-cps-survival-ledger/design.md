## Context

当前 `accounting-ledger` 已归档并落地为手动记账能力：共享包定义收入/支出、多分类、账单校验和统计函数；API 提供 `/me/accounting-ledger` 创建、列表、更新、作废和统计接口；小程序有 `记一笔` 页面、账单列表和首页账本横幅。

新的产品方向不再把该能力定位为专业记账软件，而是围绕商业 CPS 订单形成“生存账单”：用户通过外卖和打车 CPS 下单后，系统自动生成本人可见的生存消耗摘要、订单详情和本周生存报告。用户界面必须使用 `摸鱼侠` 世界观语言，不展示美团、饿了么、滴滴等平台名称。

## Goals / Non-Goals

**Goals:**

- 将账单事实源从用户手动输入改为 CPS 订单自动导入。
- 提供三类稳定生存分类：`隐者食堂`、`下午续命`、`通勤`。
- 首页底部区域展示今日三类生存消耗和 `查看详情` 入口。
- 详情页展示本人订单详情和本周生存报告。
- 隐藏普通用户 UI 中的 CPS 平台名称，同时保留服务端来源字段用于去重、退款/取消处理和排查。
- 明确旧手动记账数据与新 CPS 生存账单统计隔离。

**Non-Goals:**

- 不实现专业记账能力，包括手动记账、收入记录、多分类账本、预算、资产账户、财务净额、报表导出或发票管理。
- 不实现公交、地铁、自驾、手动通勤补录或非滴滴通勤来源。
- 不实现真实排行榜或真实“击败百分比”。
- 不在第一版实现 CPS 佣金结算后台、复杂归因分析或跨平台对账系统。
- 不把订单明细、消费来源或生存报告默认公开到社区、排行榜或公开资料。

## Decisions

### CPS 原始订单与生存账单展示模型分层

系统应把外部 CPS 原始订单与小程序展示用的生存账单快照分开。原始订单层保留 `sourceProvider`、`sourceOrderId`、订单状态、金额、下单时间、商品标题、距离等字段；展示层只输出分类、展示标题、金额、时间、状态和必要统计字段。

Rationale: CPS 平台字段、订单状态、退款和金额口径可能变化，分层可以避免小程序 UI 直接依赖平台细节，也便于隐藏平台名称。

替代方案是继续复用旧 `AccountingBillSnapshot` 并直接暴露来源字段。该方案改动短，但容易把平台字段泄露到 UI，也会让旧手动记账语义和新生存账单语义混在一起。

### 三类分类作为共享稳定枚举

共享包应提供生存分类枚举，第一版只包含 `canteen`、`afternoon_boost` 和 `commute`，展示名分别为 `隐者食堂`、`下午续命` 和 `通勤`。外卖订单默认归入 `隐者食堂`；命中下午茶关键词时归入 `下午续命`；滴滴打车订单归入 `通勤`。

Rationale: 三类分类足够支撑当前商业 CPS 接入和首页摘要，也符合用户不需要专业记账软件的方向。

替代方案是保留旧多分类。该方案会继续把产品拉向专业账本，并增加用户理解成本。

### 下午续命通过关键词规则识别

下午茶分类第一版使用商品标题、商品分类或商户标签中的关键词识别，例如咖啡、奶茶、茶饮、甜品、蛋糕、下午茶等。规则应放在共享契约或服务端可复用模块中，便于验证和后续运营调整。

Rationale: 用户明确要求下午茶按商品关键词分，而不是按下单时间。关键词规则简单、可解释，也适合第一版验证。

替代方案是按时间段自动归类。该方案会把午后正餐误归为续命，也不符合当前决策。

### 普通用户 UI 不展示 CPS 平台名称

服务端和数据库可以保留平台来源字段，但小程序普通用户响应和页面不得展示 `美团`、`饿了么`、`滴滴` 等平台名。用户看到的应是产品化文案，例如 `午间补给`、`冰咖啡补给`、`影遁通勤`。

Rationale: UI 需要保持 `摸鱼侠` 世界观一致，也避免把首页变成 CPS 平台订单列表。

替代方案是显示真实平台名。该方案便于用户核对订单，但会削弱产品包装，并可能增加平台合作展示合规风险。

### 本周生存报告先做规则统计，排行字段仅预留

本周生存报告第一版统计本人本周外卖次数、下午续命次数、通勤距离或订单数、三类消耗金额。`击败了 X% 的键隐者` 不接真实用户分布，规格只预留字段或占位，不表达为真实排行。

Rationale: 真实排行需要匿名聚合、阵营维度、反作弊、隐私边界和排行榜产品规则，应该随排行榜能力单独立项。

替代方案是在账单能力里直接做真实百分位。该方案会扩大范围，也容易误用敏感消费数据。

### 旧手动账单隔离而不是自动迁移

如果环境中存在旧手动账单数据，第一版生存账单不自动迁移、不混入新统计。实现时可以隐藏旧手动入口，保留旧数据表以便回滚或后续处理。

Rationale: 手动收入/支出和 CPS 生存消耗口径完全不同，自动迁移会造成统计污染。

替代方案是把旧支出按分类映射到三类。该方案会误导用户，以为历史手动账单来自 CPS 下单。

## Risks / Trade-offs

- [Risk] CPS 订单退款、取消或金额调整导致首页金额不准确 -> Mitigation: 生存账单状态必须跟随来源订单状态，取消和退款订单不得继续计入有效消耗。
- [Risk] 关键词规则误判下午续命 -> Mitigation: 第一版保留可验证关键词列表，并允许后续通过运营配置或共享规则扩展。
- [Risk] 平台名称从接口字段泄露到小程序 UI -> Mitigation: 普通用户响应提供展示快照，不返回或不使用平台展示名；验证脚本检查页面文案和响应字段。
- [Risk] 旧手动记账页面仍可通过路由进入 -> Mitigation: feature registry 和首页入口改为详情页，旧记录页下线或重定向。
- [Risk] 通勤距离字段在 CPS 数据中缺失 -> Mitigation: 报告中通勤距离可为空或展示订单次数，不能凭空造真实公里数。

## Migration Plan

- 新增或调整共享契约，定义 CPS 生存订单、三类分类、摘要和周报响应。
- 调整 API 模型和服务，新增 CPS 订单导入/归一化逻辑，并把旧手动账单接口下线、禁用或与新接口隔离。
- 改造小程序首页横幅为今日生存消耗摘要，`查看详情` 进入生存账单详情页。
- 改造或替换账单列表页，展示订单详情和本周生存报告；移除 `记一笔` 主流程。
- 更新 feature registry 中账单入口路由。
- 对已有旧手动账单数据不做自动迁移，新统计只读取 CPS 生存账单。
- 回滚时可以恢复旧入口和旧接口，但不得把新 CPS 生存订单误写为旧收入/支出账单。

## Open Questions

- 下午续命关键词第一版的完整词库是否放在共享包常量中，还是后续由后台运营配置。
- CPS 订单导入第一版使用模拟接口、后台管理导入还是真实平台回调，需在实现阶段结合可用 CPS 服务商确定。
- 本周报告中的通勤距离在数据缺失时展示 `通勤 N 次` 还是隐藏该行。
