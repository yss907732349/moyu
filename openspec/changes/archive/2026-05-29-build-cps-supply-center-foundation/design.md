## Context

`accounting-ledger` 已经把第一版记账能力定义为 CPS 生存账本，支持三类生存分类、订单去重、状态排除、首页今日摘要和本周报告；`feature-registry` 已保留 `supply_center`，但仍是敬请期待入口。当前缺口是用户没有自然的补给入口，CPS 点击没有归因记录，第三方订单也没有稳定流入账本。

本 change 选择聚推客作为第一期 CPS 接入方。聚推客提供活动列表、活动转链和统一订单数据接口；统一订单返回 `sid`、`order_sn`、`brand_id`、`act_id`、`pay_price`、`pay_time` 和订单状态，足以支撑点击归因、订单去重、分类和账本入账。

## Goals / Non-Goals

**Goals:**

- 建立补给铺第一期体验：`隐者食堂`、`下午续命`、`通勤补给` 三个板块。
- 选择符合上班日生存场景的第一期活动池，而不是泛商城货架。
- 建立服务端点击归因：生成不可反推用户身份的 `sid`，记录用户、补给项、活动和跳转时间。
- 通过聚推客转链接口把用户送往活动落地页，并优先使用目标小程序跳转信息。
- 通过聚推客统一订单数据接口拉取或导入订单，并映射为生存账本账单。
- 守住普通用户隐私边界：不展示 CPS 平台、佣金、订单源 ID、聚推客字段或后台归因字段。
- 让后台可以配置补给项、上下架、排序、有效期和基础同步排查信息。

**Non-Goals:**

- 不建设完整商城、购物车、搜索、SKU、库存、收货地址或支付能力。
- 不在第一期开放用户手动记账、手动补录订单或用户上传返利截图。
- 不展示佣金、返利、预计收益、平台订单源 ID 或第三方平台名称作为普通用户主文案。
- 不承诺订单实时同步；第一期允许通过定时拉取、后台触发或脚本导入完成订单同步。
- 不接入酒店、旅游、电影票、泛电商、会员充值等与当前三类生存账本弱相关的活动。

## Decisions

### Decision: 补给铺定位为“上班生存补给”，不是泛商城

第一期补给铺只展示三个板块：`隐者食堂`、`下午续命` 和 `通勤补给`。推荐活动池为：

- `隐者食堂`：美团外卖、饿了么。
- `下午续命`：瑞幸咖啡、库迪咖啡、奈雪、喜茶。
- `通勤补给`：滴滴/花小猪、高德打车。

替代方案是接入聚推客活动列表中的所有可用活动并做商城式分类。该方案短期可见内容更多，但会稀释产品主循环，并引入大量与生存账本无关的订单类型，所以不作为第一期范围。

### Decision: 使用聚推客统一订单数据作为账本入账事实来源

补给卡片点击只证明用户有意图，不能证明消费发生。生存账本只基于聚推客统一订单数据接口返回的订单生成或更新账单。订单使用 `order_sn` 或等价订单号去重，使用 `sid` 关联补给点击和用户，使用 `status` 映射为有效、待确认或排除状态。

替代方案是分别接美团、饿了么、滴滴的专用订单接口。该方案可以作为统一订单覆盖不足时的补充，但第一期优先使用统一订单接口降低适配面。

### Decision: `sid` 只作为脱敏归因令牌

服务端生成随机或签名化 `sid`，并保存 `sid -> userId + supplyItemId + actId + clickedAt` 的内部映射。`sid` 不得包含明文用户 ID、微信 openid、手机号、昵称或可反推用户身份的信息。

替代方案是直接把用户 ID 拼入 `sid`。该方案便于排查，但会把内部身份泄露给第三方和日志链路，不符合隐私边界。

### Decision: 用户端只展示产品语义，后台保留运营字段

普通用户看到的是 `午间饭票`、`咖啡回血`、`影遁通勤` 等补给语义，以及自己的三类生存消耗。聚推客 `apikey`、`brand_id`、佣金比例、佣金金额、订单源 ID、活动源名称等只允许出现在服务端配置、后台运营排查或内部日志中。

替代方案是在用户端展示平台名和优惠来源。该方案短期更直观，但与既有账本要求“普通用户界面不得展示 CPS 平台名称”的边界冲突。

### Decision: 订单分类优先继承补给项语义，再用订单字段兜底

当订单通过 `sid` 匹配到补给点击时，账本分类优先使用补给项配置的 `categoryKey`。未匹配点击或历史订单同步时，再根据 `brand_id`、`act_id`、`order_title` 和关键词规则兜底：外卖归入 `隐者食堂`，咖啡/茶饮/甜品归入 `下午续命`，打车出行归入 `通勤`。

替代方案是只用订单标题关键词分类。该方案对咖啡和奶茶可用，但对通勤或平台活动标题不稳定，容易把订单归错类。

## Risks / Trade-offs

- [Risk] 聚推客统一订单接口对部分活动覆盖不完整或字段不稳定。→ Mitigation：保留原始订单同步记录和失败原因，后台可查看未匹配订单；必要时后续 change 增加专用订单接口适配。
- [Risk] 用户点击后订单延迟回流，首页短时间仍显示 0。→ Mitigation：补给铺点击后只提示“订单同步后自动进入生存账本”，不承诺即时入账。
- [Risk] `sid` 泄露或可猜测导致第三方推断用户身份。→ Mitigation：`sid` 使用随机或签名化不可枚举值，且不包含明文用户身份。
- [Risk] 运营配置活动过多导致补给铺变成杂货铺。→ Mitigation：第一期限制三个板块和精选活动白名单；非生存场景活动不进入普通用户补给铺。
- [Risk] 订单状态口径与现有 `CpsOrderStatus` 不完全一致。→ Mitigation：建立聚推客状态到内部状态的显式映射，未付款、无效、退款和风控订单不计入有效消耗。

## Migration Plan

1. 新增补给铺共享契约、持久化模型和服务端配置，不影响现有账本读取。
2. 将 `supply_center` 入口先接入页面和 API，可通过功能注册表控制开放状态。
3. 上线补给项配置和点击归因，验证转链跳转和 `sid` 生成。
4. 接入统一订单同步，先在后台或脚本中验证订单匹配和入账，再对普通用户开放自动统计。
5. 若同步异常，可暂停 `supply_center` 入口或暂停订单同步任务；已入账订单仍按现有账本状态展示和排除。

## Open Questions

- 聚推客统一订单接口在真实账号下对美团、饿了么、瑞幸、库迪、奈雪、喜茶、滴滴/花小猪和高德打车的订单覆盖是否完全一致，需要用测试订单或沙盒数据确认。
- 微信小程序跳转第三方小程序时，目标 `app_id` 是否需要额外加入微信小程序跳转白名单，需要在发布配置中验证。
- 后台是否需要第一期支持手动重放订单同步，还是先通过脚本和接口验证即可。
