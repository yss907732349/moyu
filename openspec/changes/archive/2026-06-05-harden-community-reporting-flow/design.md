## Context

社区当前已经具备举报入口、`CommunityReport` 数据模型、用户侧举报客户端方法、后台统一审核队列和 `POST /admin/community/reports/:reportId/handle` 处理接口。现状更接近最小闭环：小程序用固定理由提交举报，后台可保留、隐藏或标记误报，但缺少用户可理解的举报面板、原因分类、重复举报约束、案件聚合、上下文取证、处理通知和有效举报对后续审核策略的明确影响。

本 change 面向论坛高要求治理场景。举报需要成为内容安全的后置人工复核入口，与微信内容安全、本地低成本审核、后台治理、社区通知和作者治理联动，但不得把举报人身份、微信身份、工作档案、生存账单、CPS 来源或内部供应商细节暴露给普通用户或不必要的后台界面。

## Goals / Non-Goals

**Goals:**

- 让用户侧举报成为明确、可提交、可反馈的功能，而不是固定理由按钮。
- 所有有效举报进入人工复核语义，运营可查看被举报内容、举报原因、上下文和历史风险后再处理。
- 防止重复举报、恶意刷举报和自己举报自己影响队列质量。
- 将同一目标的多条举报聚合为案件视图，降低后台处理负担。
- 将人工确认成立的举报结果纳入后续评论/回复风险收紧策略。
- 在举报人和内容作者之间建立克制通知边界，保护举报人身份。

**Non-Goals:**

- 不引入完整后台账号系统，仍沿用当前 `x-admin-token` 边界。
- 不做普通用户可见的举报处理公示页、仲裁页或申诉中心。
- 不因单次举报自动隐藏公开内容。
- 不开放用户上传举报证据图片；第一版仅使用平台内已有内容、图片和上下文作为证据。
- 不把举报系统扩展到补给铺、工作档案、生存账单或非社区内容。

## Decisions

### 1. 举报是人工复核入口，不是自动处罚入口

所有通过服务端校验的举报 SHALL 进入待处理状态或合并到待处理案件，由后台人工查看内容和上下文后处理。系统可以自动计算优先级、合并重复目标和拦截无效请求，但不得仅因单次举报自动隐藏内容。

Rationale: 论坛举报容易被用作攻击工具。人工复核能降低误伤，同时满足高要求社区对处置可追踪、可解释的需要。

Alternatives considered:

- 单次举报自动隐藏：安全感强，但误伤和恶意举报风险高。
- 多人举报自动隐藏：适合成熟风控，但当前缺少用户可信度、申诉和仲裁体系。

### 2. 用户侧使用结构化原因，服务端保留兼容

小程序提交举报时使用稳定 `reasonCode`，可选 `reasonText` 作为补充说明；“其他”原因必须填写补充说明。服务端可临时兼容历史 `reason` 字段，并映射为 `other`，避免已有客户端或验证脚本瞬间失效。

建议第一版原因：

| reasonCode       | 展示文案 |
| ---------------- | -------- |
| `illegal`        | 违法违规 |
| `harassment`     | 辱骂骚扰 |
| `privacy`        | 隐私泄露 |
| `spam`           | 广告诈骗 |
| `sexual`         | 色情低俗 |
| `misinformation` | 不实误导 |
| `other`          | 其他     |

Rationale: 结构化原因方便后台筛选、优先级排序和验证；补充说明保留用户上下文。

### 3. 用 `targetKey` 做去重和聚合

服务端为举报目标生成稳定 `targetKey`，例如 `post:<id>`、`comment:<id>`、`reply:<id>`。同一用户对同一 `targetKey` 只允许存在一条未撤销举报；重复提交返回已受理状态和原举报状态，不创建新记录。同一 `targetKey` 下的多条举报可被后台作为同一案件聚合展示。

Rationale: MySQL 对多 nullable 外键唯一约束不够直观，`targetKey` 能让去重、列表查询和案件聚合更稳定。

### 4. 不新增强制案件表，先做目标维度案件投影

第一版不强制新增 `CommunityReportCase` 表。后台通过 `targetKey` 聚合 `CommunityReport`，生成案件列表和案件详情。若后续需要独立案件 SLA、分派、多人协作或申诉，再引入独立案件表。

Rationale: 当前后台为单人/轻运营模式，投影足以支撑内测与早期治理；少一张状态表也降低迁移和同步复杂度。

### 5. 内容处理和举报处理分开记录

举报处理结果记录在举报记录上，内容隐藏、移除和作者治理记录在社区治理审计里。一次举报处理可以同时更新举报状态、内容状态、作者治理状态和通知，但这些结果需要分别可查。

建议举报状态扩展为：

- `pending`：待人工复核
- `kept`：已处理，内容保留
- `hidden`：举报成立，内容已隐藏
- `removed`：举报成立，内容已移除
- `false_report`：举报明显不成立或存在滥用迹象

Rationale: “保留内容”和“误报”不是同一种运营判断；“隐藏”和“移除”也需要在后续风险统计中区分。

### 6. 通知只表达必要结论

举报人收到“举报已受理”和“举报已处理”类克制反馈，不暴露具体处罚、后台备注、被举报作者身份扩展信息或其他举报人信息。内容作者只在内容被隐藏、移除或作者受到治理限制时收到可公开原因，不知道举报人是谁。

Rationale: 举报系统必须保护举报人，也不能制造围观和对抗。

### 7. 有效举报影响后续低成本审核

只有人工处理为 `hidden` 或 `removed` 的举报，才计为被举报作者近期有效举报；`pending`、`kept` 和 `false_report` 不得直接降低被举报作者的自动通过资格。`false_report` 可以作为举报人风险输入，用于举报频率和滥用判断。

Rationale: 现有低成本审核已经引用“有效举报”，本 change 明确有效举报来源，避免未处理举报造成误伤。

## Risks / Trade-offs

- [Risk] 举报量上升导致运营压力变大。→ Mitigation: 同目标聚合、重复举报去重、优先级排序和后台案件详情减少重复判断。
- [Risk] 恶意举报影响正常用户。→ Mitigation: 单用户去重、频率限制、自己举报自己拦截、误报状态和举报人风险记录。
- [Risk] 自动隐藏不足以处理紧急风险。→ Mitigation: 隐私泄露、违法违规和短时间多名用户举报进入高优先级；后续如需要再定义临时折叠或临时隐藏状态。
- [Risk] 举报通知泄露处理细节。→ Mitigation: 通知模板仅表达受理、已处理、内容被处理等必要信息，不暴露举报人、后台备注和具体处罚细节。
- [Risk] 历史举报数据缺少结构化原因。→ Mitigation: 迁移时将旧 `reason` 映射到 `reasonCode = other`，原文本保留为 `reasonText`。

## Migration Plan

1. 扩展共享契约，新增举报原因、状态、请求/响应和后台详情类型。
2. 扩展 Prisma 模型字段，新增 `targetKey`、`reasonCode`、`reasonText`、目标快照、优先级、处理动作和风险计数字段；历史数据回填 `targetKey` 与 `reasonCode`。
3. 保留旧 `reason` 请求兼容期，小程序切换到新举报面板后再收紧验证。
4. 后台先在统一审核队列展示举报案件投影，再补治理详情中的举报明细和历史处理。
5. 验证通过后再考虑是否移除旧固定理由调用路径。

Rollback: 新字段以可空或可回填方式上线，旧 `reason` 和现有处理接口保持兼容；若举报面板出现问题，可临时回退到旧按钮提交，但后台仍可处理新旧举报记录。

## Open Questions

- 举报每日频率限制第一版采用多少次更合适：例如每用户每天 10 次，还是按新用户/可信用户分层？
- 是否需要对“隐私泄露”类举报引入临时公开侧折叠状态，还是第一版只做后台高优先级？
- `kept` 与 `false_report` 的运营定义是否需要更细：普通不成立、证据不足、恶意滥用是否拆分？
