## Context

项目当前已具备本地敏感词库、低成本评论分流、AI/mock 审核结果、统一后台审核队列和社区图片资产能力。现状中，评论/回复可以基于本地规则快速通过，帖子通常保持人工审核主导，日报评论和大陆新闻评论仍更依赖 `AiContentModerationService` 的 mock/兜底分流。

微信小程序内容安全能力提供文本 `msgSecCheck` 和图片/音频 `mediaCheckAsync`。文本接口适合昵称、评论、论坛和社交日志场景；图片接口是异步审核，需要可被微信检测服务器下载的媒体 URL，并通过消息回调返回结果。接口调用必须在服务端完成，不得把微信密钥、`access_token`、完整 openid 或供应商原始响应暴露给小程序或后台普通字段。

这次变更同时调整用户体验：作者提交帖子、评论或回复后，可以立即在自己的视图看到内容，但在内容安全通过前其他用户不可见。界面应表达为“审核中/仅自己可见/通过后其他人可见”，不得把未公开内容误导成已对所有人公开。

## Goals / Non-Goals

**Goals:**

- 建立统一内容安全分层：本地规则前置、微信内容安全二次判断、人工复核兜底。
- 覆盖社区帖子、评论、回复、图片、日报评论、大陆新闻评论和公开资料字段。
- 支持微信文本审核同步结果和图片审核异步回调。
- 将微信结果归一化到现有 `approved`、`pending`、`rejected`、`hidden` 生命周期和后台审核队列。
- 提供作者自见的乐观发布体验，减少提交后等待感。
- 保留 mock、降级和验证路径，使本地开发和内测不依赖真实微信接口稳定性。

**Non-Goals:**

- 不接入第三方商业内容安全服务或额外付费模型。
- 不开放用户自定义头像上传。
- 不把所有历史内容重新送审。
- 不引入完整后台账号体系，后台仍使用当前 `x-admin-token` 边界。
- 不展示微信供应商原始响应、完整 openid、`sessionKey`、`access_token` 或接口密钥。

## Decisions

### 1. 新增供应商无关内容安全服务，微信作为第一版真实供应商

实现层新增统一内容安全服务，负责接收文本、图片和业务场景，先执行本地规则，再按配置调用微信内容安全。现有 `AiContentModerationService` 可被重命名、包裹或作为兼容适配层保留，但对外契约应逐步从“AI 审核”收敛为“内容安全审核”。

备选方案是直接在各业务服务里调用微信接口。该方案短期改动小，但会造成社区、日报、大陆新闻和资料审核规则分散，后续难以统一降级、mock、敏感字段过滤和后台摘要。

### 2. 本地规则只做前置门禁，不做唯一事实源

本地规则继续负责低成本命中、归一化、硬规则驳回和短评论快速通过。明确高危内容可以直接驳回；灰区、帖子、图片、新用户风险、治理限制和行为异常应进入微信内容安全或人工复核。

可信低风险短评论可以保留快速通过，以减少接口调用和等待；帖子和带图内容的影响面更大，至少应完成内容安全通过后再公开给其他用户。

### 3. 微信结果映射为三类业务决策

微信 `pass` 映射为 `approved` 或可公开条件之一；`risky` 映射为 `rejected`；`review`、接口失败、超时、无有效 openid、回调异常或响应格式异常映射为 `pending` 和人工复核原因。

供应商结果不直接泄露给普通用户。用户只看到可理解文案，例如“审核中，通过后其他隐者可见”或“内容未通过，请修改后再试”。

### 4. 图片审核采用异步组合状态

图片上传后先保持非公开。创建带图帖子时，文本审核和图片审核都必须完成可公开判断。任何图片 `risky` 时帖子不得公开；任一图片 `review`、超时或回调失败时帖子进入人工复核；所有文本与图片均通过后，帖子可以自动公开或从 `pending` 转为 `approved`。

如果帖子已经被人工驳回、隐藏或移除，迟到的图片回调不得把帖子重新公开。

### 5. 作者自见不是暗箱公开

提交者可在社区详情、我的帖子、我的评论或我的回复中立即看到待审核内容。API 必须通过当前登录用户判断作者身份，并为作者返回 `visibleToAuthorOnly` 或等价状态。其他用户的公开列表和详情仍只返回 `approved` 内容。

前端文案应避免“已公开”误导，采用轻量状态提示。这样既保留提交反馈，也降低用户信任和投诉风险。

### 6. 后台展示归一化摘要

后台统一审核队列、审核详情和社区治理详情展示本地审核摘要、微信审核来源、归一化风险标签、处理建议、`trace_id` 摘要或请求标识摘要、降级原因和人工复核原因。后台不得默认展示完整供应商原始响应，也不得展示完整 openid、`sessionKey`、微信密钥或 `access_token`。

### 7. 配置与 mock 可控

新增微信内容安全开关、mock 模式、mock 文本结果、mock 图片回调结果和供应商错误模拟。正式环境缺少微信配置时，系统不得假装已完成真实微信审核，应进入人工复核或使用明确标记的 mock 结果。

## Risks / Trade-offs

- 微信接口要求用户近两小时访问过小程序，后台离线补审可能缺少有效 openid。缓解：UGC 提交时立即审核；无法满足时进入人工复核。
- 图片审核异步回调可能延迟或失败。缓解：保存 `trace_id`、设置超时任务、迟到回调幂等处理。
- 自动公开帖子会降低人工审核负担，但可能带来误判风险。缓解：本地高危先拦截、微信 `review` 进人工、举报和治理可后置隐藏。
- 作者自见体验可能被误解为已公开。缓解：前端保留轻量“仅自己可见/通过后公开”提示，不使用“已公开”文案。
- 现有契约中仍有 `aiModerationResult` 字段。缓解：实现阶段可兼容旧字段，同时新增内容安全摘要；归档后再考虑单独 change 做命名清理。
- 后台若保存完整供应商响应会增加敏感字段风险。缓解：默认只保存归一化摘要和必要排查标识，完整响应仅在受控日志中脱敏保存或不保存。

## Migration Plan

1. 新增内容安全共享契约和服务层适配，不先移除旧 AI/mock 契约。
2. 为微信文本审核、图片审核、mock、降级和敏感字段过滤增加验证。
3. 将社区评论/回复、帖子文本、图片资产、日报评论、大陆新闻评论和资料字段逐步接入统一服务。
4. 增加作者自见查询和小程序状态展示。
5. 扩展后台队列、治理详情和工作台统计摘要。
6. 更新 `.env.example`、内测文档和总 smoke。
7. 若微信配置或回调出现问题，可通过开关回退到本地规则加人工复核，不影响已公开内容读取。

## Open Questions

- 低风险可信短评论是否也强制调用微信文本审核，还是继续本地快速通过，需要结合调用量和用户等待体验在实现时确认。
- 带图帖子在文字通过但图片仍在审核中时，是否在作者自己的社区首页列表展示，还是只在“我的帖子”和详情中展示，需要结合前端信息密度确定。
- 微信图片回调的消息接收服务器是否复用现有 API 域名，还是单独提供最小化回调路径，需要在部署配置时确认。
