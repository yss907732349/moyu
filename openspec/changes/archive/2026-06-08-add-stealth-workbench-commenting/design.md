## Context

`add-stealth-workbench-mode` 已经把摸鱼模式定义为独立的办公表格伪装页面组，并将论坛详情收敛为阅读、标记、归档和附件查看。当前用户反馈认为评论功能仍应进入第一版，但评论 UI 不能照搬普通社区详情页的复杂社区化表达。

现有工程已经具备可复用能力：

- 小程序社区服务已有 `createCommunityComment(postId, body)` 和 `getCommunityPost(postId)`。
- 后端社区评论已接入身份门槛、低成本内容审核、微信内容安全或人工复核、作者自见、频控和用户治理状态。
- `comment-review-automation` 和 `community-lite` 已定义低风险评论自动通过、灰区待复核、明确违规驳回和普通公开响应不暴露审核细节。

本 change 只改变摸鱼模式前端体验和验证边界，不新增后端评论能力。

## Goals / Non-Goals

**Goals:**

- 在摸鱼模式帖子详情提供 `发表评论` 输入和提交能力。
- 复用现有社区评论创建接口和完整审核流程。
- 评论提交成功后刷新表格评论列表，使自动通过或作者自见内容按现有接口返回展示。
- 保持摸鱼模式的办公表格视觉：无头像、无评论卡片、无楼中楼复杂操作、无 RPG 视觉。
- 在摸鱼模式中隐藏审核状态、审核原因、`仅自己可见` 等状态文案。
- 更新验证脚本，允许摸鱼模式引用评论创建能力，但继续禁止回复、举报、发帖、图片上传、消息中心和个人社区中心入口。

**Non-Goals:**

- 不新增后端 API、数据库表、审核模型或评论状态机。
- 不改变普通社区详情页的评论、回复、举报或状态展示。
- 不在摸鱼模式第一版开放回复评论、举报评论、删除评论、图片评论、评论管理或消息入口。
- 不绕过内容安全、审核、频控、身份门槛或用户治理规则。
- 不把待审核评论表达为已经公开给所有人。

## Decisions

### 1. 输入文案正常使用 `发表评论`

摸鱼模式评论输入区使用用户熟悉的文案 `发表评论`，不改成 `新增记录`、`补充记录` 等过度伪装词。

理由：用户已经明确希望文案正常；同时摸鱼模式整体表格视觉已经足够低暴露，不需要让核心操作文案变得难懂。

备选方案是继续使用办公伪装语义，例如 `新增记录`。该方案更隐蔽，但会降低用户对功能的理解，且容易和帖子详情中的普通记录行混淆。

### 2. 前端只做轻量输入，不显示审核状态

摸鱼模式帖子详情新增一块表格化评论输入区：

- 输入框占用合并单元格或大单元格。
- 操作文本为 `发表评论`。
- 提交中使用 `同步中` 或禁用重复提交。
- 提交成功使用轻量反馈并刷新详情。

评论响应中的 `status`、`message` 或 `reviewDecision` 可用于内部流程判断，但摸鱼模式 UI 不展示 `审核中`、`仅自己可见`、`自动通过`、`人工复核` 或审核原因等状态文案。

理由：底层流程必须正常走，但摸鱼模式要避免出现明显社区运营和审核语义。用户看到评论是否出现在列表，由后端公开/作者自见规则和刷新后的详情结果决定。

### 3. 只开放一级评论，不开放回复

第一版摸鱼模式只允许对帖子发表评论，不允许对评论回复。

理由：回复会带来“回复某人”、多层层级、引用对象、回复输入框和更多状态提示，容易让界面接近普通社区详情页。一级评论足以补齐第一版社区参与闭环。

### 4. 验证边界从“禁止评论”调整为“允许评论，仍禁止回复和治理”

现有摸鱼模式验证会禁止引用 `createCommunityComment`。本 change 需要将该规则调整为允许帖子详情引用 `createCommunityComment`，同时继续禁止：

- `createCommunityPost`
- `createCommunityReply`
- `reportCommunityPost`
- `reportCommunityComment`
- `reportCommunityReply`
- `uploadCommunityMediaAsset`
- 消息中心、我的帖子、我的评论、我的回复、收藏管理等入口

理由：评论现在属于明确开放能力；回复、举报和管理入口仍不属于摸鱼模式第一版。

## Risks / Trade-offs

- [风险] 不展示审核状态可能让用户误以为评论已经公开给所有人。 → [缓解] 提交反馈使用中性短文案，不强调公开成功；评论是否出现在列表以刷新后的详情响应为准。
- [风险] 评论输入区破坏办公表格感。 → [缓解] 使用表格标题行、合并单元格输入和文字下划线操作，不使用卡片、头像、气泡或底部固定评论栏。
- [风险] 用户期待回复能力。 → [缓解] 第一版只补齐帖子评论闭环，回复保留给后续独立 change。
- [风险] 验证脚本过度放宽导致举报或回复入口误入。 → [缓解] 只对白名单 `createCommunityComment` 放行，并继续断言禁用能力不存在。

## Migration Plan

1. 在摸鱼模式帖子详情引入现有 `createCommunityComment`。
2. 增加评论输入状态、提交中状态和提交处理函数。
3. 在帖子详情评论表格附近加入表格化评论输入区，操作文案为 `发表评论`。
4. 提交成功后清空输入并刷新帖子详情，不展示审核状态文案。
5. 更新摸鱼模式验证脚本，允许评论创建但禁止回复、举报、发帖和管理入口。
6. 运行摸鱼模式验证、社区小程序验证、类型检查和小程序构建。

回滚时可隐藏或删除摸鱼模式评论输入区，并恢复验证脚本对 `createCommunityComment` 的禁止项；后端无需回滚。
