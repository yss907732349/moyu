## Why

`摸鱼侠` 需要一个稳定的功能入口契约，把已开放、未解锁和敬请期待的模块统一表达，避免首页、我的页、社区等页面各自硬编码功能状态。

现在先定义 `feature-registry`，可以在不提前实现赛季排行、小游戏、商店等未来模块的前提下，保留 RPG 成长反馈和后续扩展入口。

## What Changes

- 新增功能入口注册表的领域契约，定义功能条目的稳定字段、状态、展示位置和排序规则。
- 新增已开放、未解锁、敬请期待、隐藏/下线等状态的用户侧行为要求。
- 新增功能入口点击时的路由、禁用、解锁条件提示和敬请期待提示规则。
- 新增前端获取功能入口配置时的基础 API 响应约束，确保不同页面复用同一配置来源。
- 新增后台管理功能入口配置时的安全边界，避免敏感配置或未审核功能被公开暴露。
- 不实现具体未来业务模块，例如完整赛季排行、复杂小游戏、称号商店、皮肤商店或高级社交关系。

## Capabilities

### New Capabilities

- `feature-registry`: 定义功能入口注册表、状态机、展示规则、点击行为、配置来源和管理边界。

### Modified Capabilities

无。

## Impact

- 影响小程序端的功能入口渲染、导航拦截、禁用态和提示文案。
- 影响后端或本地配置层的功能入口数据模型、查询接口和缓存策略。
- 影响后台运营对功能入口标题、图标、排序、状态和解锁条件的配置能力。
- 为后续首页、个人页、社区、漫画、补给、活动等模块预留稳定 `featureKey`。
