## 新增需求

### 需求：功能入口结构

系统必须为每个功能入口定义稳定的 `featureKey`、展示标题、图标引用、状态、展示区域、展示排序和可选描述。

#### 场景：校验功能 key 格式

- **当** 创建或加载功能入口时
- **那么** 系统校验 `featureKey` 使用 snake_case，并且在标题、图标、路由和排序变化时保持稳定

#### 场景：引用共享契约

- **当** 应用需要功能入口 key、状态、展示区域或入口类型时
- **那么** 应用可以从项目共享包导入共享契约，而不是在本地重新定义这些值

#### 场景：渲染功能入口

- **当** 小程序渲染已配置的功能入口时
- **那么** 小程序使用 `feature-registry` 中的 `featureKey`、标题、图标引用、状态、展示区域和展示排序

#### 场景：功能元数据变化

- **当** 管理员或版本发布调整功能标题、图标、路由或展示排序时
- **那么** `featureKey` 在导航、统计和未来配置中保持不变

### 需求：功能状态契约

系统必须支持 `enabled`、`locked`、`coming_soon`、`disabled` 和 `hidden` 作为功能入口状态。

#### 场景：点击已开放功能

- **当** 用户点击带有可用公开路由的 `enabled` 功能时
- **那么** 小程序导航到已配置路由

#### 场景：点击未解锁功能

- **当** 用户点击 `locked` 功能时
- **那么** 小程序阻止导航，并展示已配置的解锁要求

#### 场景：未解锁功能没有结构化条件

- **当** `locked` 功能只有 `unlockText`，没有结构化解锁条件时
- **那么** 小程序仍展示解锁文案，并且不要求自动计算解锁状态

#### 场景：点击敬请期待功能

- **当** 用户点击 `coming_soon` 功能时
- **那么** 小程序阻止导航，并展示已配置的敬请期待文案

#### 场景：点击停用功能

- **当** 用户点击 `disabled` 功能时
- **那么** 小程序阻止导航，并展示已配置的不可用原因

#### 场景：配置中存在隐藏功能

- **当** 普通用户请求功能入口时
- **那么** 系统不返回 `hidden` 功能入口

### 需求：基于展示区域获取功能入口

系统必须允许客户端按展示区域请求功能入口，并且只接收符合该展示区域的入口。

#### 场景：使用本地默认注册表

- **当** 第一版没有远端功能注册表服务时
- **那么** 小程序可以通过相同的基于展示区域的契约，从本地默认注册表获取入口

#### 场景：请求首页快捷入口

- **当** 小程序请求 `home_quick_entry` 入口时
- **那么** 系统返回配置在 `home_quick_entry` 的入口，并按展示排序排列

#### 场景：功能出现在多个展示区域

- **当** 一个功能配置在多个展示区域时
- **那么** 系统可以在每个已配置展示区域返回相同且稳定的 `featureKey`，并使用对应区域的排序

### 需求：公开配置响应

系统必须只向小程序暴露公开的功能配置字段。

#### 场景：客户端请求功能注册表

- **当** 小程序请求功能注册表数据时
- **那么** 响应包含公开展示元数据、状态、已开放入口路由、解锁文案、敬请期待文案和不可用文案等适用字段

#### 场景：存在敏感配置

- **当** 功能注册表数据包含内部备注、灰度控制、审核字段或未发布路由信息时
- **那么** 系统从普通用户响应中排除这些字段

### 需求：路由安全

系统必须只允许带有已配置公开路由的可见 `enabled` 入口进行导航。

#### 场景：已开放功能没有公开路由

- **当** 用户点击没有公开路由的 `enabled` 功能时
- **那么** 小程序阻止导航，并展示不可用提示

#### 场景：非开放功能内部配置了路由

- **当** 用户点击状态不是 `enabled` 的功能时
- **那么** 即使内部配置存在路由，小程序也不会导航到该路由

### 需求：后台配置边界

系统必须允许授权管理员配置功能入口展示元数据、状态、展示区域、排序和用户可见提示文案。

#### 场景：管理员更新展示元数据

- **当** 授权管理员更新功能标题、图标、状态、展示区域、排序或提示文案时
- **那么** 系统在配置对普通用户可见之前完成校验

#### 场景：未授权用户尝试配置

- **当** 没有功能配置权限的用户尝试创建、更新、发布或隐藏功能入口时
- **那么** 系统拒绝该操作

### 需求：默认 MVP 功能注册表

系统必须为第一版产品界面提供默认功能入口，包括已开放的 MVP 功能和未来功能的可见占位入口。

#### 场景：初始化默认注册表

- **当** 应用初始化第一版功能注册表数据时
- **那么** 注册表包含已摸金额、薪资和上班时间设置、隐藏模式、轻量记账入口、个人资料、阵营选择、每日签到、隐者日报、轻社区、漫画/IP 内容，以及未来锁定或敬请期待入口

#### 场景：未来功能尚未实现

- **当** 注册表在实现前包含赛季排行、复杂小游戏、称号商店或皮肤商店等未来功能时
- **那么** 该入口配置为 `locked` 或 `coming_soon`，并且不暴露可用业务能力
