## Context

社区当前已经具备微信登录、隐者档案、发帖、评论、回复、内容安全、举报、后台治理和作者治理状态。现有身份门槛主要是“已登录且已创建隐者档案”，但发布型论坛继续上线和后续扩展关注/粉丝/个人主页时，还需要补齐真实身份认证路径、隐私同意留痕和 IP 属地展示。

法规和平台审核关注的是“后台实名、前台自愿”“信息发布/跟帖评论服务的真实身份认证”“账号信息页面合理范围展示 IP 属地”“个人信息处理规则告知和同意”。对本项目而言，第一版应使用微信手机号验证作为较轻的真实身份认证路径，不采集身份证号、真实姓名或人脸核身；IP 属地应服务端计算并公开降精度展示，不能依赖小程序前端上报。

本 change 是社区合规基础层，不直接实现粉丝关注功能，但会为后续个人主页提供最近公开发布 IP 属地字段和展示口径。

## Goals / Non-Goals

**Goals:**

- 发帖、评论和回复前建立“隐私政策/用户协议同意 -> 手机号验证 -> 社区写操作”的统一准入流程。
- 手机号验证只作为发布门槛和后台合规状态，不沉淀为公开身份标签。
- 服务端在社区写操作时提取可信客户端 IP，生成发布时 IP 属地快照，并在公开内容和个人主页中按降精度规则展示。
- 后台治理可查看必要合规摘要，包括隐私同意版本、手机号验证状态、IP 属地、解析失败原因和内容发布上下文。
- 明确提审包和生产环境的 IP 属地展示策略，避免合规展示被临时开关隐藏。
- 更新文档、配置样例、隐私政策、用户协议和验证脚本，让内测与审核检查有稳定依据。

**Non-Goals:**

- 不采集身份证号、真实姓名、身份证照片或人脸核身结果。
- 不展示完整手机号、手机号尾号、完整 IP、市/区县/街道级属地或设备指纹。
- 不在公开侧展示“已手机号验证”徽章或认证状态。
- 不做用户可关闭 IP 属地展示的设置。
- 不引入完整后台账号系统、RBAC、申诉中心或实名核身供应商切换平台。
- 不把历史无 IP 属地内容强行补造为真实属地。

## Decisions

### 1. 手机号验证前必须先完成隐私同意

社区写入口统一检查当前用户是否同意当前隐私政策和社区用户协议版本。若未同意，小程序先展示合规说明面板，包含隐私政策、用户协议和微信小程序用户隐私保护指引入口；用户主动同意后，才允许触发微信手机号验证组件。

服务端记录同意版本、同意时间、同意场景和用户 ID。客户端不得用本地标记替代服务端同意记录；本地只可用于优化首屏提示。

Alternatives considered:

- 先拉起手机号验证再补充隐私说明：流程短，但不满足先告知后同意的边界。
- 只依赖微信隐私弹窗不维护业务同意版本：接入简单，但无法表达社区协议、IP 属地展示和后台治理用途的版本差异。

### 2. 手机号验证是发布门槛，不是公开身份资产

手机号验证成功后，服务端保存用户已验证状态、验证时间、验证来源和必要的脱敏/哈希材料。普通公开响应、作者快照、帖子、评论、回复、个人主页、通知和举报详情均不得展示手机号、手机号尾号或“已验证手机号”状态。

后台常规治理页面默认只展示 `phoneVerified` 布尔状态；若后续排查确有必要展示尾号，也必须另行定义权限和审计，本 change 不开放。

Alternatives considered:

- 公开展示“已验证”徽章：能提高信任感，但会把合规门槛包装成社交身份，且用户已明确不希望展示。
- 后台展示手机号尾号：排查便利，但第一版治理不依赖手机号尾号，先保持更小暴露面。

### 3. IP 属地按内容发布时快照保存

发帖、评论和回复创建时，API 从可信请求来源提取客户端 IP，解析为 `ipLocationLabel`、`ipLocationCountryOrRegion`、`ipLocationProvince`、`ipLocationSource` 和 `ipLocationResolvedAt` 等内部字段。公开响应只返回 `ipLocationLabel`，例如 `广东`、`北京`、`日本` 或 `未知`。

历史内容不因用户后来位置变化而更新；个人主页展示最近一次公开发布行为的 IP 属地摘要。若用户删除或内容被移除，是否继续作为“最近公开发布 IP 属地”需要以仍可公开内容为准。

Alternatives considered:

- 个人主页只展示当前账号最近请求 IP：容易随登录、浏览和后台同步变化，用户解释成本高。
- 历史内容动态读取用户最新 IP 属地：会破坏取证稳定性，也可能让旧内容属地跳变。

### 4. IP 属地公开展示采用“个人页 + 内容详情”双层

个人主页 SHALL 展示用户最近公开发布 IP 属地。帖子详情、评论和回复 SHALL 保留轻量展示位，提审包和生产环境默认展示。帖子列表默认不展示 IP 属地，以保持列表密度。

展示建议放在昵称、等级、发布时间同一信息行或相邻弱化文本中，例如 `2小时前 · IP属地：广东`。该信息不做醒目徽章，不进入社交身份标签，也不允许用户关闭。

Alternatives considered:

- 只在个人主页显示：符合法规文字中“账号信息页面”的较低解释，但论坛详情和评论审核可见性不足，提审风险偏高。
- 所有列表都显示：合规最直观，但会降低社区列表密度和阅读效率。
- 先隐藏，过审失败再打开：实现灵活，但会造成二次提审和合规口径不稳定，因此仅允许内测视觉调试关闭，提审/生产不得关闭。

### 5. 服务端只信任受控代理链路中的 IP 来源

API 不接受客户端提交 IP 或属地。部署在网关、CDN 或反向代理后时，服务端只信任显式配置的代理 header 和来源网段；未配置可信代理时优先使用请求 socket IP。`X-Forwarded-For` 等 header 只能在请求来自可信代理时读取第一段有效公网 IP。

IP 解析失败不应伪造属地。发帖、评论和回复可保存 `未知` 并记录解析失败原因；验证脚本需要覆盖失败路径和公开响应敏感字段过滤。

Alternatives considered:

- 由小程序端上传 IP 属地：实现简单但不可信。
- 任意读取 `X-Forwarded-For`：适配方便但可被伪造。

### 6. 身份门槛错误需要可被客户端区分

社区写接口需要区分未登录、未建档、未同意当前协议、未完成手机号验证、手机号验证接口失败、社区治理禁言/封禁等错误。小程序据此展示不同引导：去登录、去建档、同意隐私政策、完成手机号验证或查看禁言提示。

客户端不得在服务端拒绝写操作后本地伪造提交成功，也不得把未验证状态下的内容保存为待审核内容。

### 7. 文档和隐私政策是验收项

隐私政策和用户协议必须明确说明手机号验证、IP 属地展示、内容安全审核、举报治理、后台处理和必要保存期限。内测检查清单需要覆盖：未同意不得拉起手机号验证、未验证不得发布、公开侧无手机号验证状态、公开内容有 IP 属地、后台无完整手机号和明文 IP。

## Risks / Trade-offs

- [Risk] 手机号验证增加发布阻力。→ Mitigation: 只在发帖、评论、回复前触发；浏览、点赞、收藏和举报保持较低门槛。
- [Risk] IP 属地展示影响社区视觉。→ Mitigation: 列表不展示，详情/评论/回复使用弱化信息行，避免徽章化和强装饰。
- [Risk] 可信 IP 提取在代理配置错误时不准确。→ Mitigation: 增加可信代理配置、开发环境 fallback、解析失败提示和验证脚本。
- [Risk] 微信手机号接口失败导致用户无法发布。→ Mitigation: 给出明确重试提示；不绕过发布门槛，避免合规缺口。
- [Risk] 历史内容缺少 IP 属地。→ Mitigation: 历史内容不伪造属地；仅新写操作强制生成快照，历史内容可显示未知或不显示该段。
- [Risk] 后台排查想看更多身份信息。→ Mitigation: 第一版只给合规状态和属地摘要；完整手机号、明文 IP、openid 等继续禁止出现在常规后台响应。

## Migration Plan

1. 扩展共享契约，新增隐私同意版本、手机号验证状态、社区身份门槛错误码和 IP 属地公开/后台摘要类型。
2. 扩展数据模型，保存用户隐私同意记录、手机号验证状态、社区内容发布 IP 属地快照和解析失败摘要。
3. 新增或扩展服务端手机号验证接口，使用微信手机号动态 `code` 换取手机号并保存验证状态。
4. 在社区发帖、评论和回复写入口加入同意/手机号/治理状态统一门槛，并保存发布时 IP 属地快照。
5. 小程序发帖、评论、回复入口接入隐私同意面板、手机号验证按钮和身份门槛错误提示。
6. 小程序帖子详情、评论、回复和个人主页接入 IP 属地展示位；提审/生产默认显示，列表默认不展示。
7. 后台治理详情和审核队列补充合规摘要，验证敏感字段黑名单。
8. 更新隐私政策、用户协议、内测检查清单、环境变量样例和验证脚本。

Rollback: 新字段应以可空或默认未验证方式上线；若手机号验证服务异常，可临时禁止新发布并展示维护提示，不能绕过门槛发布。IP 属地解析失败可降级为 `未知`，不影响内容审核链路。

## Open Questions

- IP 属地库第一版使用本地库、商业服务还是部署环境自带能力？需要在实现时结合包体、离线能力和费用决定。
- 隐私政策版本号是否使用独立配置文件，还是复用后台运营配置？建议第一版放在共享配置并由验证脚本检查。
- 历史已公开内容无 IP 属地时，公开详情是显示 `IP属地：未知` 还是隐藏该字段？建议实现时以视觉密度和审核口径再定，但不得伪造属地。
