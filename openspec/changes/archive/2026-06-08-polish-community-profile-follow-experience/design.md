## Context

`add-community-follow-profile-system` 已经把公开个人页、关注/取消关注、关注列表/粉丝列表和“我的”页三项数据打通。本 change 不重新定义社交关系模型，而是在既有能力上补交互细节、视觉层级和安全抽查。

当前用户路径可以拆成四段：

```text
帖子/评论/回复作者入口
        │
        ▼
公开个人页（隐者名片）
        │
        ├── 关注 / 取消关注
        │
        ├── 关注列表 / 粉丝列表
        │
        └── 公开帖子列表

我的页
        │
        ├── 关注数 -> 本人关注列表
        ├── 粉丝数 -> 本人粉丝列表
        └── 连续签到 -> 签到成长反馈
```

这次 polish 的判断标准不是“多放信息”，而是“用户一眼知道可点、点了有反馈、状态空时不假装热闹、敏感信息继续克制”。

## Goals / Non-Goals

**Goals:**

- 让作者头像/昵称入口有明确可点击感，同时不影响评论阅读、举报和回复操作。
- 让公开个人页首屏像“隐者名片”，强化身份感、阵营感和社交关系，不像后台资料表。
- 让关注/取消关注更舒服：关注轻反馈，取消有防误触，失败可恢复。
- 让关注列表/粉丝列表在加载、空、失败和分页时都不显得粗糙。
- 让“我的”页三项数据在小屏和长数字下仍稳定。
- 弱化 IP 属地，不让合规信息压过身份主信息。
- 扩展敏感字段黑名单和抽查范围，覆盖 API 响应、列表项、操作响应和小程序缓存/渲染。

**Non-Goals:**

- 不把关注关系升级为内容分发、通知或好友关系。
- 不新增更多公开资料字段。
- 不为公开个人页增加编辑表单、资料管理或头像上传。
- 不通过推荐用户、假用户或运营占位填充空关注/粉丝列表。

## Decisions

### 1. 作者入口用“轻交互暗示”，不做重按钮化

帖子详情、评论和回复里的作者头像/昵称应显示为可进入主页的身份入口，但不能把评论区变成按钮墙。

建议口径：

- 可点击作者：头像增加轻微描边/发光或角标，昵称使用更明确的高亮色，点击区域覆盖头像和昵称主体。
- 按压态：短暂降低透明度或背景浮起，证明已接收点击。
- 不可点击作者：只展示作者快照，不显示入口暗示，不出现假箭头或假主页。
- 入口失败：统一提示“该隐者主页暂不可访问”，不暴露内部用户 ID、治理状态、手机号状态或审核原因。

### 2. 公开个人页首屏是名片，不是资料表

公开个人页首屏应形成一个“隐者名片”结构：

```text
┌─────────────────────────────┐
│  头像       昵称 / 阵营      │
│  徽章感信息  等级 / 称号      │
│  IP 属地（弱）        关注按钮│
├─────────┬─────────┬─────────┤
│  关注   │  粉丝   │  公开帖 │
└─────────┴─────────┴─────────┘
    公开帖子列表开头露出
```

名片优先级：

1. 头像、昵称、阵营身份。
2. 等级、称号、关注操作。
3. 关注数、粉丝数、公开帖数。
4. IP 属地。

IP 属地应保留但弱化：小字号、低对比、低装饰，不使用高亮色、徽章式强框、主按钮色或统计位。

### 3. 关注和取消关注采用不同反馈强度

关注是低风险正向动作，可以即时反馈；取消关注更容易误触，应增加二次确认或等价意图表达。

推荐策略：

- 关注：按钮进入处理中态，服务端成功后切换为已关注，统计同步更新，使用轻 toast 或页面内短反馈。
- 取消关注：点击已关注按钮时先出现确认弹窗、底部确认或“再次点击取消”的短窗口；确认后才发请求。
- 失败：按钮和统计恢复为请求前或重新拉取后的服务端事实状态，提示“操作暂未成功，请稍后再试”一类泛化文案。
- 快速连点：按钮处理中态锁定，服务端幂等仍作为最终保护。

### 4. 列表状态不造热闹

关注列表和粉丝列表的空状态应承认真实空，而不是塞推荐用户。

状态口径：

- 首次加载：使用骨架行或轻量加载态，保留列表结构预期。
- 空状态：一句短文案 + 可选返回/刷新，不出现假用户、推荐用户或长篇功能说明。
- 失败态：显示泛化失败原因和重试按钮，不暴露目标用户内部状态。
- 分页加载：底部按钮或触底加载应有“加载中/没有更多”状态，不重复追加同一页数据。

### 5. “我的”页三项数据像身份卡的一部分

“我的”页三项数据不是运营仪表盘，而是本人身份卡的一部分。关注和粉丝可点击，但连续签到保留成长反馈。

排版口径：

- 三项保持稳定三列或等价紧凑布局。
- 关注和粉丝有清晰点击态，可通过轻箭头、按压态或 icon 暗示入口。
- 连续签到不伪装成链接，不与关注/粉丝混淆。
- 长数字使用压缩、分组缩写或省略策略，但不得影响真实数据含义；如使用 `9999+`，必须只作为展示格式，不改变服务端数值。

### 6. 小屏验收作为任务，不留给感觉

需要明确人工验收输入：

- 昵称：12 字、20 字、含连续英文/数字长串。
- 称号：常规称号、超长称号、含连续无空格文本。
- 数字：0、9、99、999、9999、999999、99999999。
- 屏宽：320px、375px、414px。

验收页面：

- 帖子详情作者区。
- 评论区和回复区作者入口。
- 公开个人页。
- 关注列表和粉丝列表。
- “我的”页顶部身份区域。

### 7. 敏感字段过滤要覆盖“新体验路径”

这次不只检查公开个人页 API，还要覆盖因 polish 新增或改动的路径：

- 作者入口响应。
- 公开个人页响应。
- 公开帖子列表项。
- 关注/取消关注响应。
- 关注列表/粉丝列表响应。
- “我的”页三项统计响应。
- 小程序本地缓存、页面绑定字段和调试日志。

黑名单继续覆盖：手机号、手机号尾号、手机号验证状态、真实姓名、完整 IP、市/区县/街道、IP 来源 header、IP 解析原始结果、微信 `openid`、微信 `unionid`、微信 `sessionKey`、应用登录态、薪资、工作时间、工作档案、隐藏模式、隐币、能量、生存账单、消费统计、CPS 来源、后台治理状态、内容安全内部风险标签、供应商原始响应和后台审核备注。

## Risks / Trade-offs

- [Risk] 取消关注确认增加一步，可能显得慢。→ Mitigation: 只对取消关注加确认，关注保持轻快。
- [Risk] 作者入口可点击感太强，干扰评论阅读。→ Mitigation: 使用头像/昵称局部暗示，不把整段评论正文变成入口。
- [Risk] 名片视觉过重导致公开帖子列表被压到首屏外。→ Mitigation: 首屏底部必须露出公开帖子列表开头。
- [Risk] 弱化 IP 属地被误解为隐藏合规信息。→ Mitigation: 展示位保留，只降低视觉优先级，不提供关闭或编辑。
- [Risk] 长数字缩写影响精确性。→ Mitigation: 缩写只用于视觉展示，API 和内部状态保持完整数值；必要时点击列表页面查看真实列表。
