## Context

`work-value-tracker` 正式规格已经定义工作档案、简单月薪估算、本地实时计算、未配置状态、隐私边界和隐藏模式。`build-work-profile-settings` 已经落地独立设置页、`/me/work-profile` API、持久化模型、小程序工作档案客户端服务，并在保存或读取成功后把 `WorkProfileSnapshot` 写入本地存储。

当前首页仍使用视觉演示逻辑：固定基准金额、`setInterval` 模拟自增、静态倒计时卡和未接入真实状态的隐藏图标。这个 change 需要把首页核心卡片切换为真实工作档案快照驱动，同时保持前端本地计算、敏感字段不公开和第一版不接入复杂日历的边界。

## Goals / Non-Goals

**Goals:**

- 首页基于已保存 `WorkProfileSnapshot` 调用共享 `calculateWorkValueState`，展示今日已摸金额、每秒收益、工作状态、工作进度和工作档案可推导倒计时。
- 首页在 `onShow` 刷新快照，支持用户从设置页保存后返回首页立即看到最新结果。
- 首页未配置工作档案时展示配置引导，不展示或继续增长演示金额。
- 首页隐藏模式遮挡显性敏感金额和每秒收益，并切换低调视觉状态。
- 首页在 API 不可用时优先使用本地已保存快照；没有本地快照时展示未配置或加载失败引导。
- 保留现有视觉系统结构、feature registry 功能入口、工作档案设置页和底部记账横幅定位。

**Non-Goals:**

- 不修改后端 `/me/work-profile` API、Prisma 工作档案模型或临时本人上下文机制。
- 不实现真实微信登录、openid 绑定、session 或用户资料体系。
- 不新增薪资模式、税务口径、奖金、十三薪、社保公积金或真实工资结算。
- 不接入法定节假日、调休、请假、大小周、多班次轮班或复杂排班。
- 不实现成长发奖、签到、记账模型、账单分类、账单存储或社区公开展示工作档案。

## Decisions

### Decision: 首页以本地快照为首屏输入，后台同步 API 结果

首页进入或重新显示时先读取本地 `moyuxia.workProfileSnapshot`，若存在合法快照则立即计算并渲染；随后可通过统一工作档案客户端服务读取 `/me/work-profile`，成功后更新本地快照和首页状态。

Rationale: 小程序首页需要快速展示，且设置页已经把保存后的快照写入本地存储。优先本地快照可以避免 API 延迟导致首页短暂回到未配置状态，同时仍允许后端数据在可用时校正本地缓存。

Alternatives considered:

- 每次首页只请求 API。数据来源更单一，但 API 不可用或网络慢时首页核心体验会退化。
- 只读本地缓存。实现简单，但跨设备或缓存陈旧时无法同步后端最新档案。

### Decision: 首页使用 `onShow` 驱动快照刷新

首页应在页面首次加载和从设置页返回时刷新工作档案快照。实现上优先使用小程序页面显示生命周期，而不是只依赖组件 `onMounted`。

Rationale: 设置页保存后返回首页是本 change 的关键路径。`onMounted` 不保证每次返回都触发，`onShow` 更符合小程序页面恢复场景。

Alternatives considered:

- 设置页保存后主动跳转并携带参数通知首页。路径耦合更强，也无法覆盖用户通过其他方式更新缓存的场景。
- 使用全局状态管理。当前项目还没有全局 store，单页消费快照不需要引入新架构。

### Decision: 真实计算只由共享函数提供

首页展示层只负责读取快照、定时传入 `now`、格式化展示结果和映射状态文案；金额、每秒收益、工作状态、休息段排除、工作进度和倒计时秒数由 `packages/shared` 中的 `calculateWorkValueState` 产生。

Rationale: 工作档案设置页、后端校验和首页计算必须共享同一套契约。首页重写算法会导致休息段、发薪日和自定义工作日口径漂移。

Alternatives considered:

- 在首页直接按字段计算金额和倒计时。短期改动少，但后续维护风险高。

### Decision: 计时器只刷新计算结果，不模拟收益

首页计时器每秒或接近每秒重新调用共享计算函数，金额变化来自当前时间和快照，不再使用固定基准金额和演示每秒收益自增。视觉层可以格式化为两位金额，但不得用假数据补足未配置状态。

Rationale: 真实计算的最小单位是业务时间，不是动画插值。第一版应优先保证金额口径正确和省电，后续若需要更丝滑跳字，可单独设计展示层动画。

Alternatives considered:

- 保留 100ms 演示自增。视觉更流畅，但会混淆真实金额和演示金额。

### Decision: 倒计时只展示工作档案可推导项

首页倒计时卡优先展示距上班、距下班、距休息日和距发薪日。法定节假日、调休或运营活动倒计时在外部日历或运营配置接入前不得显示为真实倒计时，可保留为“待接入”的非真实占位。

Rationale: 正式规格已经明确工作档案不是法定节假日数据源。继续展示静态端午倒计时会造成事实错误。

Alternatives considered:

- 暂时硬编码节假日日期。实现快，但会快速过期，并违反工作档案边界。

### Decision: 隐藏模式遮挡主金额和每秒收益

隐藏模式开启时，首页主金额和每秒收益 SHALL 被遮挡或替换为安全文案；工作状态、配置入口和非敏感倒计时可以保留。发薪日倒计时是否展示按第一版最低隐私边界保留，但不得展示月薪、时薪或可直接反推薪资的文案。

Rationale: 已摸金额和每秒收益是最显性的敏感金额信息。第一版不要求遮挡所有可推断薪资的信息，但每秒收益与薪资高度相关，应一并处理。

Alternatives considered:

- 只遮挡主金额。实现更小，但每秒收益仍会直接暴露薪资估算。
- 全量遮挡所有倒计时和工时信息。隐私更强，但会削弱首页工具价值，不符合第一版最低要求。

## Risks / Trade-offs

- [Risk] 本地快照陈旧导致首页短暂展示旧设置 -> Mitigation: 首页后台同步 `/me/work-profile` 成功后更新缓存和展示；设置页保存成功必须写入最新快照。
- [Risk] API 不可用时用户误以为未保存 -> Mitigation: 有本地快照时继续展示本地计算，并用轻量错误状态提示同步失败；无本地快照时再展示配置引导。
- [Risk] 隐藏模式仍可能通过发薪日或工作时段间接推断信息 -> Mitigation: 第一版明确只遮挡显性金额和每秒收益，后续如有更高隐私要求再扩展。
- [Risk] 计时器泄漏或页面隐藏后继续运行 -> Mitigation: 页面卸载或隐藏时清理计时器，页面显示时重新启动。
- [Risk] 倒计时卡在无对应秒数字段时显示异常 -> Mitigation: 展示层统一处理 `undefined`，用“待配置”或“待接入”而不是拼接空值。

## Migration Plan

1. 在首页读取本地快照并接入统一工作档案客户端服务进行后台同步。
2. 将首页金额、每秒收益、工作状态和工作进度改为由 `calculateWorkValueState` 输出驱动。
3. 将静态倒计时数组替换为工作价值状态中的倒计时映射。
4. 将未配置、加载失败、隐藏模式和已配置状态分别渲染。
5. 增加或更新验证脚本，覆盖首页快照消费、保存后刷新、未配置状态、隐藏模式和倒计时映射。

Rollback strategy: 如首页真实计算出现阻断问题，可以临时只展示未配置引导和设置入口，保留设置页、API 和共享契约不变；不得恢复会被误认为真实数据的演示金额。
