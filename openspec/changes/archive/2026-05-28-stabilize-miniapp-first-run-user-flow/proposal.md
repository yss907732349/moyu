## Why

小程序首跑链路已经具备登录、用户成长资料、工作档案、首页已摸金额、日报、社区和生存账本等能力，但首次用户体验中仍存在静态演示数据、入口状态不准确和服务 API 地址分散的问题。

本次变更将小程序首跑流程稳定为基于真实状态推导的链路：未登录、未建档、未配置工作档案和已完成状态都应有明确入口和可恢复展示；首页、个人页和各内容模块不得用演示数据伪装真实用户状态。

## What Changes

- 统一小程序端 API base URL 配置，登录、用户资料、工作档案、功能入口、日报、社区和生存账本服务使用同一配置来源。
- 新增首跑状态推导，基于应用登录态、用户成长资料状态和工作档案状态串联登录、建档、配置工作档案和回首页查看已摸金额。
- 首页身份区改为读取真实用户成长资料或合法本地快照，不再展示静态“键影隐者 · 一阶”。
- 工作档案未配置时首页展示配置引导，不再展示演示金额、演示进度或假工作状态。
- 建档完成后在个人页提供配置工作档案的下一步入口；用户已完成工作档案配置后，该入口不再作为待办提示展示。
- 日报、社区和生存账本补齐首跑空状态、错误态和开发演示种子边界，生产公开侧不得自动伪造内容或用户消费。
- 保留个人页“我的成就”等既有区域，不因首跑改造删除或重命名。

## Capabilities

### New Capabilities

- `miniapp-first-run-user-flow`: 定义小程序首跑状态推导、统一 API 配置、链路串联和降级边界。

### Modified Capabilities

- `user-growth-profile`: 首页身份读取真实成长资料，建档完成后引导配置工作档案。
- `work-value-tracker`: 首页未配置工作档案引导，保存后首页即时计算。
- `daily-content-feed`: 日报首跑空状态和开发演示种子策略。
- `community-lite`: 社区首跑空状态和开发演示种子边界。
- `accounting-ledger`: 生存账本首跑空状态和不使用用户消费种子。

## Impact

- 小程序端：`apps/miniapp` 首页、个人页、工作档案设置页、日报、社区、生存账本和统一服务配置。
- 共享契约：首跑状态推导、用户成长资料快照、工作档案快照和各模块空状态验证。
- 验证：新增或更新小程序首跑链路、API 配置、工作档案、用户成长资料、日报、社区和生存账本相关验证。
