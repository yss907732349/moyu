## Context

`accounting-ledger` 是 MVP 的独立业务域，需要从首页辅助工具横幅进入正式记账流程，并为用户提供基础收支记录和统计。记账数据属于本人财务习惯数据，敏感性接近工作档案，不应被社区、排行榜、公开资料或功能入口公开暴露。

当前商业模式计划接入美团 CPS、饿了么 CPS、京东外卖 CPS、滴滴打车 CPS 和会员充值 CPS。CPS 订单能帮助用户降低记账成本，但订单状态、金额口径、退款取消和归因链路都不稳定，因此第一版账本必须先建立独立事实源，再为 CPS 来源预留扩展字段。

## Goals / Non-Goals

**Goals:**

- 定义第一版账单模型、分类、统计口径、本人 API、持久化和小程序入口行为。
- 支持用户手动记录收入和支出，并展示当日/本月统计。
- 让首页记账横幅进入正式记账流程，而不是停留在本地演示计数。
- 在账单模型中预留来源字段，允许未来 CPS 订单生成待确认账单草稿。
- 明确隐私边界，确保账本数据默认只对本人可见。

**Non-Goals:**

- 第一版不实现 CPS 订单同步、回调接收、佣金结算、订单归因或自动入账。
- 第一版不实现预算、资产账户、多币种、报表导出、多人共享、发票管理或复杂财务分析。
- 第一版不将记账数据接入社区、排行榜、公开资料或运营公开展示。
- 第一版不处理真实支付渠道对账，也不把记账金额表达为银行流水或支付凭证。

## Decisions

### 账本以用户确认记录为事实源

正式账本只统计用户手动创建或用户确认后的账单。未来 CPS 订单不得直接写入正式账本，而是生成待确认草稿或入口提示，用户确认后才可转为正式账单。

替代方案是 CPS 订单自动入账。该方案短期看减少输入成本，但外部订单存在取消、退款、部分退款、重复回调、金额口径不一致和隐私信任风险，不适合作为第一版默认行为。

### 账单状态与来源解耦

账单状态用于表达账本内的业务状态，例如 `posted`、`draft`、`voided`。账单来源用于表达记录来自手动输入还是未来的 CPS 订单，例如 `manual`、`cps_order`。来源字段可以包含 `sourceProvider`、`sourceOrderId` 和 `sourceStatus` 等可选信息，但这些字段不改变正式统计口径。

替代方案是为每个 CPS 平台建立不同账单类型。该方案会让账本模型过早受外部平台细节影响，也会增加小程序端展示和统计分支。

### 第一版分类采用内置稳定分类

第一版提供内置收入/支出分类，使用稳定 `categoryKey`。分类展示名、图标和排序可在共享契约中定义，小程序端和 API 复用同一套分类事实源。第一版不提供用户自定义分类。

替代方案是立即支持自定义分类。该方案会引入分类归属、删除迁移、统计历史口径和管理复杂度，超出第一版范围。

### 金额使用最小货币单位存储与计算

账单金额在共享契约和 API 中使用最小货币单位表示，例如 `CNY` 的分，避免浮点误差。第一版默认币种为 `CNY`，不支持多币种统计。

替代方案是用小数金额贯穿前后端。该方案展示简单，但统计和持久化容易出现精度问题。

### 本人接口复用应用登录态和本人上下文

账本 API 使用 `/me/accounting-ledger` 语义并复用现有本人上下文：优先使用应用登录态，开发环境仅在明确允许时使用 `temporary-dev-placeholder` fallback。接口不得让客户端提交任意用户 ID。

替代方案是在账本接口中自行解析用户标识。该方案会复制认证逻辑，并增加越权风险。

### 首页入口只负责进入账本流程

首页底部 tab 上方横向辅助工具横幅作为记账入口，可以展示今日/本月摘要或待确认草稿提示，但不在首页内维护独立账单事实。账单创建、列表和统计通过统一账本客户端服务获取。

替代方案是首页继续维护本地计数或演示数据。该方案会让首页展示与正式账本脱节。

## Risks / Trade-offs

- CPS 预留字段过多导致第一版模型臃肿 → 仅预留通用来源字段，不引入平台专属字段。
- 用户手动记账和未来 CPS 草稿可能重复 → 未来实现 CPS 草稿时需要基于来源订单 ID、金额、日期和分类做重复提示，第一版仅定义不得直接自动入账。
- 分类过少影响实用性 → 第一版覆盖餐饮、交通、购物、娱乐、居住、通讯、医疗、学习、工资、副业、其他等基础分类，后续再扩展自定义分类。
- 统计口径被草稿或取消账单污染 → 当日/本月统计只纳入 `posted` 且用户确认的账单。
- 财务数据隐私泄露 → 所有账本接口必须限定本人上下文，公开接口默认不得返回账单、统计或分类偏好。

## Migration Plan

- 新增共享账本契约、分类、校验和统计函数。
- 新增 API 持久化模型和本人账本接口。
- 新增小程序账本客户端服务、记录页、列表/统计页，并改造首页横幅入口。
- 第一版上线时不需要迁移历史账单；若已有本地演示计数，不迁移为正式账单。
- 回滚时可以隐藏或禁用记账入口，但不得删除用户已经创建的账单数据。

## Open Questions

- 第一版是否需要“待确认草稿”数据结构只在契约中预留，还是在数据库中一并建表但不接 CPS？
- 首页横幅第一版展示当日支出、本月净额，还是只展示“记一笔”主操作和账本入口？
- 内置分类的第一版图标是否复用现有像素资产，还是先用 token 化图标 key 占位？
