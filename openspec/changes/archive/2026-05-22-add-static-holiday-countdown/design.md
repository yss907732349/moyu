## Context

`work-value-tracker` 已经要求首页倒计时不能把未接入的外部日历信息表达为真实倒计时。当前项目还没有节假日数据源，首页如果直接硬编码某个节日日期会很快过期；如果小程序端实时请求第三方免费接口，则会把首屏体验绑定到不稳定的外部服务。

中国大陆法定节假日的权威来源是国务院办公厅正式通知，但通常不是稳定 JSON API。第三方免费接口可以提供机器可读数据，但只能作为同步来源或辅助来源，不能替代权威核验。

## Goals / Non-Goals

**Goals:**

- 建立节假日缓存数据结构，支持小程序首页本地查询“下一个法定节假日”。
- 增加同步脚本，从候选免费接口拉取结构化节假日数据，并记录同步来源与时间。
- 增加验证脚本，检查日期格式、年份覆盖、区间顺序、来源元数据、过期风险和权威核验状态。
- 首页倒计时卡展示“距下一个法定节假日”，在无有效数据时展示非真实占位。
- 明确 2027 年等未来年份在权威安排未发布前不得被标记为已核验。

**Non-Goals:**

- 不让小程序端运行时直接请求第三方免费节假日接口。
- 不新增数据库模型、管理后台运营日历或服务端节假日 API。
- 不实现调休上班日、请假、大小周、多班次、复杂排班或个人实际休假计算。
- 不把节假日数据用于修正 `calculateWorkValueState` 的工作日判断。
- 不预测国务院尚未发布的未来年份安排。

## Decisions

### Decision: 采用“同步生成缓存数据”，而不是首页实时请求第三方接口

节假日数据由 `scripts/sync-holidays` 类脚本拉取候选免费接口，输出到项目内可版本化的缓存文件或共享包数据模块；首页只消费 `packages/shared` 导出的查询函数和已验证缓存。

Rationale: 免费接口可能有调用限制、网络抖动、字段变化或服务下线。把同步放在开发/发布流程中，可以通过验证脚本拦截错误，首页运行时不会因第三方服务不可用而退化。

Alternatives considered:

- 小程序端实时请求第三方接口。实现直接，但稳定性、隐私合规、域名配置和首屏体验风险更高。
- 完全手写静态数据。最简单，但维护成本高，且容易在跨年后过期。

### Decision: 权威来源元数据与机器来源分离

缓存数据每个年份 SHALL 包含机器同步来源、同步时间、覆盖年份和权威核验状态；权威核验元数据 SHALL 能记录国务院办公厅正式通知 URL、通知标题和核验日期。第三方接口数据不得默认视为权威已核验。

Rationale: 中国法定节假日最终以国务院办公厅通知为准。把“机器可读来源”和“权威核验状态”拆开，可以先同步候选数据，再由验证脚本和人工核验决定是否允许标记为正式可用。

Alternatives considered:

- 只记录第三方接口 URL。缺少权威追溯，后续日期争议时无法判断数据是否可信。
- 只记录权威网页并人工录入。可信度高，但无法满足“可更新”的自动化需求。

### Decision: 共享包提供查询函数，首页只做展示映射

`packages/shared` SHALL 定义节假日类型、缓存结构和 `getNextStatutoryHoliday` 一类查询函数。首页传入当前日期，获得下一个节假日名称、开始日期、结束日期、剩余天数和数据状态，再映射为倒计时卡。

Rationale: 日期选择逻辑需要被验证脚本和小程序复用。放在共享包可以避免页面内散落日期筛选和格式处理。

Alternatives considered:

- 在首页直接遍历缓存。页面实现更快，但测试和复用性差。

### Decision: 节假日倒计时不影响工作价值计算

本 change 只新增首页法定节假日倒计时展示。`calculateWorkValueState` 继续只使用工作档案推导工作状态、金额和工作档案相关倒计时，不因为法定节假日或调休数据改变工作日判断。

Rationale: 项目边界已经明确工作档案第一版不处理调休、请假和复杂排班。节假日倒计时是信息展示，不应悄悄改变收入估算口径。

Alternatives considered:

- 用节假日数据修正标准工作日。看似更准确，但会把调休、补班和个人排班问题一并引入，超出第一版范围。

### Decision: 验证脚本必须允许“未来年份未发布”的显式状态

验证脚本 SHALL 区分“已核验可展示”“已同步待核验”“未来年份待发布/待同步”。如果当前年或下一年权威安排尚未发布，脚本 SHALL 给出明确提示；不得要求用预测数据补齐并通过正式核验。

Rationale: 2027 年安排在 2026-05-22 尚未检索到国务院办公厅正式通知。为了满足“今年和明年”的规划目标，同时避免预测写入，需要让数据状态表达事实不完整，而不是假装完整。

Alternatives considered:

- 缺失未来年份时直接失败。能强制补齐，但会诱导录入预测数据。
- 允许任意缺失。实现宽松，但容易发布后首页无数据。

## Risks / Trade-offs

- [Risk] 免费接口字段变化导致同步脚本失败 -> Mitigation: 同步脚本只作为数据更新入口，失败时保留上一版缓存；验证脚本阻止坏数据进入可用状态。
- [Risk] 第三方接口数据与国务院通知不一致 -> Mitigation: 缓存中保留权威核验状态；未核验数据不得展示为已核验法定节假日。
- [Risk] 未来年份权威通知未发布时首页无法显示下一年假期 -> Mitigation: 首页展示“待发布/待同步”占位，不展示预测倒计时。
- [Risk] 用户误以为节假日倒计时会影响已摸金额 -> Mitigation: 页面文案避免表达为排班计算，规格明确不改变工作价值计算。
- [Risk] 同步脚本需要网络访问 -> Mitigation: 同步脚本作为显式命令运行；日常验证可基于已缓存数据离线执行。

## Migration Plan

1. 在 `packages/shared` 新增节假日缓存类型、数据入口和下一个节假日查询函数。
2. 在 `scripts/` 新增节假日同步脚本，用候选免费接口生成或更新缓存数据，并保留来源元数据。
3. 在 `scripts/` 新增验证脚本，覆盖日期、年份、来源、核验状态和过期风险。
4. 在根 `package.json` 增加 `sync:holidays` 和 `verify:holidays` 命令。
5. 首页倒计时卡接入共享查询函数，展示下一个法定节假日或明确占位。
6. 执行 `pnpm verify:holidays`、`pnpm verify:miniapp-work-profile` 和 `pnpm typecheck`。

Rollback strategy: 若节假日数据源或验证规则出现阻断，可保留共享类型和验证脚本，首页临时展示“节假日待同步”占位；不得恢复过期硬编码日期或预测日期。

## Open Questions

- 首个同步脚本默认接入哪个免费接口，需要在实现时根据接口稳定性、字段完整度和无需密钥优先级确定。
- 权威核验是否完全人工填写，还是后续增加对 `gov.cn` 页面标题和 URL 的半自动检查。
