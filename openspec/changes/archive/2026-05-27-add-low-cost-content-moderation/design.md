## Context

论坛评论当前在开发期使用同步本地规则模拟 AI 审核，规则覆盖不足，且未来直接接入付费供应商会带来成本、延迟和并发风险。项目前期需要一套低成本、可控、可逐步升级的审核方案：本地硬规则拦截明显风险，低风险内容快速通过，灰区进入人工复核，后续再按真实量级接入易盾等第三方服务。

用户指定的 `https://github.com/konsheng/Sensitive-lexicon` 是 MIT 许可的中文敏感词库，可作为本地词库来源。当前环境下载 GitHub 失败，因此实现应提供显式同步脚本，网络不可用时保留已有缓存，不能让构建或运行时依赖实时访问外部仓库。

## Goals / Non-Goals

**Goals:**

- 提供本地敏感词库同步、缓存、验证和运行时加载能力。
- 对用户生成文本进行归一化处理，降低空格、符号、大小写和简单变体绕过。
- 对评论和回复执行分层审核：明显违规直接驳回，灰区待人工复核，可信低风险短评论可快速通过。
- 发帖继续保持人工审核，不因本地规则安全而自动公开。
- 提供重复内容去重、用户频率限制和违规用户收紧策略，避免刷评论撑爆审核队列。
- 后台审核队列展示命中词、风险等级、处理建议和审核来源。

**Non-Goals:**

- 第一版不接入付费内容安全供应商，不依赖易盾、阿里云或腾讯云在线接口。
- 第一版不开放评论图片。
- 第一版不做复杂机器学习模型训练、语义大模型判定或跨平台黑产识别。
- 第一版不保证本地词库覆盖所有规避表达；高风险场景仍需人工复核和后续供应商接入。

## Decisions

### 1. 词库同步采用显式脚本，不运行时联网

新增 `scripts/sync-sensitive-lexicon.mjs`，从用户指定 GitHub 仓库同步文本词库到项目缓存，例如 `packages/shared/src/content-moderation/sensitive-lexicon.data.ts` 或 JSON 缓存。同步脚本只在开发者显式运行时访问网络；运行时 API 只读取项目内缓存。

替代方案是 API 启动时远程拉取词库，但这会引入启动不稳定、GitHub 可用性依赖和生产网络风险，因此不采用。

### 2. 审核采用本地分层策略

输入先经过文本归一化，再按词库分组和规则输出风险等级：

- `reject`: 性暴力、暴力威胁、严重辱骂、违法、广告引流、隐私泄露等明确违规。
- `review`: 涉政、灰区、疑似广告、重复刷屏、命中较弱词等需要人工判断。
- `pass`: 无命中、短文本、用户可信且未触发限流。

替代方案是所有评论默认人工审核，但会影响正常互动；另一个替代方案是全部自动通过后举报治理，风险过高，因此不采用。

### 3. 评论和回复可快速通过，帖子不自动公开

评论/回复属于高频低成本互动，可以在可信用户和低风险短文本情况下自动公开。帖子承载更多内容、图片和传播面，仍保持人工审核为主。

### 4. 可信用户为第一版轻量规则

第一版可信度不引入复杂风控分数，只基于可查询事实：账号已建档、近期无被驳回/隐藏记录、当天提交频率未超限、内容长度和重复度正常。后续可扩展为正式信誉分。

### 5. 后台展示可解释命中结果

审核结果保存归一化风险标签、命中词、命中字段、风险等级、处理建议和审核来源。普通公开响应不得返回命中词细节，后台可用于人工复核效率提升。

## Risks / Trade-offs

- **词库误伤正常表达** → 通过分组风险等级处理，弱命中进入人工复核而不是直接拒绝。
- **规避表达仍可能漏过** → 归一化降低简单绕过，后续可接第三方供应商和举报反馈补词。
- **词库过大影响性能** → 使用 Set/Trie 或预编译正则，服务启动时加载，避免每次请求读文件。
- **同步 GitHub 失败** → 脚本失败时不得覆盖已有缓存，并给出明确错误。
- **可信用户快速通过存在风险** → 快速通过只适用于短文本、无命中、低频、无违规历史用户；被举报或违规后自动收紧。
- **人工审核压力仍存在** → 通过本地硬规则、去重、限流和风险排序降低队列数量。

## Migration Plan

1. 增加本地审核契约、缓存词库和同步脚本。
2. 将社区评论/回复审核从“直接 AI 结果”改为“本地低成本审核结果优先”。
3. 将 AI 内容审核服务保留为可选增强层，当前默认不调用外部付费服务。
4. 更新后台队列展示命中信息和处理建议。
5. 更新小程序评论提交文案和“我的论坛”审核状态。
6. 增加验证脚本覆盖词库同步、明显违规拒绝、灰区待审、可信短评通过、限流和公开过滤。
