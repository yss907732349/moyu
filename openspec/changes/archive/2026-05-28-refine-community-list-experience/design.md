## Context

社区首页当前已经接入真实帖子列表、分区、筛选、发布、消息和漫画/IP Banner。问题集中在小程序端的信息架构和视觉密度：顶部存在解释型 Banner，占用首屏；“我的帖子”入口出现在社区首页筛选区，与个人论坛内容的归属不一致；帖子卡片纵向展示图片，单图使用 `widthFix` 后会随图片比例拉高卡片；作者头像已由公开快照返回，但页面仍展示 `avatarKey` 字符串。

现有后端和共享契约已经能支撑本次优化：`CommunityAuthorSnapshot` 包含 `avatarKey` 和 `faction`，帖子摘要包含 `mediaAssets`、`imageKeys`、互动统计和日报引用快照。因此本次设计以小程序端重排和渲染修正为主，不改变 API 契约。

## Goals / Non-Goals

**Goals:**

- 提升社区首页首屏信息密度，让用户更快看到帖子列表。
- 保留漫画/IP Banner 作为社区内漫画内容唯一入口，并让它呈现为正式内容入口。
- 从社区首页移除“我的帖子”入口，个人论坛内容统一从“我的”页进入。
- 固定帖子卡片中缩略图尺寸，避免列表高度随图片尺寸变化。
- 用内置头像资产展示社区作者头像，不暴露 `avatarKey` 字符串。

**Non-Goals:**

- 不恢复独立漫画底部 tab。
- 不新增漫画列表、漫画详情、章节解锁或漫画后台能力。
- 不改变社区发帖、审核、评论、回复、举报、通知或治理流程。
- 不新增社区公开 API 字段。
- 不引入真实推荐算法、精华算法或搜索能力。

## Decisions

### 1. 漫画/IP Banner 保留，但从占位说明改为正式入口

社区页仍保留顶部 Banner，因为 `fold-comic-ip-content-into-community` 已明确漫画/IP 内容第一版收敛到社区页 Banner 或社区内入口。Banner 文案应表达“隐者大陆漫画更新”“新的剧情等你来看”等内容入口语义，不再表达“模块封印中”“敬请期待”等临时占位感。

替代方案是删除 Banner 或隐藏到个人页。删除会失去漫画板块当前唯一入口；隐藏到个人页会削弱社区页承载世界观内容的定位，因此不采用。

### 2. 社区首页只保留公共浏览与发帖/消息操作

“我的帖子”属于个人内容管理入口，应保留在“我的”页。社区首页保留标题、消息和发布：消息用于社区互动反馈，发布用于 UGC 主操作。这样社区页的主线是浏览、切分区、筛选、发帖和查看消息，不再混入个人内容管理。

### 3. 帖子卡片采用左右信息结构和固定缩略图

帖子卡片应以左侧作者、标题、摘要、统计为主，右侧最多展示一张正方形缩略图。缩略图使用 `aspectFill`，容器固定尺寸；多图仅显示第一张和数量提示。无图帖子不展示图片占位，但整体排版仍保持紧凑。

替代方案是继续展示三图宫格。三图宫格更适合图片社区，不适合当前以文字讨论为主的论坛首页，会降低浏览效率。

### 4. 作者头像优先按快照头像资产渲染

公开作者快照已经包含 `avatarKey` 和 `faction`。小程序端应提供统一解析逻辑，把快照映射为项目内置头像资产。若无法识别具体 `avatarKey`，可按 `faction` 回退到阵营默认头像；仍失败时使用行影默认头像或安全占位。

这不会扩大公开作者字段，也不会允许用户上传头像。

## Risks / Trade-offs

- 漫画 Banner 目前可能没有完整落地内容页 → 入口可先跳转到现有兼容页或社区内轻量承接页，但 UI 文案不得再表现为无功能占位。
- 固定缩略图会裁切部分图片内容 → 使用 `aspectFill` 换取列表稳定性，详情页仍可展示更完整图片。
- 移除社区页“我的帖子”入口可能降低老用户发现效率 → “我的”页已有“我的帖子”“收藏帖子”“论坛消息”，并可保留未读提示作为主要回访入口。
- 仅按阵营回退头像会让同阵营作者头像相同 → 第一版符合内置头像资产策略，后续头像库解锁可在不改社区契约的情况下扩展映射。
