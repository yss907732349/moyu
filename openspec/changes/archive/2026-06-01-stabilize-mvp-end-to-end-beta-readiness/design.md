## Context

MVP 已经具备多个纵向能力：小程序首跑链路、日报和社区内容链路、补给铺 CPS 链路、生存账本、后台运营中心、实时待办与审核队列。当前风险不在单点能力是否存在，而在内测环境中这些能力是否能在同一套配置、同一批数据和同一组操作角色下稳定串联。

现有验证脚本主要按能力拆分，适合守住局部契约；内测前还需要一层横向验证，证明普通用户、运营人员和开发者可以完整走过 MVP 闭环。该层必须区分确定性本地验证、人工联调检查和真实外部服务配置检查，避免把微信、聚推客、AI 服务或网络代理的不稳定性混入默认 CI 式 smoke。

## Goals / Non-Goals

**Goals:**

- 定义 MVP 内测版的端到端就绪标准，覆盖首跑、内容、CPS、后台、配置、数据和验收链路。
- 增加一个默认确定性的总 smoke 命令，复用现有纵向验证并补充跨链路断言。
- 增加人工内测检查清单，覆盖微信开发者工具、管理后台实时交互和真实外部服务联调。
- 梳理环境变量样例和运行说明，确保内测成员能准备 API、后台、微信、CPS、AI、数据库和代理配置。
- 明确开发种子、真实 0 数据、空状态、错误态、降级状态和测试数据复位策略。
- 保持敏感字段边界，任何 smoke 输出、清单说明和普通用户响应不得泄露密钥、令牌或供应商原始数据。

**Non-Goals:**

- 不新增玩法、成长体系、社区形态、补给品类或商业模式。
- 不要求默认 smoke 连接真实微信、真实聚推客、真实 DeepSeek 或真实生产数据库。
- 不把开发种子、mock 登录、mock AI 或临时用户 fallback 作为生产能力。
- 不建设完整自动化 UI E2E 框架；本变更只提供内测前必要的确定性脚本和人工检查清单。
- 不改变已有能力的普通用户可见字段、审核规则和隐私边界。

## Decisions

### 1. 新增横向能力 `mvp-beta-readiness`

将内测就绪作为独立能力定义，而不是修改每个既有能力的核心需求。这样可以让 change 聚焦跨模块闭环、配置和验收，不把已经归档的玩法契约重新打开。

备选方案是分别修改 `miniapp-first-run-user-flow`、`daily-content-feed`、`community-lite`、`supply-center`、`accounting-ledger` 和 `admin-operations`。该方案更贴近模块归属，但会让一个稳定化 change 分散成多份 delta，增加归档和验收成本。

### 2. 默认 smoke 走确定性本地路径

新增总 smoke 命令时，默认只依赖本地源码、构建结果、mock/fixture 和已有验证脚本。真实微信登录、真实聚推客转链与订单回流、真实 AI 服务只进入人工或显式外部服务检查层。

备选方案是把真实外部服务纳入默认 smoke。该方案能更接近真实内测，但会让默认验证受账号、网络、代理、供应商可用性和真实密钥影响，不适合作为稳定准入。

### 3. 总 smoke 复用纵向脚本并补充跨链路断言

总 smoke 不重写所有能力验证，而是编排现有 `verify:*` 命令，并新增少量跨链路断言，例如日报评论进入审核、引用发帖进入社区审核、补给点击归因关联订单同步、生存账单只统计有效订单、后台实时事件不暴露敏感字段。

备选方案是写一个全新的大脚本覆盖所有细节。该方案容易重复现有验证逻辑，后续维护成本更高。

### 4. 测试数据采用“准备、运行、复位”三段式

内测检查需要可重复的测试数据策略：准备阶段创建或确认测试用户、测试日报、社区待审内容、补给项和模拟订单；运行阶段执行 smoke 或人工检查；复位阶段清理或隔离测试数据，避免污染下一轮。

备选方案是依赖手动临时造数据。该方案启动快，但会导致审核队列、订单同步、生存账本和通知状态漂移，排查成本高。

### 5. 配置就绪同时覆盖样例、说明和失败提示

环境变量样例需要覆盖根目录和应用目录的实际读取路径；README 或内测说明需要解释本地默认值、真机调试局域网地址、后台 token、微信 mock/真实配置、CPS 配置和代理配置。脚本应能检查缺失、占位值和敏感值误提交风险。

备选方案是只补 `.env.example`。该方案能改善配置入口，但不能解决内测成员不知道何时使用 mock、何时使用真实配置的问题。

## Risks / Trade-offs

- [Scope creep] 一个横向 change 容易演变成重做所有模块。Mitigation: 任务只允许修复阻断内测闭环的问题，新玩法和体验扩展另开 change。
- [External dependency instability] 微信、聚推客、AI 服务和代理可能不可用。Mitigation: 默认 smoke 不依赖外部服务，真实服务检查放入人工清单或显式命令。
- [Fallback masking production issues] mock 登录和临时用户 fallback 可能掩盖真实配置失败。Mitigation: 清单必须区分 mock 验收和真实配置验收，并标注生产前不得依赖 fallback 的项。
- [Data contamination] 测试订单、审核内容和通知可能影响后续检查。Mitigation: 使用稳定测试前缀、测试用户隔离和复位说明；有效消费和普通用户数据不得由种子伪造。
- [Sensitive output leakage] smoke 输出可能打印 token、sid、apikey 或原始订单响应。Mitigation: 验证脚本只输出脱敏摘要，并增加敏感字段黑名单断言。
- [SSE determinism] 实时待办流式连接难以完全自动验证。Mitigation: 脚本验证接口鉴权、payload 边界和降级轮询，人工清单验证浏览器实时提示。

## Migration Plan

1. 创建 `mvp-beta-readiness` 规格、总 smoke 任务和内测检查清单。
2. 补齐环境变量样例与 README 或内测运行说明。
3. 新增或扩展脚本，先串联现有 `verify:*` 命令，再加入跨链路断言和敏感字段检查。
4. 增加开发数据准备和复位说明，必要时提供只面向开发/测试环境的脚本。
5. 运行总 smoke、类型检查、构建和 OpenSpec strict 验证。

如出现阻断问题，只回滚本变更新增的脚本、文档和配置说明；不回滚已归档业务能力。

## Open Questions

- 总 smoke 命令命名使用 `verify:mvp-beta-readiness` 还是 `verify:mvp-smoke`。
- 开发数据准备是否做成脚本，还是先以检查清单和 fixture 形式落地。
- 真实聚推客订单回流是否能在内测阶段获得稳定沙箱或测试订单，否则应以模拟同步记录验证入账链路。
