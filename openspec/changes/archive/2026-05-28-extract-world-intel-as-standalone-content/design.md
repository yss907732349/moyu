## Context

当前 `大陆新闻` 使用 `daily-content-feed` 内的 `world_intel` 板块承载。后台已经把日报三板块拆成独立工作区，但底层仍保留日报栏目语义：大陆新闻文章挂在某条 `DailyContentIssue` 的 `DailyContentSection.items` 下，校验、归一化、公开列表和预览都保留“文章栏目最多 10 篇”的限制。

这与新的运营预期冲突：大陆新闻不是每天被新一期覆盖的栏目，而是持续累积的新闻内容库。运营希望一次上传多条，旧文章继续可见，用户可以查看历史文章，后台可以单独管理并修改已发布内容。

## Goals / Non-Goals

**Goals:**

- 将大陆新闻从日报板块记录中抽离为独立内容库和独立后台页面。
- 支持大陆新闻文章持续累积、分页公开、历史可回看。
- 支持后台批量新增任意数量的大陆新闻文章，不再使用 10 条栏目上限作为业务规则。
- 支持编辑已发布大陆新闻内容，并让公开侧读取最新版本。
- 保留现有来源完整性、公开响应敏感字段边界和后台令牌边界。
- 为历史 `world_intel` 日报栏目文章提供迁移或兼容读取策略。

**Non-Goals:**

- 不把 `离谱卷宗` 同步改造成独立内容库。
- 不引入复杂 CMS 权限、多人协同、版本 diff、回滚历史或完整审计后台。
- 不引入自动新闻采集、第三方爬取或 AI 自动发布。
- 不改变日报评论、社区引用和社区审核的基础生命周期。
- 不要求小程序新增底部 tab；大陆新闻可以继续从隐者日报入口进入。

## Decisions

### 1. 新增大陆新闻独立内容模型，而不是继续复用 `DailyContentIssue`

新增独立的 `WorldIntelArticle` 或等价模型，字段至少包含：

- `id`
- `title`
- `summary`
- `body`
- `source`
- `coverImageUrl` 或 `coverImageKey`
- `status`
- `publishedAt`
- `updatedAt`
- `createdAt`
- `sortOrder` 或 `pinnedWeight`
- `allowLike`
- `allowCommunityQuote`
- `quotePrompt`

理由：继续把大陆新闻放在 `DailyContentIssue -> Section -> Item` 下，会让“发布一次大陆新闻”天然带有一次日报板块记录的边界，历史聚合、跨期分页、编辑已发布内容和后台列表检索都会变得绕。独立模型能直接表达“文章库”。

备选方案是保留 `DailyContentItem` 表，新增 `contentKind = world_intel` 字段。这个方案迁移少，但会继续混用日报文章、评论、点赞、引用等语义，后续维护边界不清晰。

### 2. 公开侧大陆新闻使用分页文章库 API

新增或等价实现：

```text
GET /world-intel/articles
GET /world-intel/articles/:articleId
```

列表接口按 `publishedAt` 倒序返回已发布文章，支持 `page/pageSize` 或 cursor 分页。服务端 SHALL 设置最大 `pageSize`，避免一次返回无限历史数据。

理由：用户需要看历史文章，但公开接口不能一次返回全部文章。分页是产品“不限制总量”和工程“控制响应体”的交界。

### 3. 后台大陆新闻独立页面提供批量新增和已发布编辑

后台新增大陆新闻管理页面，列表展示所有状态文章，支持：

- 新建单篇文章
- 批量新增多篇文章
- 编辑草稿或已发布文章
- 发布草稿
- 下线或隐藏已发布文章
- 搜索和状态筛选

后台单次批量新增不设置业务条数上限，但仍由 API 限制请求体大小、字段长度和超时边界。

理由：运营诉求是“几条都可以”，这里不应再出现 10 条业务上限。但工程层必须防止误传超大请求导致服务不可用。

### 4. 日报首页保留大陆新闻入口，但不再返回大陆新闻文章正文列表

隐者日报摘要可以继续返回大陆新闻入口卡，入口卡的 `articleCount` 改为大陆新闻已发布总数或近期可读数量。小程序点击后进入大陆新闻独立列表。日报 `world_intel` 板块不再参与单板块保存、发布和 10 条校验。

理由：从用户心智看，大陆新闻仍是隐者日报下的重要内容入口；从运营和数据模型看，它已经不是日报板块。

### 5. 历史 `world_intel` 数据采用一次迁移优先、兼容读取兜底

实现时优先提供迁移脚本或迁移逻辑，将已发布日报中的 `world_intel` 内容项写入大陆新闻独立文章表，并保留原 `DailyContentItem.id` 作为可追溯来源字段或 `legacyDailyContentItemId`。

如果本地或开发环境没有执行迁移，公开列表可以临时兼容读取已发布 `world_intel` 项，去重后合并展示；生产稳定后应以独立表为事实来源。

理由：已经发布的大陆新闻不能因为模型升级消失。一次迁移能降低长期复杂度，兼容读取能降低开发环境和历史数据缺口风险。

### 6. 点赞、评论和社区引用先复用文章级能力边界，但不扩大范围

大陆新闻文章可以保留 `allowLike`、`allowCommunityQuote` 和引用快照字段。日报文章评论是否直接迁移到大陆新闻文章评论，需要以现有数据关联为准：如果历史评论绑定到旧 `DailyContentItem.id`，迁移时可保留旧文章详情兼容或把评论迁到新文章 ID。

理由：用户侧已经有文章详情、点赞、评论和引用体验。拆模型时要避免把互动数据丢掉，但本次不扩展新互动形态。

## Risks / Trade-offs

- [Risk] 新增独立文章模型会引入迁移复杂度。
  Mitigation: 先设计迁移脚本和 `legacyDailyContentItemId` 去重字段，允许开发环境兼容读取旧 `world_intel` 数据。

- [Risk] “不限制每次发布数量”可能导致后台误传超大请求。
  Mitigation: 明确取消业务条数上限，但保留请求体大小、字段长度、服务端分页和超时保护。

- [Risk] 小程序日报首页与大陆新闻独立列表之间的接口边界容易混乱。
  Mitigation: 日报摘要只给入口和计数；大陆新闻列表/详情只走 `world-intel-content` API。

- [Risk] 已发布内容可编辑可能改变用户已看过或已引用的上下文。
  Mitigation: 第一版至少展示 `updatedAt`，引用快照继续保存发帖时的标题和摘要；复杂版本历史不在本次范围。

## Migration Plan

1. 新增共享契约和独立 API，暂不删除旧日报 `world_intel` 字段。
2. 新增大陆新闻持久化模型和迁移脚本，把已发布 `world_intel` 内容项迁入独立文章库。
3. 后台新增大陆新闻页面，并从日报运营页面移除大陆新闻编辑工作区。
4. 小程序日报首页大陆新闻入口切换到独立列表，列表和详情改用新 API。
5. 验证迁移后旧文章、来源、配图、点赞/评论/引用边界可用。
6. 稳定后移除公开侧对日报 `world_intel` 列表的 10 条截断依赖。

Rollback 策略：保留旧日报 `world_intel` 数据和接口一段时间；如新列表出现问题，可临时把小程序入口切回旧日报栏目列表，但不得删除已迁移的独立文章数据。

## Open Questions

- 已有日报文章评论是否必须随历史大陆新闻迁移到新文章 ID，还是允许旧文章详情兼容读取一段时间。
- 大陆新闻列表第一版是否需要搜索和置顶，还是后台可搜索、公开侧只做分页倒序。
