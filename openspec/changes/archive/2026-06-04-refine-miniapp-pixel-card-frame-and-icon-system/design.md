## Context

当前小程序已经归档完成 `unify-miniapp-global-camp-ui`，并在 `docs/miniapp-global-ui-design-prd.md`、`docs/miniapp-visual-illustration-system.md`、`apps/miniapp/src/styles/visual-system.css` 和 `apps/miniapp/src/visual-system/illustration-registry.ts` 中沉淀了暗色像素 Material、真实插画资产、语义 icon fallback 和全局营地 UI 规则。

现在的问题不是缺少视觉风格，而是视觉权重边界还不够细。`vs-panel`、`camp-card`、信息流卡、表单卡和状态面板已经在多个页面复用；如果新像素框挂到这些基础样式上，会同时影响首页主卡、帖子列表、评论、回复、订单行、表单分组、tab 和 badge，导致页面信息噪音过高。

本 change 聚焦在小程序用户端展示层，目标是在现有全局营地 UI 之上补一层“视觉权重治理”：哪些卡可以拥有重点像素框，哪些卡只做轻立体，哪些控件保持扁平。像素 icon 体系也在同一 change 中收敛，但不改变 tab 数量、页面路由或业务能力。

## Goals / Non-Goals

**Goals:**

- 定义重点像素框的准入规则和页面级应用白名单。
- 将普通卡片统一为轻立体质感，使列表、内容、补给和账本卡片与重点像素框共处但不抢视觉主位。
- 将表单、评论、回复、筛选 tab、badge、输入框和订单行保持为扁平或低强调容器。
- 为首页、社区、我的三个微信原生 tabBar 补齐未选中/选中像素 icon。
- 收敛菜单、状态、补给、账本、日报、工作设置等语义 icon 到稳定 registry 或资产映射。
- 保留现有真实插画、页面板块、三 tab 信息架构、feature key、路由和业务逻辑。

**Non-Goals:**

- 不实现自定义 tabBar。
- 不新增底部主 tab，不恢复漫画独立主 tab。
- 不新增业务功能、入口、接口、数据库字段或服务端逻辑。
- 不改变 CPS 转链、点击归因、订单同步、账本统计、退款/取消回滚、社区审核、内容安全、首跑状态推导或工作价值计算。
- 不把所有 CSS fallback icon 一次性替换成最终美术资产；第一版允许 registry 中仍有可发布 fallback。
- 不替换现有项目内真实插画、头像、徽章、Banner 和封面。

## Decisions

### 1. 重点像素框使用显式 opt-in 样式，不修改基础面板

重点像素框应作为独立样式或组件能力，例如页面显式添加的主视觉 frame class，而不是直接修改 `vs-panel`、`camp-card`、`camp-feed-card` 或通用状态面板。

理由：基础面板被列表、表单、评论和订单等高密度区域复用。修改基础类会让全站同时变重，破坏“重点像素框只服务主视觉”的目标。

备选方案是增强 `vs-panel` 默认视觉。该方案实现最快，但会让所有普通卡片都获得同等仪式感，不符合本 change 的视觉治理目标。

### 2. 用三档视觉层级管理卡片，而不是只区分“有框/无框”

第一版卡片层级定义为：

- 重点像素框：页面主卡、Banner、身份卡、主推荐和核心统计。
- 轻立体卡：信息流、内容入口、补给 item、账本报告、普通统计和常规可点击卡。
- 扁平容器：表单字段、评论、回复、tab、badge、输入框、订单行和筛选控件。

理由：小程序同时有 RPG 氛围和工具属性。只提供重框会让页面吵；只提供轻卡又无法突出身份和仪式感。三档层级可以让首页金额、我的身份和社区 Banner 明确成为主角。

### 3. 页面级白名单优先于组件自动判断

重点像素框的第一版应用应按页面白名单人工指定：

- 首页：今日已摸金额主卡。
- 社区：首页顶部世界观 Banner。
- 我的：身份主卡；角色页阵营身份主卡。
- 补给铺：主推荐卡。
- 隐者日报：首页今日话题/今日参悟主卡。
- 生存账本：账本页顶部今日生存消耗统计卡。

首页今日生存消耗入口、日报/补给入口、文章列表、帖子列表、评论、回复、订单行、工作档案表单均不进入重点像素框白名单。

理由：是否应使用重框取决于页面语义和视觉权重，不是由“是否是卡片”自动决定。白名单能降低误用风险，也便于验收。

### 4. 首页只能保留一个最强视觉主角

首页“今日已摸金额”主卡是 MVP 核心循环，应保留最强重点像素框。首页“今日生存消耗”虽然重要，但在首页语境中应默认使用轻立体卡；账本页自身的顶部统计卡可以使用重点像素框。

理由：首页如果同时出现两个同等级重框，工作价值主循环会被削弱。生存消耗在账本页是主任务，在首页是入口摘要，视觉等级应随页面语境变化。

### 5. 微信原生 tabBar icon 先行，不引入自定义 tabBar

首页、社区、我的三个 tab 使用 `pages.json` 的 `iconPath` / `selectedIconPath` 接入未选中/选中像素 icon。第一版不做自定义 tabBar，也不自行同步 tab 状态。

理由：当前底部导航只有三个稳定主 tab，原生 tabBar 足够承载第一版 icon 需求。自定义 tabBar 会引入状态同步、安全区、选中态和页面切换复杂度，超过本 change 的视觉修补范围。

### 6. 语义 icon 通过 registry 收敛，允许分阶段替换 CSS fallback

菜单、状态、补给、账本、日报、工作设置等 icon 应优先通过稳定 asset key 或 `resolveSemanticIconClass` 一类 registry 解析。已有 CSS fallback 可以作为第一版兜底，但新增 icon 不应在页面内临时手画一套。

理由：项目已经有 `IllustrationKeys` 和语义 icon fallback。继续页面内各画各的，会让 icon 风格漂移，并增加后续替换正式 PNG 像素资产的成本。

### 7. 文档与实现同步更新

实现时应同步更新 `docs/miniapp-global-ui-design-prd.md` 和 `docs/miniapp-visual-illustration-system.md`，把重点像素框白名单、三档卡片层级、tabBar icon 资产路径、语义 icon registry 和禁用位置写入文档。

理由：近期归档 change 已把文档作为当前事实来源。只改代码不更新文档，会让后续 change 重新误判哪些卡可以使用重框。

## Risks / Trade-offs

- **Risk: 重框白名单过窄，部分页面显得不够 RPG。** → 先保持克制，验收后再按页面语义单独扩展，不把重框一次性铺开。
- **Risk: 重框白名单过宽，信息流和表单变吵。** → 规格明确禁止列表项、评论、回复、订单行、表单、tab、badge 和输入框使用重点像素框。
- **Risk: 原生 tabBar icon 受微信能力限制，无法实现复杂动效或特殊布局。** → 第一版接受原生限制，等视觉诉求超过原生能力时再新建 change 评估自定义 tabBar。
- **Risk: 新增 PNG icon 增加小程序包体。** → icon 使用小尺寸透明 PNG，复用色板和资产目录，并继续执行包体检查。
- **Risk: CSS fallback 和 PNG icon 混用导致风格不一致。** → 统一 asset key、色彩、描边和选中态规范；页面新增 icon 必须走 registry。
- **Risk: 调整卡片样式时误碰业务模板或事件。** → 只改 class、样式、静态资产引用和文档；验收时检查路由、feature key、点击操作和已有板块完整性。

## Migration Plan

1. 补充视觉系统文档和规格中关于重点像素框、轻立体卡、扁平容器和 icon registry 的规则。
2. 在小程序静态资产目录新增首页、社区、我的三组原生 tabBar 像素 icon，并在 `pages.json` 中配置。
3. 在全局视觉样式中新增或整理重点像素框、轻立体卡和扁平容器的可复用样式，不改变通用基础类默认语义。
4. 按白名单逐页应用重点像素框，并将其他卡片降为轻立体或扁平层级。
5. 补齐或收敛菜单、状态、补给、账本、日报、工作设置等语义 icon registry 使用。
6. 更新文档，记录应用白名单、禁用位置、资产路径和验收标准。
7. 执行小程序构建、包体检查和相关页面 verify，人工检查小屏、三 tab、主卡、列表、表单、评论、订单行和内部字段隐藏边界。

回退策略：

- 如果重点像素框在某页造成拥挤，回退该页的 opt-in frame class，保留轻立体卡样式。
- 如果 tabBar icon 资源异常，回退 `pages.json` iconPath 配置，保留三 tab 文本导航。
- 如果全局轻立体样式影响范围过大，回退对应公共样式，并逐页恢复到已归档的全局营地 UI 基线。
