## Context

当前正式规格中尚未生效 `work-value-tracker`，暂停版 MVP 规划只粗略描述了月薪、工作时间、午休、工作制和发薪日。小程序首页目前仍是视觉演示模拟，数据库也没有业务模型。

这个 change 只定义工作档案和计算契约，不实现页面、不创建数据库迁移、不接入真实节假日数据源，也不调整其他用户身份系统。

## Goals / Non-Goals

**Goals:**

- 定义工作档案的稳定字段、本人可见边界和基础校验规则。
- 支持第一版简单月薪估算、固定上班时段、一个或多个休息段、标准或自定义工作日和发薪日。
- 明确首页已摸金额应基于已保存档案在前端本地实时计算。
- 明确工作状态和倒计时输出，避免首页组件各自重写计算逻辑。
- 保留未来法定节假日、复杂排班、税后薪资和高级薪资结构的扩展点。

**Non-Goals:**

- 不实现税前/税后切换、个税、社保、公积金、奖金、十三薪或真实工资结算。
- 不实现复杂排班、大小周、调休、请假或多班次轮班。
- 不确定最终法定节假日数据源。
- 不实现后台工作档案管理或运营分析。
- 不实现用户身份推荐或排行榜。
- 不扩大隐藏模式为遮挡所有可推断薪资信息。

## Decisions

### Decision: 工作档案作为已摸金额计算的唯一用户输入来源

工作档案保存用户本人填写的薪资、工作时间、休息段、工作日规则和发薪日规则。首页、设置页和后续计算器都应读取同一个工作档案快照，而不是在页面内维护独立字段。

Rationale: 首页金额、倒计时和上班状态都依赖同一组输入。统一契约可以避免字段漂移，并为后续跨端同步和接口校验留下空间。

Alternatives considered:

- 只在首页本地保存设置。实现简单，但跨设备、后端同步和后续用户资料联动都会受限。
- 每个功能自行保存所需字段。短期灵活，但会导致计算口径不一致。

### Decision: 第一版薪资采用简单月薪估算

薪资模式第一版使用 `simple_monthly`，用户输入月薪金额和币种，默认币种为 `CNY`。该金额只用于娱乐化估算“今日已摸金额”，不是工资结算结果。

Rationale: 年轻打工人场景中月薪是最容易理解的输入。个税、社保、公积金、奖金和补贴会显著增加复杂度，不适合第一版契约。

Alternatives considered:

- 一开始支持税前/税后和五险一金。更精确，但会让设置流程变重，也容易被误认为严肃工资计算器。
- 只输入时薪。计算简单，但不符合多数国内用户的薪资表达习惯。

### Decision: 休息段使用数组表达

工作档案使用 `breaks` 数组表达休息段。第一版 UI 可以只引导用户填写午休，但契约允许多个休息段。

Rationale: 数组结构能覆盖午休、下午茶、晚餐休息或未来夜班中断，不需要后续改字段结构。

Alternatives considered:

- 固定 `lunchStartTime` 和 `lunchEndTime`。更直观，但扩展性差。

### Decision: 工作日规则先支持标准工作日和自定义星期

工作日规则支持 `standard_weekdays` 和 `custom_weekdays`。第一版可以通过星期数组表达双休、单休或简单排班，不强制接入法定节假日和调休。

Rationale: 这覆盖 MVP 主要用户，同时避免过早承诺节假日数据源和复杂排班。

Alternatives considered:

- 立即支持中国法定节假日调休。用户体验更准，但数据源和维护责任尚未确定。
- 只支持周一到周五。实现最省，但会排除单休、大小周或服务业用户。

### Decision: 隐藏模式只处理首页敏感展示

隐藏模式第一版用于遮挡首页已摸金额等显性敏感展示，并切换为安全视觉状态。本 change 不要求遮挡每秒收益、月薪、发薪日、工时等所有可推断信息；这些字段的展示位置和隐私策略由后续具体页面 change 再细化。

Rationale: 隐藏模式的主要场景是避免他人一眼看到首页核心金额。扩大到全量推断信息会显著增加规则复杂度，也可能影响正常设置和使用。

Alternatives considered:

- 遮挡所有可能推断薪资的信息。更严格，但第一版成本和交互复杂度更高。

### Decision: 倒计时分为工作档案可推导和外部配置提供

工作档案可以直接推导距上班、距下班、距休息日和距发薪日。法定节假日、运营活动或特殊补给倒计时应由外部日历或运营配置提供，不属于工作档案本身。

Rationale: 工作档案是用户设置，不应承担公共日历和运营活动配置职责。

Alternatives considered:

- 把所有倒计时都放进工作档案。短期方便首页读取，但会混淆用户配置和运营配置。

## Risks / Trade-offs

- [Risk] 简单月薪估算与真实工资不一致 -> Mitigation: 文案和规格明确为娱乐化估算，不做工资结算承诺。
- [Risk] 不接入法定节假日导致倒计时不准 -> Mitigation: 契约保留外部日历扩展点，第一版只展示基于工作日规则可推导的结果或配置化占位。
- [Risk] 多休息段增加 UI 复杂度 -> Mitigation: 契约支持数组，第一版 UI 可以只提供一个午休入口。

## Migration Plan

1. 在 `packages/shared` 中定义工作档案、薪资模式、工作日规则、发薪日规则、计算输入输出和状态枚举。
2. 后端实现本人工作档案保存与读取接口，并只向本人返回完整档案。
3. 小程序设置页读取和保存工作档案，首页通过统一计算方法消费工作档案快照。
4. 首页替换视觉演示金额为本地实时计算结果，并处理未配置状态。
5. 后续接入节假日、后台能力或用户身份相关能力时复用计算边界，不修改第一版计算口径。
