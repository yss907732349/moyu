## Context

隐者日报当前已经具备文章化首页、`今日参悟`、栏目入口，以及从日报资源引用到社区发帖的基础契约。问题集中在小程序首页体验：`今日参悟` 现在更像一句静态文案，缺少“这是今日话题、可引用发帖、会进入社区讨论”的明确引导。

本 change 不重新定义日报内容生产，也不引入完整话题系统。它把 `今日参悟` 的小程序首页表现从“短句展示”调整为“页内话题引导卡”，并把点击行为收敛为直接打开社区发帖体验。

## Goals / Non-Goals

**Goals:**

- 让用户在日报首页明确理解 `今日参悟` 是可参与的今日话题，而不是单纯装饰文案。
- 保持内容轻量：一句参悟不单独做二级详情页。
- 点击引用入口后直接进入社区发帖页，减少转化路径。
- 发帖页展示引用来源，让用户知道自己正在围绕哪句参悟发帖。
- 保存并公开展示通过审核的日报参悟引用快照。

**Non-Goals:**

- 不建设今日话题详情页、话题广场、热门讨论聚合页或历史话题列表。
- 不改变日报后台采编流程，不要求运营额外维护复杂话题配置。
- 不改变社区帖子审核规则，引用官方日报不代表自动公开。
- 不新增社区分区，也不把今日话题做成独立底部 tab 或主入口。

## Decisions

### 1. `今日参悟` 作为日报首页话题卡，不做独立详情页

采用日报首页内嵌卡片，标题可使用 `今日话题`、`今日参悟话题` 或等价语义，卡片内展示参悟短句、引导说明、引用发帖按钮和可选参与状态。

原因：第一版 `今日参悟` 只有一句话或短段落，单独二级页面信息密度不足，会增加跳转和流失。日报首页本身就是触发用户阅读和表达的上下文，页内转化更直接。

备选方案：建设 `今日参悟` 详情页。该方案暂不采用，因为它需要额外内容、参与帖子列表或话题聚合能力才能显得成立，已经超出本 change 范围。

### 2. 点击引用入口直接打开社区发帖体验

日报首页点击 `引用发帖` 后直接打开社区发帖页，并携带引用草稿：

- 来源类型：`daily_reflection`
- 日报 ID
- 业务日期
- 参悟正文
- 引用提示或话题引导文案

原因：目标是给论坛引流，链路越短越容易验证转化。发帖页是用户真正输入观点的位置，不需要中间承接页。

### 3. 引用内容作为帖子来源快照保存

社区帖子保存日报引用快照，而不是在公开展示时实时依赖日报当前内容。公开帖子展示时可展示“引用今日参悟”、参悟正文、业务日期和可跳回日报资源的标识。

原因：发帖时的引用上下文应稳定，避免运营后续修改日报导致历史帖子语义漂移。

### 4. 身份和审核边界沿用社区规则

未登录或未创建隐者档案的用户点击引用发帖时，仍进入既有身份引导。已创建隐者档案的用户提交引用帖子后，帖子仍为 `pending`，等待社区审核。

原因：引用来源是官方内容，但用户补充的标题和正文仍是 UGC，必须遵守社区内容治理和审核生命周期。

## Risks / Trade-offs

- [Risk] 话题卡文案过于游戏化，用户仍不理解行为结果 → Mitigation：按钮使用直接动作词，例如 `引用发帖`，辅助文案明确写出“去社区写下今日摸鱼心得”。
- [Risk] 参与人数在早期为 0，展示后反而降低参与意愿 → Mitigation：参与状态设为可选字段；无数据时展示“成为第一个引用今日参悟的隐者”或不展示人数。
- [Risk] 发帖页预填过多内容导致用户误以为已经完成发帖 → Mitigation：引用来源只作为独立引用块展示，正文输入区保持空白或仅显示轻提示，不自动伪造用户正文。
- [Risk] 引用快照字段与现有社区引用契约不一致 → Mitigation：优先复用现有 `daily_reflection` 引用字段，只补齐小程序草稿传递和展示缺口。

## Migration Plan

无需数据迁移。已发布日报的 `今日参悟` 可直接按新卡片展示；现有不允许引用或缺少参悟正文的日报继续展示不可用或空状态。

如后续需要话题聚合页，应另建 OpenSpec change，在已有引用帖子数据基础上扩展查询和聚合能力。
