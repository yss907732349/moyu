## Context

当前小程序已经具备暗黑像素忍者 RPG 的基础视觉系统，并在社区页试用了用户认可的“隐者营地”暗色像素 Material 方向。试用过程中也暴露了两个典型问题：全局网格背景在空白区显得过重、长连续字符在帖子列表和详情页中横向溢出。这说明后续全局 UI 统一不能只替换色值和卡片样式，还需要明确页面背景、文本可读性、时间显示、板块完整性和业务边界。

本 change 以 `docs/miniapp-global-ui-design-prd.md` 为核心 PRD，目标是把社区页已验证的视觉方向沉淀为全局视觉系统，并逐页应用到小程序用户端高频页面。

## Goals / Non-Goals

**Goals:**

- 统一小程序用户端页面的暗色像素 Material + 暗黑忍者 RPG 视觉语言。
- 抽象可复用的 camp UI token、卡片、按钮、标签、tab、输入框、状态面板和信息流样式。
- 保持首页、社区、详情页、我的、补给铺、日报、大陆新闻、生存账本和工作档案的板块完整。
- 统一时间展示格式，禁止普通用户看到 raw ISO 时间。
- 修复长连续数字、英文、URL 和中英混排文本在小屏中的横向溢出。
- 保留项目内真实插画、头像、徽章、Banner 和封面资产，仅调整遮罩、裁切、安全区和容器样式。
- 保证普通用户界面不暴露 CPS、聚推客、佣金、订单源 ID、apikey、sid、brand_id、act_id、后台备注或原始同步响应。

**Non-Goals:**

- 不新增小程序业务功能。
- 不新增底部主 tab。
- 不恢复漫画独立主 tab。
- 不新增未定义功能宫格、空成就系统或空徽章系统。
- 不改 API 契约、服务端业务逻辑、Prisma schema、共享业务契约或数据库迁移。
- 不改内容安全审核、微信审核、订单同步、点击归因、退款回滚、账本统计和首跑状态推导。
- 不更换已存在的真实视觉资产。

## Decisions

### 1. 将“营地 UI”沉淀为全局 token，而不是继续页面内复制色值

本 change 应优先在小程序全局视觉系统中沉淀 camp UI 语义 token，例如页面底、卡片底、描边、主紫、状态青、金额金、错误色和弱文字。页面可在必要时保留少量局部样式，但同类视觉语义不得在多个页面中各自硬编码。

理由：全局 UI 优化涉及多个页面，继续页面内复制样式会让后续维护困难，也容易出现社区页与首页、我的页风格再次分裂。

备选方案是只改每个页面的 `<style>`。该方案短期快，但会重复引入同类色值、阴影和卡片样式，不利于归档后作为当前事实来源。

### 2. 全局背景保持干净深色，局部纹理只用于 Banner 和卡片

全局页面背景使用极暗渐变和克制紫色氛围，不再铺满高对比网格。扫描线、像素纹理和局部发光只用于 Banner、选中态、引用卡或状态卡。

理由：用户已经指出满屏网格背景不符合预期。深色工具页面需要先保证阅读和空白区质感。

### 3. 板块完整性优先于视觉重排

每个页面优化前需要列出现有板块，优化后逐项对照确认。允许改变间距、卡片样式、字号、遮罩和排列密度，但不得删除、隐藏、合并或新增业务板块。

理由：本 change 的目标是视觉统一，不是信息架构重做。已有 MVP 能力和近期归档 change 已经定义了当前页面边界。

### 4. 展示格式化可以调整，但必须隔离于业务计算

允许新增或调整纯展示格式，例如时间展示、长文本换行、金额遮挡文案和状态文案映射。但这些格式化不得改变接口数据、统计口径、订单状态、审核状态、用户权限或本地缓存事实。

理由：时间格式和长文本可读性属于 UI 成功的必要条件，但不能变成业务逻辑重构。

### 5. 真实资产保留，只优化展示容器

已有插画、头像、徽章、Banner 和封面保持原文件。允许调整 `aspectFill` / `aspectFit`、遮罩、圆角、容器比例、暗色覆盖和安全区。

理由：用户明确要求插画不换；当前项目已经建立资产矩阵，UI change 不应重新引入外链图或生成图。

### 6. 视觉验收使用社区页作为标杆页

社区列表当前营地 UI 是用户确认满意的方向，后续页面应以它作为视觉基准：暗底、像素卡片、紫色激活、青色状态、金色主操作、克制发光和清晰信息流密度。

理由：有真实反馈的标杆页能降低主观审美偏差。

## Risks / Trade-offs

- **Risk: 全局样式过强导致某些业务页面信息拥挤。** → 分页面走查小屏 `320px / 375px / 414px`，对表单和详情页降低装饰强度。
- **Risk: 视觉统一时误删已有入口或改变路由。** → 每页维护板块完整性清单，模板和脚本改动需和视觉改动分开审查。
- **Risk: 长文本换行规则影响按钮、tab、短标签美观。** → `word-break: break-all` 仅用于用户输入和服务端长文本，不用于按钮、tab、状态短标签。
- **Risk: 时间格式化引入时区或日期边界差异。** → 仅做用户侧显示格式化，不改变服务端时间、排序字段或业务日期计算。
- **Risk: 发光和纹理过多导致暗色页面变脏。** → 强 glow 只用于当前激活、状态和主操作；全局背景不使用明显网格。
- **Risk: 补给铺和账本视觉改动暴露内部商业字段。** → 普通用户字段隐藏边界作为规格验收项，不以开发调试字段填充 UI。

## Migration Plan

1. 先抽象或补充全局 camp UI token 和公共展示工具。
2. 以社区列表和帖子详情为标杆页验证信息流、长文本和时间格式。
3. 再统一首页和我的页，确保主循环和身份中心完整。
4. 再统一日报、大陆新闻、补给铺、生存账本和工作档案页面。
5. 最后处理 WebView、comics 兼容页和状态面板。
6. 每个阶段运行对应构建和验证命令，并做人工页面走查。

回退策略：

- 视觉 token 和页面样式应保持可局部回退。
- 如果某页视觉不符合预期，优先回退该页面样式，不回退业务脚本或共享契约。
- 若全局 token 引发大面积问题，回退 token 修改并保留已验证的展示格式化修复。

## Open Questions

- 是否在本 change 中同时把所有页面的页面内重复样式彻底迁移到全局样式文件，还是先完成视觉统一后再做样式收敛。
- 首页、我的页、补给铺是否需要单独输出静态 HTML 或截图预览给用户确认，再进入批量页面改造。
- 时间格式化工具放在小程序本地服务层还是共享包中，需要在实现时根据复用范围确认。
